## Todo 体系

项目 todo 列表位于 `.self-workflow/todo.md`。

### 结构

```
# Todo
├── Vx.y.z：标题（优先级）    ← 已排期的版本
│   ├── P0 🔴 阻断成熟度       ← 必须在此迭代完成
│   ├── P1 🟡 质量改善          ← 应在近期完成
│   └── P2 🟢 愿景             ← 远期规划
├── Vx：远期愿景               ← 已识别但近期不排期
├── 已关闭                     ← 已完成（折叠）+ 已拒绝（wontfix）
└── 新增（待评审排期）          ← 未分配版本的待办收件箱
```

### 优先级

| 等级 | 含义 | 行为 |
|------|------|------|
| P0 🔴 | 阻断框架成熟度 | 必须在本迭代完成，完成前不推进版本 |
| P1 🟡 | 质量改善 | 应在近期完成，搁置需说明理由 |
| P2 🟢 | 愿景/远期规划 | 不设时间压力 |

### 版本命名

- 已排期版本：`Vx.y.z`（语义化版本，如 V1.6、V1.7）
- 每个版本有明确的主题（如"质量收尾"、"docs 结构"），各项粒度应相近
- 未排期的远期项：使用 `Vx`（如"Vx：Agent 能力增强"）
- 版本内的项按 P0 → P1 → P2 排序

### 项的流转

```
新增（待评审） → 评审通过 → 排入 Vx.y.z 版本 → 实施完成 → 移入已关闭
                    ↓ 否决
               [wontfix]（附拒绝理由）
```

### 已关闭

- 已完成项：移入 `## 已关闭`，按版本用 `<details>` 折叠，最旧的版本在最下面
- 已拒绝项：标注 `[wontfix]` 并在其下方附拒绝理由

### 新增（待评审排期）

- 未分配版本的临时想法存放于此
- 评审通过后：
  - 粒度匹配 → 排入现有版本（如 V1.6）
  - 新主题 → 创建新版本章节
  - 远期 → 移入 `Vx`
  - 否决 → 标记 `[wontfix]` 移入已关闭
- ⚠️ **此章节是"待办收件箱"，整理 todo 时必须保留，不可随版本清理被删除**

### Agent 修改 todo 时的规则

1. 新增想法 → 追加到 `新增` 章节，不要自行排入版本
2. 完成一个版本的所有项 → 将该版本移入 `已关闭`，用 `<details>` 折叠
3. 整理 todo → 可以重新排序、合并重复项、调整优先级标记。**不可删除 `新增` 章节**
4. 发现新的 P0 问题 → 追加到 `新增`，由 Human 决定排入哪个版本
