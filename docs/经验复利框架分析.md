## 二、exp-governance skill/spec/feat-workflow 质量评审

### 2.1 exp-governance spec（`specs/default/exp-governance.md`）

**优点**：
- 生命周期状态定义清晰（draft→verified→outdated→refreshed→archived），经过本次治理实战验证——全部 21 份文档按此模型完成了 Draft→Verified 晋升
- 去重规则强调"语义判断而非机械匹配"——本次合并的 3 组正是此原则的产物（同根异面/互补/因果链三类关系均非算法可检测）
- SHOULD 级规则"晋升 verified 前检查内容与最新源码/spec 一致"在实战中有效——4 份早先标记 verified 的文档经重新评估后确认合格

**可优化**：
- 缺少"freshness 检查"触发条件——verified 文档在关联源码变更后如何自动标记为"需重新审查"？
- 去重维度目前依赖人工逐对通读——文档 > 50 时不可行。可补充辅助方法（如 tag 交集 > 80% 作为疑似重复信号，但仍以语义判断为最终判定）

### 2.2 exp-governance skill（`.opencode/skills/exp-governance/SKILL.md`）

**优点**：
- 执行步骤清晰（收集→一致性→去重→质量→生命周期→报告）
- 输出格式标准化（Markdown 表格）

**可优化**：
- **缺少自动化步骤**：一致性审查中的"tag 规范性"检查可增加正则 `^tags:.*[\u4e00-\u9fa5]` 扫描（本次 Tier5 遗漏 3 份文档的教训）
- **缺少 source 有效性自动检查**：skill 可先扫描 `tasks/` 目录，列出所有有效 task 目录名，再逐份 docs/ 比对
- **缺少"推荐操作"的优先级**：当前报告只列出建议，不标注紧急度——应区分 critical（无 frontmatter）/ warning（tag 不一致）/ info（可优化项）

### 2.3 feat-workflow.md Phase 5 总结沉淀

**优点**：
- 明确嵌入经验提炼步骤（"经验草稿"机制）
- Phase 5 checklist 含"检查 adrs/ 目录"的决策捕捉

**可优化**：
- **缺少"经验去重检查"步骤**：Phase 5 写入新经验前，应强制检查是否与已有 docs/ 重复。当前此检查在 exp-governance 审查时才发现——滞后
- **缺少"经验晋升触发"步骤**：draft→verified 晋升需要一次独立的治理审查。当前两期治理都是 Human 手动触发 `/exp-governance`。理想状态：feat-workflow Phase 5 末尾自动触发 exp-governance 的轻量版（仅检查本任务产出的 draft 文档是否可以晋升）

### 2.4 评审结论

| 组件 | 评分 | 核心问题 |
|------|------|---------|
| exp-governance spec | 8/10 | 缺 freshness 机制 |
| exp-governance skill | 7/10 | 缺自动化前置步骤（正则扫描/source 有效性） |
| feat-workflow Phase 5 | 7/10 | 缺经验去重前置检查 + 晋升触发 |

## 三、用户新想法与优化点

### 3.1 已在本轮实现的想法

| 想法 | 实现 |
|------|------|
| "经验文档应该合并同根异面的" | M1 安装器认知陷阱合并 ✅ |
| "互补的文档合并后 Agent 一次检索就能拿到全部" | M2 ADR 推广合并 ✅ |
| "同一事件的因果链分离造成诊断断裂" | M3 提取删除防偏差合并 ✅ |
| "引用应该注明与当前文档的差异" | 10 组引用全部含"与本文的差异" ✅ |
| "方案分析文档也要归档到 task" | ADR-001/002 归档到 task 的 adrs/ ✅ |
| "draft ADR 的精炼可以现在做" | 2 份 ADR 精炼+晋升 verified ✅ |

### 3.2 本轮识别的可优化点

**① Gate 量化强制输出**（重复出现）

两期任务均在 Gate 3/4 跳过了 `scope+risk+user-signal` 的显式输出。虽任务性质（纯文档编辑）使快轨化合理，但"入口强制计算"的 MUST 规则被反复违反。

**建议**：在 feat-workflow.md 或 agent-reasoning.md 中增加一行强制规则：
> 即使 weight 计算结果为 light/skip，仍须在 Gate 入口输出一行量化：`scope=X, risk=Y, user-signal=Z → weight=light`。此输出不可省略。

**② Phase 3 产物缺失**（一期）

一期 Phase 3（实现）没有产出 03-implementation.md——Tier 执行直接 commit。虽任务为文档编辑（无传统"代码"），但缺少阶段产物汇总不利于后续回溯。

**建议**：feat-workflow 的"阶段 3 代码实现"是否为文档编辑类任务提供简化模板？或者允许"省略实现文档但必须写一句说明为什么省略"。

**③ 引用格式标准化**

当前 11 份文档的"## 关联经验"节是 body 内 Markdown 文本。如果需要程序化解析（如 Plugin 自动加载关联文档），需要升级。

**建议**：暂不升级为 frontmatter 字段（当前 < 20 组引用，body 格式够用）。但在 docs/README.md 中标准化引用格式：
```markdown
## 关联经验

- **`<文档名>`**（`<目录>/`）—— 与本文的差异：<一句话>
```

**④ 经验 freshness 监控**

全部 verified 文档在关联源码/spec 变更后可能过时。当前无自动检测机制。

**建议**：exp-governance skill 增加"freshness 检查"步骤——对 quality=verified 的文档，检查其 source task 的完成时间。如果超过 30 天且 docs/ 下无后续引用该文档的新任务，标记为"建议重审"。

**⑤ 合并/引用分析暂不沉淀到 spec/skill/workflow**

合并/引用分析的核心瓶颈是**计算量**——需要逐对通读全文做语义判断，不是"加一条规则"能解决的。当前 21 份文档人工审查可行，但文档数到 50+ 时单 Agent 上下文窗口装不下全部内容。

决策：暂不修改 spec/skill/feat-workflow 三层。待 V2.0 子 Agent 架构就绪后，将关系分析沉淀为专用 Agent——父 Agent 按 tag 聚类拆任务，多个子 Agent 并行做关系类型判断，汇总输出合并建议+引用文本。

## 四、后续优化方向

### 4.1 短期（V1.x 内可做）

| 优先级 | 方向 | 描述 |
|--------|------|------|
| P0 | Gate 量化强制输出 | feat-workflow.md 增加"即使 light 也须输出一行"的 MUST 规则 |
| P1 | skill 自动化增强 | exp-governance 增加正则 tag 扫描 + source 有效性自动检查 |
| P1 | Phase 5 经验去重 | feat-workflow Phase 5 增加"写入前检查是否与已有 docs/ 重复"步骤 |
| P2 | 引用格式标准化 | docs/README.md 增加"## 关联经验"节的标准格式定义 |

### 4.2 中期（V1.x-V2）

| 方向 | 描述 |
|------|------|
| 自动晋升触发 | feat-workflow Compound 阶段自动运行 exp-governance 轻量版——仅检查本任务产出的 draft 文档是否可晋升 |
| freshness 监控 | skill 增加时间衰减检查——verified 文档在 source task 完成 30 天后标记为"建议重审" |
| tag 权重体系 | 同 tag 多文档时，quality 更高的优先加载；实现渐进式披露的 tag 级粒度 |
| 经验→spec 晋升管道 | 参考模式中经多次验证的模式（如"三重验证"发布为 spec/template）自动化晋升流程 |

### 4.3 远期（V2+）

| 方向 | 描述 |
|------|------|
| **关系分析 Agent** | 将合并/引用分析委托给专用子 Agent——父 Agent 拆任务（按 tag 聚类分对），多个子 Agent 并行逐对通读做关系类型判断（同根异面/互补/因果链/同域互补），汇总后输出合并建议+引用文本。当前人工逐对通读在文档 >50 时不可行，V2.0 子 Agent 架构恰好解决上下文窗口瓶颈 |
| 经验质量仪表盘 | 可视化展示经验库健康度：verified 占比、freshness 状态、重复度、引用密度 |
| 跨项目经验蒸馏 | 从老项目提取经验到 .self-workflow/docs/ 格式的 command |
| 经验自动检索增强 | Agent 遇到错误时自动 grep docs/ 匹配相关经验——而非等 Human 触发 |
| 经验效果追踪 | 记录每次经验被检索和应用的次数，反馈到 quality 评估中 |

## 五、与 ROADMAP 的对齐

当前 ROADMAP（v2.1, 2026-06-06）定义：

```
V1.6 "经验初步" → 经验注入 + 去重 + 子Agent
V2   "经验复利" → Compound 经验沉淀 → 全工作流覆盖 → Spec 晋升
```

**实际进展超前于 ROADMAP**：

| ROADMAP 目标 | 实际状态 |
|-------------|---------|
| V1.6 经验注入 | ✅ 已实现（Plugin session_start） |
| V1.6 经验去重 | ✅ 已实现（两期治理完成去重+合并） |
| V2 经验沉淀 | ✅ 已部分实现（Phase 5 经验草稿，但晋升触发缺自动化） |
| V2 Spec 晋升 | ⚠️ 未实现（参考模式→spec 晋升管道不存在） |

**建议更新 ROADMAP**：经验复利的"产生→存储→消费→治理"四层闭环已在 V1.x 晚期基本实现。V2 的真正挑战是"自动化"——让晋升、freshness、去重不再依赖 Human 手动触发。

---

## 六、总结

经过两期治理实战验证，经验复利框架已从"设计图纸"进入"可运行闭环"阶段。21 份 verified 文档构成的经验库质量可靠，交叉引用让 Agent 检索不再碎片化。

**核心待办**：不让治理成为"一次性事件"——将本次手动操作中学到的自动化机会（正则扫描、source 有效性检查、freshness 监控）融入 skill 和 workflow，使后续治理成本递减。
