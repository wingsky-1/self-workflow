# V1.5 验收标准

> 版本：v1.0
> 最后更新：2026-06-06
> 用途：每次自举开发完成后，逐项回顾开发过程是否达标。
> 来源：`V1.5-需求设计.md` §8 + `V1.5-质量保障需求设计.md` + V1/V1.5 实施经验

---

## 使用说明

完成一个 feat 工作流后，按以下三层防线逐条检查。每条三个状态：

| 标记 | 含义 |
|------|------|
| ✅ | 通过——有明确证据 |
| ⚠️ | 部分通过——有缺陷但可接受 |
| ❌ | 未通过——需要修复或记录为已知缺陷 |

**证据栏**填写具体文件路径、Git tag、或操作记录。不能用"看起来没问题"代替。

## 第一层：流程防错（4 项）

> 目标：让遵循流程的人不容易犯错。
> 对应缺陷：D1 线性不可回溯、D2 缺方向 Gate、D3 决策遗漏、D4 Gate 标准模糊

| # | 检查项 | 如何验证 | 证据 |
|---|--------|---------|------|
| P1 | **Checkpoint 回溯可用**：每个 Gate 通过后创建了 Git tag | `git tag | grep <workflow-id>` 应有 5 个 tag（ph1~ph5） | |
| P2 | **方向审查已执行**：方案设计 Gate 包含方向审查（架构一致性、替代方案、影响范围） | 审查报告或 Gate 自审记录中有方向审查结论 | |
| P3 | **决策捕捉每阶段执行**：每个阶段末尾 Agent 自问"有需要记录的决策吗？" | 各阶段产物末尾有"决策捕捉检查"章节 | |
| P4 | **Gate 重量已量化**：每个 Gate 执行前计算了 scope + risk + user-signal 总分 | Gate 审查记录中有三维分值和最终 weight（skip/light/full） | |

### P1 详细标准

```
git tag 命名：<workflow-id>-ph<N>-<英文阶段名>-gate-passed
示例：feat-user-login-20260606-ph1-analysis-gate-passed
数量：5 个（对应 5 个 Gate），缺一不可
```

### P2 详细标准

方向审查至少回答四个问题，每个有明确的 ✅/⚠️/❌ 结论：
1. 方案与项目现有 ADR 是否一致？
2. 有没有更简单的替代方案？
3. 有没有遗漏的约束条件？
4. 这个方案的影响范围是什么？

### P3 详细标准

判断标准（任一满足即需要记录）：
- 做了架构选择 → 需要 ADR（通过 `/adr` 命令创建）
- 遇到了陷阱并解决了 → 需要经验记录
- 偏离了最初设计 → 需要 ADR
- 只是按计划执行 → 不需要记录（显式标注"无新增决策"）

### P4 详细标准

```
三维分值来源：
  scope:   single-file(-1) / multi-file(0) / cross-module(+1)
  risk:    typo-config(-1) / logic-change(0) / architecture(+1)
  user-signal: quick-mode(-1) / default(0) / full-review(+1)

结果映射：
  ≤ -1 → skip（跳过所有审查）
  = 0  → light（仅程序化验证，跳过对抗性审查）
  ≥ 1  → full（完整审查：方向 + 程序化 + 对抗性 + 行为）
```

> **注意**：量化结果覆盖 Gate 在 `feat-workflow.md` 附录表中的声明权重。Human 可覆盖（要求 full even if ≤0）。

## 第二层：Agent 约束（3 项）

> 目标：让 Agent 行为可控、可预期。
> 对应缺陷：Agent 不质疑方向、交互不规范、职责混淆

| # | 检查项 | 如何验证 | 证据 |
|---|--------|---------|------|
| A1 | **质疑报告已输出并经 Human 确认**：方案设计入口前，Agent 输出了方向质疑+约束检查+风险提示 | 聊天记录中有质疑报告内容 + Human 的"确认"回复 | |
| A2 | **question 工具已用于所有选项选择**：没有出现纯文本列举选项的情况 | 检查整个会话中涉及"二选一"或"多选一"的交互点 | |
| A3 | **Review Agent 已被实际调用**：至少在设计审查 Gate（full）中，对抗性审查由 Review Agent 执行，而非主 Agent 自审 | 审查报告由 Review Agent 输出（含 behavior 维度），或自审时有显式说明"Review Agent 不可用" | |

### A1 详细标准

质疑报告必须包含三个段落，每个有实质内容（不能是"无问题"）：
1. **方向质疑**：方案方向是否合理？合理说明理由，不合理提替代方案
2. **约束检查**：列出已识别约束 + 可能遗漏的约束
3. **风险提示**：至少 1 个风险点及其缓解措施

### A2 详细标准

检查清单（逐项勾）：
- [ ] Phase 1 Gate 确认 → 用了 question？
- [ ] Phase 2 质疑确认 → 用了 question？
- [ ] Phase 2 设计审查确认 → 用了 question？
- [ ] 其他交互中的选项选择 → 用了 question？

### A3 详细标准

```
合格：审查报告中有 Review Agent 的输出（含 behavior: passed/warning/failed）
可接受：主 Agent 自审，但在 Gate 记录中显式标注
        "Review Agent 不可用，由主 Agent 执行自审"
不可接受：主 Agent 自审但未标注，或直接跳过对抗性审查
```

## 第三层：产物验证（4 项）

> 目标：让产出自洽、可追溯。
> 对应缺陷：D6 缺引用工件、D7 模板分批提取、D8 task.yaml 缺失

| # | 检查项 | 如何验证 | 证据 |
|---|--------|---------|------|
| T1 | **task.yaml 从启动即存在**：工作流启动时（Phase 1 开始前）已创建，含完整的 structure + artifacts 字段 | `task.yaml` 文件存在，`created` 日期 ≤ Phase 1 开始日 | |
| T2 | **5 个阶段产物完整且格式匹配模板**：01~05 全部存在，每个按对应模板填写，必填字段无缺失 | 对比产物 frontmatter 中的 validation.required-fields 与实际内容 | |
| T3 | **交叉引用无断裂**：task.yaml 中引用的文件全部存在；ADR 间互相引用有效 | Compound 阶段交叉引用检查通过（无 broken ref） | |
| T4 | **经验双级沉淀已完成**：task 级（05-summary.md）+ doc 级（.self-workflow/docs/）均有产出 | `.self-workflow/docs/` 下有本次任务对应的经验文档 | |

### T1 详细标准

task.yaml 启动时即有字段（非事后补）：
```yaml
name: <slug>
title: <描述>
status: in_progress → completed
created: <YYYY-MM-DD>
updated: <YYYY-MM-DD>
structure:
  root: [task.yaml, workflow.yaml]
  adrs: [...]    # 随 ADR 创建增量更新
  logs: [...]
  artifacts: [...] # 随阶段产物增量更新
milestones: [...]  # 随阶段推进增量追加
artifacts: [workflow.yaml, artifacts/, ...]
cross-refs: [...]  # 跨文档引用
```

### T2 详细标准

| 产物 | 模板 | 必填字段（缺一不可） |
|------|------|-------------------|
| 01-analysis.md | analysis-template.md | 需求概述、功能清单、验收标准（Given-When-Then）、不纳入范围 |
| 02-design.md | design-template.md | 架构决策记录（含 ≥2 备选方案）、接口设计、数据模型 |
| 03-implementation.md | implementation-template.md | 变更清单、自检结果、与设计对齐检查 |
| 04-verification.md | verification-template.md | 验收标准逐条验证、边界条件覆盖 |
| 05-summary.md | summary-template.md | 回顾（得与失）、经验草稿、决策捕捉 |

### T3 详细标准

Compound 阶段交叉引用检查清单：
- [ ] task.yaml 中 `structure.adrs` 列出的所有 ADR 文件都存在
- [ ] task.yaml 中 `structure.artifacts` 列出的所有产物文件都存在
- [ ] task.yaml 中 `cross-refs[].source` 指向的文件都存在
- [ ] ADR 文件间互相引用有效
- [ ] 删除的文件已从所有引用中移除

### T4 详细标准

```
task 级：tasks/<id>/artifacts/05-summary.md
  - 包含：本次的"得与失"、经验草稿、决策记录
  - 读者：后续查看该任务历史的 Agent

doc 级：.self-workflow/docs/<feature>-实施经验.md
  - 包含：跨任务可复用的教训、最佳实践、发现的框架缺陷
  - 读者：所有 Agent、所有任务
  - quality: draft（可后续晋升为 confirmed）
```
