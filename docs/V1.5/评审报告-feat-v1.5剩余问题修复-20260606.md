## 4 个 ❌ 详细分析

### P1：Checkpoint 不完整（第一层 · 流程防错）

**现象**：只创建了 3/5 个 Git tag（ph1/ph3/ph5），缺少 ph2（设计审查）和 ph4（验证审查）的 checkpoint。

**需要的 tag**：
- [x] `feat-v1.5剩余问题修复-20260606-ph1-analysis-gate-passed`
- [ ] `feat-v1.5剩余问题修复-20260606-ph2-design-gate-passed` ← 缺失
- [x] `feat-v1.5剩余问题修复-20260606-ph3-implementation-gate-passed`
- [ ] `feat-v1.5剩余问题修复-20260606-ph4-verification-gate-passed` ← 缺失
- [x] `feat-v1.5剩余问题修复-20260606-ph5-summary-gate-passed`

**根因**：feat-workflow.md Gate 步骤中没有把"创建 git tag"写为强制性步骤。Agent 在 3 个 Gate 记得做了，2 个忘了。Gate 通过条件应该包含"Git tag 已创建"检查项。

**框架改进方向**：每个 Gate 的"通过条件"中增加 `[ ] Git tag <workflow-id>-ph<N>-<name>-gate-passed 已创建`

---

### P3：决策捕捉 Agent 不自发执行（第一层 · 流程防错）

**现象**：Human 至少 2 次明确提醒"为什么不记录决策"（AD-2 设计修正时、.opencode 约束时）。Agent 在阶段 1/2 产物中写了"决策捕捉检查"章节，标注"待方案设计"，但从未主动创建 ADR 文件——直到 Human 提醒。

**Human 的两次提醒**：
1. "上面我做的决策为什么不记录" → Agent 才创建 ADR-001/002/003
2. "还有对于.opencode的约束也要记录" → Agent 才创建 ADR-004 和 installer-错误经验.md

**根因**：feat-workflow.md 的检查清单说的是"决策捕捉检查"（检查有没有），但没说"创建 ADR 文件"（执行创建动作）。Agent 把"写了检查结论"等同于"完成了决策捕捉"。

**框架改进方向**：
- 检查清单从"决策捕捉已执行"改为"ADR 文件已创建（如本阶段有决策）"
- 每个阶段的"输出产物"中增加 ADR 作为隐性产出（如有决策）

---

### V1-3："决策记录是额外工作"问题重现（第四层 · V1 回归）

**现象**：这是 P3 的历史根因。V1 时期就发现 Agent 把决策记录当作"额外工作，等 Human 说再做"。V1 Adapter 方案试图通过在 SKILL.md 中塞入指令来解决——当前验证：**仍未解决**。

**为什么 V1 的修复没生效**：V1 试图通过"Agent 行为指令"来解决问题。但 Agent 的行为指令和项目约束混在一起（如本次错误地把安装器规则写入 agent-reasoning Skill），且检查清单没有把"创建 ADR"和"检查有没有决策"区分开。

**框架改进方向**：决策捕捉不应仅靠 Agent 自觉——需要在工作流层面把"决策→ADR"绑定为必然步骤。每个阶段结束时，如果有架构选择/设计偏离/Human 偏好，Agent 必须产出 ADR 文件（而非仅"标注有决策待记录"）。

---

### V1-6：目录职责混淆重现（第四层 · V1 回归）

**现象**：Agent 直接修改了 `.self-workflow/configs/guides/feat-workflow.md`，后续安装器 --force 覆盖导致修改丢失。V1 ADR-004 已明确定义三层架构和修改路径，但 Agent 没有查阅即行动。

**为什么 V1 的修复没完全生效**：ADR-004 的规则写在 ADR 文件中（需要 Agent 主动查阅），而没有在 Agent 每次改文件前的必经路径上拦截。AGENTS.md 说了"不要修改.opencode"，但没说 `.self-workflow/configs/` 也是同样规则——这是一个覆盖漏洞。

**框架改进方向**：
- AGENTS.md 扩展覆盖所有安装器管理目录（已完成本次修复）
- 更长远的：Agent 修改 `.self-workflow/configs/` 下文件时，自动检查是否有对应的安装器模板源并提示

## 框架成熟度评估

| 防线 | 设计 | 实际执行 | 差距 |
|------|------|---------|------|
| M0 框架就绪 | ✅ | ✅ | 无 |
| 流程防错 | 4 项机制 | 1 项完全执行，3 项有缺口 | 检查清单有但无强制力 |
| Agent 约束 | 3 项机制 | 1 项完全执行，2 项部分执行 | Review Agent 调用的环境稳定性 |
| 产物验证 | 5 项机制 | 4 项完全执行 | 交叉引用的文件路径链接 |
| 历史回归 | 20 项监控 | 2 项明确重现 | V1-3 和 V1-6 说明"文档型修复"的局限 |
| 自举闭环 | 2 项 | 1 项完成 | 新会话实测流程缺失 |

**核心结论**：框架的设计方向正确——5 层防线覆盖了主要风险。但执行层有两个系统性缺口：
1. **检查清单 ≠ 强制执行**：Agent 把"清单有这一项"等同于"完成了"，但清单只检查不驱动
2. **文档型修复不可靠**：V1-3 和 V1-6 都是"V1 用文档修了但 V1.5 Agent 仍然重现"——说明单靠文档无法改变 Agent 行为
