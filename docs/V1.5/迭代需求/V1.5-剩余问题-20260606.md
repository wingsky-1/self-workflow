## 一、本次自举验证暴露的框架缺陷（4 项 · 需短期修复）

这些问题在 `/feat` 命令自举实施中首次被系统性地暴露——不是在设计阶段预见的，而是在实际跑通全流程后才发现的。

### F1：Gate 声明权重 vs 量化结果优先级冲突

**现象**：设计审查 Gate 在 `feat-workflow.md` 附录表中声明 `weight: full`，但按三维量化公式计算得 0 分 → `light`。当前文档未定义二者冲突时的优先级。

**影响**：不同 Agent 可能做出不同判断——有的看声明权重（full → 对抗性审查），有的看量化结果（light → 跳过对抗性）。行为不可预期。

**建议**：
1. 在 `feat-workflow.md` 中显式声明：**量化结果覆盖声明权重**。声明权重仅是"典型场景默认值"
2. 若 Human 要求 full（即使量化 ≤0），Human 可覆盖——已在 P4 注意事项中定义

**修复位置**：`.self-workflow/configs/guides/feat-workflow.md` Gate 重量量化章节

---

### F2：Review Agent 调用无强制执行机制

**现象**：4 个 Gate（分析/设计/实现/验证）全部由主 Agent 自审，Review Agent 一次都未被 `task(subagent_type="review-agent")` 调用。不是 Review Agent 不存在——`.opencode/agents/review-agent.md` 文件完好——而是流程中没有强制触发。

**影响**：自审的客观性存疑。V1 经验 #2 明确指出"方向判断比实现审查更重要"——缺少独立审查意味着方向性错误要到 Human 介入才能发现。

**建议**（分阶段）：
- **短期（V1.5）**：在 `feat-workflow.md` 各 Gate 步骤中明确写入 `task(subagent_type="review-agent", prompt="审查 <阶段> 产物...")`，让 Agent 无法跳过
- **长期（V2）**：Gate 定义为可被 Hook 系统自动触发的独立单元，不依赖 Agent 主动调用

**修复位置**：`.self-workflow/configs/guides/feat-workflow.md` 各 Gate 的对抗性审查步骤

---

### F3：双级经验模型未在 workflow 中体现

**现象**：Phase 5（总结沉淀）的检查清单只提"经验草稿"，未区分 task 级（上下文保留）和 doc 级（跨任务复用）。本次实施中 task 级经验在 Phase 5 完成，doc 级经验是事后由用户提醒才补充的（"也要沉淀经验，这里是一级索引，task是二级"）。

**影响**：doc 级经验可能被遗漏——Agent 默认只做 task 级，不做 doc 级。

**建议**：
1. 在 `feat-workflow.md` Phase 5 检查清单中拆分为两条：
   - `[ ] task 级经验：artifacts/05-summary.md 含本次得与失`
   - `[ ] doc 级经验：判断是否有跨任务可复用经验 → 有则写入 .self-workflow/docs/`
2. doc 级经验的判断标准：本任务是否遇到了框架缺陷？是否发现了可推广的模式？是否验证了一个新机制？

**修复位置**：`.self-workflow/configs/guides/feat-workflow.md` 阶段 5 检查清单

---

### F4：`/adr` 命令的定位偏移

**现象**：V1.5 框架融入方案将 `/adr` 定位为"用户或 Agent 触发，快速创建 ADR"。但 V1.5 验收标准 v1.2 已重新定位：**Agent 自主归档是主路径，`/adr` 是兜底**。`/adr` 命令的设计（含类型参数 simple/complex/review）仍是"手动触发工具"的定位。

**影响**：命令设计与实际使用模式不一致。Agent 在自主归档时不需要"交互式填写模板"——它已经在设计文档中写好了 ADR 内容，只需要写入文件。

**建议**（分两条路径）：
1. **`/adr` 命令**：去掉类型参数（`simple|complex|review`），由 Agent 自行判断模板复杂度。增加 `--auto` 模式——Agent 已准备好 ADR 内容时，跳过交互式收集步骤
2. **Agent 自主归档指引**：在 `feat-workflow.md` 或 `agent-reasoning` Skill 中增加 ADR 自主创建的判断标准和操作指引

**修复位置**：`.opencode/commands/adr.md`、`packages/installer/templates/commands/adr.md`

## 三、经验分级与渐进式披露（todo.md #9 · 需设计）

### 现状

`.self-workflow/docs/` 当前仅有 `*-实施经验.md` 一种模式。所有经验（流程设计、Agent 行为、项目结构、协作模式）混在一个文件中。

### 设计方向

**分类原则**（从实际产出中归纳）：

| 类别 | 说明 | 示例 | Agent 何时读 |
|------|------|------|------------|
| `*-实施经验` | 从具体任务中提取的教训和模式 | "安装器从项目根目录运行" | 启动同类任务时 |
| `*-参考模式` | 可复用的解决方案模板 | "双文件分发模式：安装器模板 + 运行时" | 设计阶段——问"之前怎么解决类似问题？" |
| `*-错误经验` | 遇到过的陷阱和错误模式 | "从 packages/installer/ 运行安装器导致路径错误" | 任务启动前——预防性加载 |

**渐进式披露机制**：

```
任务启动 → Agent 扫描 .self-workflow/docs/ 文件名
  ├─ 匹配关键词（任务描述 vs 文件名/标签）
  └─ 匹配到 → 加载对应文档的摘要（不加载全文）

设计阶段 → Agent 主动检索 *-参考模式
  └─ 问："之前解决过类似架构问题吗？"

遇到错误 → Agent 检索 *-错误经验
  └─ 问："这个错误模式之前遇到过吗？"

Phase 5 → Agent 将新经验写入对应分类
```

### 待决策

- [ ] 三种分类是否足够？是否需要"设计模式""反模式"等更细的分法？
- [ ] 渐进式披露的实现方式：Agent 主动检索 vs 系统自动注入（Hook）？
- [ ] 文件名约定：`<领域>-<分类>.md` vs frontmatter `category` 字段？
