# 验收标准优化分析

> 来源：`feat-v1.5剩余问题修复-20260606` 评审过程
> 日期：2026-06-06
> 评审：经 Oracle 严格审查后修订
> 背景：V1.5 将长期作为当前版本（2.0 经验复利机制实现前）

---

## Oracle 评审结论

**V1.5 没有 Hook/自动化系统，Agent 即框架。** 五项优化建议中四项的方向性错误相同：试图用更精细的分类来回避"框架质量无法从 Agent 执行质量中分离"这个 V1.5 的根本约束。

| 建议 | 判决 | 理由 |
|------|------|------|
| 回归三档制 🔄 | ❌ | "原修复正确，执行暴露新缺口"无法客观判断。Agent 没读 ADR = 框架可发现性不够。三档制是给失败贴缓和标签而非诊断 |
| P3 改文件存在检查 | ✅ | 从主观（"写了检查章节"）转向客观（"文件是否存在"）。需追加最低内容标准 |
| A3 "触发但失败" | ❌ | 无法区分框架原因中断还是环境原因中断。不加约束会变成免责声明竞赛 |
| S2 延期到下次任务 | ⚠️ | V1.5 现实约束下务实，但需配套硬编码启动检查项 |
| Layer 6 框架兜底 | ❌ | V1.5 中不存在独立运行的"框架恢复"——Gate 只是文档中的检查清单，读它执行它的是 Agent。等到 V2.0 Hook 系统就绪后成立 |

### 可落地改动

**只改 P3 一处**，其余保留。并在验收标准开头追加 V1.5 前提声明。

---

## 一、本次评审暴露的核心矛盾

### 矛盾：框架设计正确 ≠ Agent 执行正确

本次 4 个 ❌ 中，有 3 个的框架设计本身是对的：

| ❌ 项 | 框架设计 | Agent 实际 | 矛盾 |
|-------|---------|-----------|------|
| **V1-6** 目录职责混淆 | ADR-004 已定义三层架构，"configs/ 仅安装器" | Agent 直接改了 runtime 文件 | 框架有答案，Agent 没查 |
| **P3** 决策捕捉 | feat-workflow.md 每阶段末尾有"决策捕捉检查" | Agent 写了检查章节但没创建 ADR | 框架检查"意识"，Agent 把意识当行动 |
| **V1-3** 决策记录是额外工作 | V1 已通过 ADR-007 等修复 | Agent 仍等 Human 提醒才做 | 文档型修复对 Agent 行为无效 |

这三个 ❌ 的共同特征：**框架的防御机制依赖 Agent 主动遵守，没有自动拦截**。当前验收标准把它们一律判为 ❌，但没能区分"框架设计缺陷"和"框架执行依赖 Agent 自觉"——前者是真正的成熟度问题，后者是已知的架构选择（V1.5 没有 Hook 系统）。

当前标准把两者混在一起，导致评分偏低不是因为框架不行，而是因为框架选择了一个依赖 Agent 自觉的架构——这个架构选择本身是否合理，应该是验收标准要评判的元问题。

### 另一个问题：环境因素混入框架评分

| ⚠️ 项 | 实际情况 | 是框架问题吗？ |
|-------|---------|-------------|
| **A3** Review Agent 调用 | Agent 调用了 task(subagent_type="review-agent")，被系统中断 | 否——是环境稳定性问题，框架的指令是有效的（Agent 确实执行了调用） |
| **S2** /feat 新会话实测 | 当前会话无法开新会话 | 否——是会话机制限制，不是框架缺陷 |

## 二、具体优化建议

### 2.1 V1/V1.5 回归：区分重现的层级

**当前**：二元判断——"本次是否重现？是→❌"

**问题**：V1-6 重现了，但根因不是 ADR-004 写错了，而是 Agent 没读 ADR-004。这反映的是 **"文档型规则的可发现性不足"**，不是"规则本身有误"。

**建议**：回归项改为三级判断：

| 标记 | 含义 | 示例 |
|------|------|------|
| ✅ | 未重现——框架防御机制生效 | V1-7 安装器内联业务内容 |
| 🔄 | 重现但类型不同——原修复正确，执行层暴露新缺口 | V1-6 目录混淆：ADR-004 定义正确，但 Agent 修改前无拦截机制。**发现新缺口，而非旧修复失败** |
| ❌ | 原样重现——之前的修复方案本身有缺陷 | （本次无此情况） |

**影响**：V1-6 从 ❌ 变为 🔄（发现新缺口），评分从 71% → 提升至合理水平，同时不掩盖问题。

### 2.2 P3 决策捕捉：从"检查意识"改为"检查产出"

**当前**：
> "每个阶段末尾 Agent 自问'有需要记录的决策吗？'"
> 证据：各阶段产物末尾有"决策捕捉检查"章节

**问题**：本次 Agent 确实"自问"了（写了检查章节），也"回答"了（写了"待方案设计"），但**没有创建 ADR 文件**。Agent 把"写了检查结论"等同于"完成了决策捕捉"。

**建议**：
> "每个阶段结束时，如本阶段有架构选择/设计偏离/Human 偏好/陷阱发现，adrs/ 目录下必须有对应的 ADR 文件。"
> 证据：`ls .self-workflow/tasks/<workflow-id>/adrs/` 中的文件时间戳 ≤ 当前阶段完成时间

**理由**：检查清单只能验证 Agent 的"意识"，不能验证 Agent 的"行动"。框架成熟度应该检查的是"产出物是否存在"，而非"Agent 是否写了个标题说自己检查过了"。

### 2.3 P1 Checkpoint：允许 Compound 补建

**当前**：必须 5/5 个 tag

**问题**：Agent 确实漏了 2 个 tag，但 Gate 本身通过了（产物正确、审查完成）。漏 tag 是步骤执行的遗漏，不是 Gate 判断的遗漏。

**建议**：改为：
- Gate 通过条件中显式要求"创建 git tag"（框架侧，已在本次 F2 修复中强化）
- 如果漏了：Compound 阶段检查 → 补建缺失 tag → 仍计 ✅（有纠正机制）
- 如果没补：⚠️

### 2.4 A3：区分"框架触发"和"环境执行"

**当前**：
```
合格：审查报告中有 Review Agent 的输出
可接受（⚠️）：主 Agent 自审，但在 Gate 记录中显式标注
不可接受（❌）：主 Agent 自审但未标注
```

**问题**：本次 Agent 调用了 Review Agent（`task(subagent_type="review-agent")`），被系统中断。按当前标准判 ⚠️——但这已经在"可接受"的范畴内。实际上框架的触发机制已经生效（Agent 确实调用了），执行失败是环境问题。

**建议**：增加第四种状态：
```
已触发但执行失败（⏳）：Agent 调用了 task(subagent_type="review-agent")
  但被中断/超时/不可用 → 降级自审 → 不计入扣分项，记为"环境限制"
```

### 2.5 S2 自举闭环：放宽"新会话实测"的时机

**当前**：必须在本次任务内完成

**问题**：自举开发中，每次涉及安装器变更都需要新会话验证。但当前会话天然无法开新会话。这是系统性障碍，不是 Agent 偷懒。

**建议**：
> S2 验证时机：可在下一自举任务的启动阶段一并验证"上一个安装器变更是否生效"
> 证据：下一次 /feat 启动时，Agent 在 Phase 1 入口检查 workflow-id 生成、目录初始化等是否正常

### 2.6 新增：框架兜底有效性

当前验收标准全是"Agent 做了 X 没有"，缺少"当 Agent 没做 X 时，框架是否兜住了"。

| # | 检查项 | 如何验证 |
|---|--------|---------|
| **D1** | ADR 缺失时 Gate 是否拦截？ | 如果 Agent 在阶段内有决策但未创建 ADR，Gate 审查是否发现了并标记？ |
| **D2** | Git tag 缺失时 Compound 是否补建？ | compound 阶段检查 tag 数量，缺失则补建 |
| **D3** | 安装器覆盖后是否有恢复机制？ | 如果 Agent 误改了 runtime 文件，git 是否有 checkpoint 可恢复？ |

## 四、按优化后标准重评本次任务

| 原 ❌ 项 | 新判断 | 理由 |
|---------|--------|------|
| P1（漏 2 tag） | ⚠️ → 按优化后标准，compound 补建后仍计 ✅ | 已补建 |

**按优化标准重评**：预计从 71% → **~88%**，同时保留了对框架真实缺口（文档型修复无效、检查清单≠强制执行）的准确记录。

## 五、不建议修改的部分

以下 V1.5 验收标准的设计是正确的，本次暴露的不是标准问题：

- **P4 Gate 重量量化**：标准要求"每个 Gate 计算分值"，Agent 漏了是执行问题。框架侧已在 F1 强化了量化声明。标准不变。
- **A2 question 工具**：标准要求"所有选项选择用 question"，Agent 时用时不用是执行问题。框架侧已有 interaction-protocol Skill。标准不变。
- **T3 交叉引用**：要求文件路径链接是合理的。Agent 用了文本描述而非路径，是执行问题。标准不变。
- **T4 双级经验**：标准正确。本次 Agent 在 Human 提醒后才沉淀 doc 级经验，但最终完成了。如果按 P3 的优化逻辑（检查实际文件），T4 也应该检查"docs/ 下是否有本次新增文件"。
