# V1.5 质量保障需求设计

> 版本：v0.1（草案）
> 对应：V1 回顾总结中的 8 个优化点，属于 V1.5 质量加固阶段
> 核心问题：**如何保障不同 Human + Agent 组合下的工作流产出质量一致？**

---

## 1. 问题域分析

### 1.1 角色能力差异矩阵

| 能力维度 | V1 合作中的 Human（当前） | 未来 Human（未知） | 差距 |
|---------|------------------------|-------------------|------|
| **架构判断力** | 强——能识别目录结构错误、职责边界问题 | 不确定——可能意识不到架构问题 | 🔴 大 |
| **原则坚持度** | 强——坚持"开发使用分离"、"安装器唯一入口" | 不确定——可能随意修改 `.opencode/` | 🔴 大 |
| **反馈及时性** | 高——Agent 走偏立即纠正 | 低——可能等很久才反馈，或默默弃用 | 🟡 中 |
| **决策记录意识** | 强——推动 ADR 归档流程 | 弱——可能不觉得需要记录 | 🟡 中 |
| **交互体验要求** | 高——要求 question 工具、总结先行 | 不确定——可能接受低质量交互 | 🟢 小 |

### 1.2 Agent 行为缺陷

| 缺陷 | 表现 | 根因 | 如何约束 |
|------|------|------|---------|
| **不质疑方向** | 接受 Human 的方向性错误（目录结构混装） | 缺少"质疑-对齐-执行"的强制流程 | 流程级约束 |
| **职责混淆** | 安装器内联业务内容 | 缺少关注点分离的硬性检查 | Gate 级检测 |
| **交互不规范** | 纯文本列举选项而非 question 工具 | 缺少行为规范检查 | 工具级约束 |
| **决策敏感度低** | 未主动记录关键决策 | 缺少触发机制 | 流程嵌入 |
| **模板提取分批** | feat-workflow.md 多次提取 | 缺少"一次性识别"的分析步骤 | 流程改进 |

### 1.3 流程缺陷（来自 V1 回顾）

| # | 缺陷 | 影响范围 |
|---|------|---------|
| 1 | 线性阶段模型不支持回溯 | 所有复杂任务 |
| 2 | 缺少方向判断 Gate | 架构一致性 |
| 3 | 决策记录是额外工作而非自然产出 | ADR 合规性 |
| 4 | Gate 降级标准模糊 | Gate 行为不可预期 |
| 5 | Quick Mode 无法中途升级 | 变更规模误判 |
| 6 | 阶段性输入缺少引用工件 | 上下文不完整 |
| 7 | 模板提取缺乏一次性识别方法 | 开发效率 |
| 8 | task.yaml 非强制 | 元数据缺失 |

## 2. 质量保障架构

### 2.1 三层质量防线

```
第一层：流程防错（Process Gate）
├─ 设计目标：让遵循流程的人不容易犯错
├─ 手段：Phase Gate 增强 + 回溯路径 + 决策嵌入
└─ 防御对象：方向性错误、架构一致性

第二层：Agent 约束（Agent Discipline）
├─ 设计目标：让 Agent 的行为可控、可预期
├─ 手段：强制质疑流程 + 检查清单 + Review Agent 互审
└─ 防御对象：Agent 的自作主张、行为漂移

第三层：产物验证（Artifact Validation）
├─ 设计目标：让产出自洽、可追溯
├─ 手段：结构化验证 + 交叉引用检查 + 元数据强制
└─ 防御对象：信息缺失、引用断裂、决策无痕迹
```

### 2.2 质量保障矩阵

| 质量维度 | 第一层（流程） | 第二层（Agent） | 第三层（产物） |
|---------|--------------|----------------|---------------|
| **方向正确性** | 架构一致性 Gate | 强制质疑阶段 | ADR 前置审查 |
| **实现正确性** | 程序化验证 | 检查清单执行 | 测试覆盖验证 |
| **行为规范性** | 交互规则入流程 | Question 工具强制 | 交互日志 |
| **决策可追溯** | 决策嵌入阶段流程 | 决策触发提醒 | ADR 元数据校验 |
| **产物完整性** | 阶段输出强制检查 | 模板强制使用 | 结构化验证 |

## 3. 具体设计

### 3.1 流程防错设计

#### 3.1.1 Phase Gate 增强——引入"方向 Gate"

**现状**：当前 Gate = 程序化验证 + 对抗性审查。对抗性审查聚焦于"做对了没有"，不检查"该不该做"。

**V1.5 设计**：在方案设计 Gate 中拆分两层审查：

```
方案设计 Gate（weight: full）
├─ Step 1: 方向审查（新增）
│   ├─ Review Agent 回答：
│   │   ├─ 这个方案与项目现有架构一致吗？
│   │   ├─ 有没有更简单的替代方案？
│   │   ├─ Human 考虑过 X 约束吗？
│   │   └─ 这个方案的影响范围是什么？
│   └─ 不通过 → 返回方案设计阶段修正
│
├─ Step 2: 程序化验证
│   └─ （同 V1）
│
└─ Step 3: 对抗性审查
    └─ （同 V1）
```

**验收标准**：
- [ ] 方向审查在方案设计 Gate 中严格执行
- [ ] Review Agent 能识别与现有架构不一致的方案
- [ ] 方向审查不通过时阻断 Gate，记录到审查报告

#### 3.1.2 回溯路径——解决线性流程不匹配现实

**现状**：从阶段 3 返回阶段 1 需要完整经过 Gate 2、Gate 1，路径过长。

**V1.5 设计**：引入 **Checkpoint 回溯机制**。

```
Checkpoint 回溯规则：
├─ 每个阶段完成后自动创建 Checkpoint
├─ 从阶段 N 回到阶段 M（M < N）时：
│   ├─ 跳过 M 到 N-1 之间已通过的 Gate
│   └─ 只验证：
│       ├─ M 阶段被修改的产物
│       └─ 受修改影响的下游阶段（Review Agent 判断）
│
├─ Checkpoint 内容：
│   ├─ 当前阶段产物的快照路径
│   ├─ 已通过的 Gate 列表
│   └─ 时间戳
│
└─ Checkpoint 存储在：
    └─ .self-workflow/tasks/<workflow-id>/checkpoints/
```

**边界条件**：
- 如果回溯涉及**架构决策变更**（ADR 被推翻），必须重新经过方案设计 Gate 的方向审查
- 如果回溯只是**实现细节修正**（代码 bug、文档补充），跳过中间 Gate
- 是否涉及架构决策变更，由 Review Agent 判断

#### 3.1.3 决策嵌入阶段流程

**现状**：ADR 创建是"总结沉淀"阶段的独立行为，实施过程中的决策容易被遗漏。

**V1.5 设计**：每个阶段末尾嵌入"决策捕捉"节点。

```
每个阶段末尾（Gate 之前）：
├─ 步骤 A：完成阶段任务
├─ 步骤 B：决策捕捉（新增）
│   ├─ Agent 自问：这个阶段有没有值得记录的决策？
│   ├─ 判断标准：
│   │   ├─ 做了架构选择？→ 需要 ADR
│   │   ├─ 遇到了陷阱并解决了？→ 需要经验记录
│   │   ├─ 偏离了最初的设计？→ 需要 ADR
│   │   └─ 只是按计划执行？→ 不需要记录
│   ├─ 如果有 → 创建轻量记录（单文件），标记为 draft
│   └─ 如果没有 → 跳过
│
├─ 步骤 C：Gate 执行
└─ 步骤 D：推进到下一阶段（或回溯）
```

**轻量记录 vs 完整 ADR**：
- **轻量记录**：一句话 + 关联文件，存入 `tasks/<id>/logs/`
- **完整 ADR**：标准 ADR 格式，存入 `tasks/<id>/adrs/`
- 轻量记录可以在总结阶段升级为完整 ADR

#### 3.1.4 Gate 重量量化标准

**现状**：Gate 降级标准过于模糊（"如果变更很小"），Agent 和 Human 都无法预期。

**V1.5 设计**：Gate weight 由三个维度量化决定：

```yaml
gate-weight-calculation:
  dimensions:
    - name: scope          # 变更范围
      levels:
        - single-file: -1   # 单文件 → 减重
        - multi-file: 0     # 多文件 → 中性
        - cross-module: +1  # 跨模块 → 加重

    - name: risk           # 变更风险
      levels:
        - typo/config: -1    # typo/配置 → 减重
        - logic-change: 0    # 逻辑变更 → 中性
        - architecture: +1   # 架构变更 → 加重

    - name: user-signal    # 用户信号
      levels:
        - quick-mode: -1     # 用户指定 quick
        - default: 0
        - full-review: +1    # 用户指定 full

  result:
    sum <= -1: skip
    sum == 0: light
    sum >= 1: full
```

**示例**：
- typo 修复：`scope=-1 + risk=-1 + user=-1 = -3` → `skip` ✅
- 新增功能（跨 2 文件）：`scope=0 + risk=0 + user=0 = 0` → `light`
- 重构核心模块：`scope=+1 + risk=+1 + user=0 = +2` → `full` ✅

### 3.2 Agent 约束设计

#### 3.2.1 强制质疑流程

**现状**：ADR-007 已补充"先质疑再委托"原则，但只是原则，没有执行保障。

**V1.5 设计**：在方案设计阶段开头插入**质疑节点**，作为阶段性入口。

```
方案设计阶段的质疑节点（强制）：
├─ 输入：需求分析文档 + 现有项目结构
├─ Agent 必须输出"质疑报告"：
│   ├─ 方案方向：我认为这个方向是否合理
│   │   ├─ 合理 → 说明理由
│   │   └─ 不合理 → 提出替代方案
│   ├─ 约束检查：我发现了以下约束可能被遗漏
│   └─ 风险提示：我认为以下风险需要关注
│
├─ 质疑报告在进入方案设计前提交给 Human 确认
├─ Human 可以：接受质疑 / 解释为什么不需要 / 调整方向
└─ 只有在 Human 确认后方可进入方案设计
```

**质疑不是形式主义**。如果 Agent 没有提出任何质疑就通过了，Review Agent 应在 Gate 中检查"Agent 是否真正思考了替代方案"。

#### 3.2.2 检查清单驱动执行

**现状**：Agent 执行阶段时按照自然语言描述的指引，可能遗漏步骤。

**V1.5 设计**：每个阶段采用**检查清单**（Checklist）驱动，而非自然语言描述。

```
阶段 3：代码实现

[ ] 对照设计文档逐项检查所有接口是否实现
[ ] 新增代码的单元测试覆盖率 ≥ 80%
[ ] lint 检查通过（0 error, 0 warning）
[ ] typecheck 通过
[ ] 检查是否引入了不必要的依赖
[ ] 检查是否需要更新 README / API 文档
[ ] 决策捕捉：这个实现过程中有需要记录的决策吗？
```

检查清单的粒度应该达到**"没有歧义"的标准**——任何 Agent 读到的理解都一样。

#### 3.2.3 Review Agent 互审

**现状**：Review Agent 只审查阶段产物，不审查主 Agent 的行为。

**V1.5 设计**：Review Agent 增加对**主 Agent 行为**的审查：

```
Review Agent 审查清单（新增）：
├─ 行为审查：
│   ├─ 主 Agent 在每个阶段是否执行了决策捕捉？
│   ├─ 主 Agent 是否按要求使用 question 工具？
│   └─ 主 Agent 的质疑报告是否充分？
│
├─ 一致性审查（增强）：
│   ├─ 本次实现与当前项目已建立的 ADR 一致吗？
│   └─ 本次实现是否遵循了 Spec（如存在）？
│
└─ 审查报告新增：
    └─ behavior: passed / warning / failed
```

### 3.3 产物验证设计

#### 3.3.1 结构化产物验证

**现状**：产物写入后没有自动验证，可能缺失字段或格式错误。

**V1.5 设计**：为每个产物定义**验证规则**，写入到产物模板中：

```yaml
# configs/templates/analysis-template.md 的 frontmatter 扩展
---
phase: 1
workflow: feat
validation:
  required-fields:
    - "需求概述"
    - "功能清单"
    - "验收标准"
  required-format:
    "验收标准": "Given-When-Then"
  min-length:
    "功能清单": 1
---
```

Agent 写入产物后，自行按验证规则检查。Review Agent 在 Gate 中也按验证规则审查。

#### 3.3.2 交叉引用一致性检查

**现状**：文件增删改时，交叉引用容易断裂（ADR 引用不存在的文件、task.yaml 未同步）。

**V1.5 设计**：在 Compound（总结沉淀）阶段执行交叉引用一致性检查：

```
交叉引用检查清单：
├─ task.yaml 中 artifacts 列出的文件是否都存在？
├─ task.yaml 中 milestones 的 ref 指向的章节是否存在？
├─ adrs/ 下的 ADR 互相引用的文件是否存在？
├─ 新增的文件是否被 task.yaml 记录？
└─ 删除的文件是否从引用中移除？
```

如果检查不通过，Compound 阶段提示修复，但不阻断。

#### 3.3.3 模板强制使用

**现状**：Agent 可以选择是否使用模板（feat-workflow.md 只建议"格式参考"）。

**V1.5 设计**：模板从"参考"升级为"必须"，在 Gate 中验证。

```
Phase Gate 验证（新增）：
├─ 阶段产物的格式是否匹配对应模板？
├─ 模板中的必填字段是否都填写了？
└─ 不匹配 → Gate 不通过
```

## 4. 与 V1 的差异总结

| 维度 | V1 | V1.5 |
|------|-----|-----|
| **Gate 结构** | 程序化验证 + 对抗性审查 | 方向审查 + 程序化验证 + 对抗性审查 + 行为审查 |
| **阶段流程** | 线性不可回溯 | Checkpoint 回溯路径 |
| **决策记录** | 总结阶段独立进行 | 每个阶段末尾嵌入决策捕捉 |
| **Gate 重量** | Agent 主观判断 | 三维量化计算（scope/risk/user-signal） |
| **Agent 约束** | 无（信任 Agent） | 强制质疑 + 检查清单 + 行为审查 |
| **产物验证** | 无自动验证 | 结构化验证规则 + 模板强制使用 |
| **交叉引用** | 靠人维护 | Compound 阶段自动检查 |
| **错误处理** | 简单记录 | 错误 → ADR → 修复 → 验证闭环 |

## 5. 未解决问题

- **Checkpoint 回溯在底层 Agent 不支持 checkpoint 机制时如何实现？** 可能需要先降级为"手动跳转"模式
- **Review Agent 互审主 Agent 行为的可靠性**——Review Agent 本身也是 Agent，谁来审 Review Agent？
- **结构化验证是嵌入模板 frontmatter 还是独立验证规则文件？** 前者耦合但简单，后者灵活但复杂
- **Gate 重量量化计算可能导致过度工程化**——如果 Human 觉得"我就是想全量审查"，量化结果是否应该被 Human 覆盖？
