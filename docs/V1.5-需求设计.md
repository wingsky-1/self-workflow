# V1.5 质量加固 —— 完整需求设计

> 版本：v1.0
> 最后更新：2026-06-06
> 前置版本：V1（已完成）
> 后续版本：V2 经验复利
> 核心目标：**让工作流达到生产可用标准**

---

## 1. 概述

### 1.1 背景

V1 完成了 Self-Workflow 的基础框架：安装器、feat 工作流、Review Agent、Session Catchup。但在实际使用中暴露了 8 个流程缺陷，导致工作流在复杂任务中不可靠。

四专家联合评审结论：**在 8 个已知缺陷上构建经验系统会导致"垃圾进垃圾出"。质量必须先于功能。**

### 1.2 8 个流程缺陷

| ID | 缺陷 | 影响 | 触发场景 |
|----|------|------|---------|
| D1 | 线性阶段模型不支持回溯 | 任务卡死、挫败感 | 从阶段 3 返回阶段 1 修改需求 |
| D2 | 缺少方向判断 Gate | 架构偏离、重构成本高 | Agent 做了"正确但方向错误"的事 |
| D3 | 决策记录是额外工作 | ADR 遗漏、决策无痕迹 | 实施过程中未及时记录 |
| D4 | Gate 降级标准模糊 | 行为不可预期 | 不知道当前 Gate 会多严格 |
| D5 | Quick Mode 无法中途升级 | 规模误判 | 看起来小但其实复杂的变更 |
| D6 | 缺少引用工件 | 上下文不完整 | Agent 不知道要参考哪些现有文件 |
| D7 | 模板提取分批 | 效率低、重复劳动 | 多次分批提取输出格式 |
| D8 | task.yaml 非强制 | 元数据缺失 | 任务目录没有统一入口 |

### 1.3 V1.5 目标

**质量目标**：让不同水平的 Human + Agent 组合都能产出一致质量的工作流产出。

**自举目标**：修复后，能用工作流在 Self-Workflow 项目自身上完成一个真实功能开发。

**约束**：不引入新的工作流类型（仍只有 feat），不引入经验系统（那是 V2）。

### 1.4 范围

| 包含 | 不包含 |
|------|--------|
| 流程防错（Gate 增强、回溯、决策捕捉） | 经验复利系统（V2） |
| Agent 约束（Skills、检查清单、行为审查） | debug/doc/review 工作流（V2） |
| 产物验证（结构化验证、交叉引用、模板强制） | Compound 经验提取（V2） |
| ADR 融入框架（`/adr` command、Skills） | Adapter V2（V2） |
| task.yaml、目录分层强制 | 平台化、多 Agent（V3+） |

## 2. 架构总览

### 2.1 三层质量防线

```
第一层：流程防错（Process Gate）
├─ 防御对象：方向性错误、架构不一致
├─ 手段：方向审查 Gate + Checkpoint 回溯 + 决策捕捉 + Gate 重量量化
└─ 对应缺陷：D1, D2, D3, D4

第二层：Agent 约束（Agent Discipline）
├─ 防御对象：Agent 自作主张、行为漂移
├─ 手段：Skills + 检查清单 + Review Agent 行为审查 + 质疑流程
└─ 对应缺陷：D2, D3, D4, D5

第三层：产物验证（Artifact Validation）
├─ 防御对象：信息缺失、引用断裂、决策无痕迹
├─ 手段：结构化验证 + 交叉引用检查 + 模板强制 + task.yaml 强制
└─ 对应缺陷：D6, D7, D8
```

### 2.2 缺陷与方案映射

```
D1 线性回溯    → Checkpoint 回溯机制（M1）
D2 方向 Gate   → 方向审查 Gate + 强制质疑流程（M1 + M2）
D3 决策记录    → 决策捕捉嵌入 + /adr Command（M0 + M1）
D4 Gate 重量   → Gate 重量量化计算（M1）
D5 Quick 升级  → 检查清单 + Gate 重量量化（M2）
D6 引用工件    → 产物结构化验证 + 交叉引用检查（M3）
D7 模板分批    → 模板强制使用（M3）
D8 task.yaml  → task.yaml 创建强制（M0）
```

## 3. 里程碑 M0：框架融入

> 将 V1 验证的 5 条 ADR 融入框架，作为后续质量加固的基础。

### 3.1 `interaction-protocol` Skill

**对应 ADR**：ADR-001（交互式问答优先）

**需求**：
- 作为 Skill 定义，安装到 `.opencode/skills/interaction-protocol/SKILL.md`
- 安装器将其纳入 MANIFEST，`self-workflow init` 时部署
- 包含以下指令：
  - 涉及 2 个以上选项时，必须使用 `question` 工具，而非纯文本
  - 询问是否继续前，先总结已完成，再说明下一步
  - 开放式提问不需要使用 question 工具

**验收标准**：
- [ ] Skill 文件存在于安装器模板 `.opencode/skills/interaction-protocol/SKILL.md`
- [ ] 安装后 Skill 存在于 `.opencode/skills/interaction-protocol/SKILL.md`
- [ ] Agent 加载 Skill 后，选项交互使用 question 工具

### 3.2 `agent-reasoning` Skill

**对应 ADR**：ADR-007（委托优先原则）

**需求**：
- 作为 Skill 定义，安装到 `.opencode/skills/agent-reasoning/SKILL.md`
- 包含以下指令：
  - 收到 Human 方向后，先独立思考合理性，再执行
  - 主 Agent 上下文只保留决策链路，不保留执行细节
  - 委托粒度判断标准（>30 秒委托，<5 秒自己做）
  - 委托后必须验证结果

**验收标准**：
- [ ] Skill 文件存在于安装器模板
- [ ] Agent 加载后能在收到方向时主动提出质疑

### 3.3 `/adr` Command

**对应 ADR**：ADR-003（关键决策即时归档）

**需求**：
- 作为 Command 定义，安装到 `.opencode/commands/adr.md`
- 功能：
  - 自动识别当前 in_progress 的 task
  - 自动递增 ADR 编号
  - 从模板创建标准格式 ADR 文件
  - 写入 `tasks/<task-id>/adrs/ADR-<编号>-<标题>.md`
  - 自动更新 task.yaml 的 artifacts 和 structure 列表

**验收标准**：
- [ ] `/adr` 命令在 30 秒内创建标准 ADR 文件
- [ ] ADR 文件包含背景、决策、理由、影响、关联章节
- [ ] task.yaml 的 artifacts 列表自动更新

### 3.4 task.yaml 创建强制

**对应 ADR**：ADR-005

**需求**：
- feat-workflow.md 的快速入门中将"创建 task.yaml"列为启动步骤
- Review Agent 在 Gate 中检查 task 目录是否包含 task.yaml

**验收标准**：
- [ ] 工作流启动指引明确要求创建 task.yaml
- [ ] Review Agent 在 Gate 中检查 task.yaml 是否存在

### 3.5 目录分层强制

**对应 ADR**：ADR-006

**需求**：安装器已内置 adrs/ + logs/ 结构，无需额外开发。

**状态**：✅ 已完成

## 4. 里程碑 M1：流程防错

### 4.1 方向审查 Gate

**对应缺陷**：D2

**需求**：
- 在方案设计 Gate 中增加"方向审查"节点
- 方向审查在程序化验证之前执行
- Review Agent 回答：
  - 方案与项目现有架构是否一致
  - 有没有更简单的替代方案
  - Human 有没有遗漏约束
  - 方案的影响范围是什么
- 方向审查不通过 → 返回方案设计阶段修正

**验收标准**：
- [ ] feat-workflow.md 的方案设计 Gate 包含方向审查步骤
- [ ] Review Agent 能识别与现有 ADR 不一致的方案
- [ ] 方向审查不通过时阻断 Gate，记录到审查报告

### 4.2 Checkpoint 回溯

**对应缺陷**：D1

**需求**：
- 每个阶段完成后自动创建 Checkpoint
- 从阶段 N 回到阶段 M（M < N）时：
  - 跳过 M 到 N-1 之间已通过的 Gate
  - 只验证：M 阶段被修改的产物 + 受修改影响的下游阶段
- 涉及架构决策变更时，必须重新经过方向审查 Gate
- Checkpoint 存储在 `.self-workflow/tasks/<workflow-id>/checkpoints/`

**验收标准**：
- [ ] feat-workflow.md 定义了 Checkpoint 回溯路径
- [ ] 从阶段 3 回到阶段 1 时，跳过已通过的 Gate 2
- [ ] 涉及 ADR 变更的回溯，重新经过方向审查

### 4.3 决策捕捉嵌入

**对应缺陷**：D3

**需求**：
- 每个阶段末尾（Gate 之前）嵌入决策捕捉步骤
- Agent 自问：这个阶段有没有值得记录的决策
- 判断标准：
  - 做了架构选择 → 需要 ADR
  - 遇到陷阱并解决了 → 需要经验记录
  - 偏离了最初设计 → 需要 ADR
  - 只是按计划执行 → 不需要
- 有决策 → 执行 `/adr` 或创建轻量记录

**验收标准**：
- [ ] feat-workflow.md 的每个阶段末尾包含决策捕捉步骤
- [ ] Agent 在阶段结束时主动询问是否有需要记录的决策

### 4.4 Gate 重量量化

**对应缺陷**：D4

**需求**：
- Gate weight 由三个维度量化计算：

| 维度 | 条件 | 分值 |
|------|------|------|
| scope（范围） | single-file / multi-file / cross-module | -1 / 0 / +1 |
| risk（风险） | typo-config / logic-change / architecture | -1 / 0 / +1 |
| user-signal（信号） | quick-mode / default / full-review | -1 / 0 / +1 |

- 总分 ≤ -1 → skip；= 0 → light；≥ 1 → full

**验收标准**：
- [ ] feat-workflow.md 的 Gate 降级描述使用量化标准
- [ ] 示例：typo 修复 → skip，新增功能 → light，重构核心 → full

## 5. 里程碑 M2：Agent 约束

### 5.1 强制质疑流程

**对应缺陷**：D2

**需求**：
- 方案设计阶段开头插入质疑节点
- Agent 必须输出质疑报告：
  - 方案方向是否合理（合理说明理由，不合理提出替代）
  - 约束检查：可能被遗漏的约束
  - 风险提示：需要关注的风险
- 质疑报告提交给 Human 确认后，方可进入方案设计

**验收标准**：
- [ ] feat-workflow.md 的方案设计阶段入口包含质疑节点
- [ ] Agent 在没有提出质疑时，Review Agent 在 Gate 中标记

### 5.2 检查清单驱动执行

**对应缺陷**：D5

**需求**：
- 每个阶段的执行内容从自然语言描述改为检查清单
- 检查清单的粒度达到"没有歧义"标准
- 每个阶段末尾包含"决策捕捉"检查项

示例——阶段 3 检查清单：

```
[ ] 对照设计文档逐项检查所有接口是否实现
[ ] 新增代码的单元测试覆盖率 ≥ 80%
[ ] lint 检查通过（0 error, 0 warning）
[ ] typecheck 通过
[ ] 检查是否引入了不必要的依赖
[ ] 决策捕捉：有需要记录的决策吗？
```

**验收标准**：
- [ ] feat-workflow.md 的 5 个阶段均以检查清单驱动
- [ ] 每个检查项无歧义、可验证

### 5.3 Review Agent 行为审查

**对应缺陷**：D2, D3

**需求**：
- Review Agent 审查范围从产物扩展到主 Agent 行为
- 审查新增维度：

```
├─ 行为审查：
│   ├─ 主 Agent 是否执行了决策捕捉？
│   ├─ 是否按要求使用 question 工具？
│   └─ 质疑报告是否充分？
│
├─ 一致性审查（增强）：
│   ├─ 实现与已有 ADR 一致吗？
│   └─ 遵循了 Spec（如存在）吗？
│
└─ 审查报告新增：
    └─ behavior: passed / warning / failed
```

**验收标准**：
- [ ] Review Agent 定义文件包含行为审查指令
- [ ] 审查报告包含 behavior 维度

## 6. 里程碑 M3：产物验证

### 6.1 结构化产物验证规则

**对应缺陷**：D6

**需求**：
- 产物模板 frontmatter 定义验证规则：

```yaml
---
phase: 1
workflow: feat
validation:
  required-fields:
    - "需求概述"
    - "功能清单"
    - "验收标准"
  required-format:
    "验收标准": "Given-When-Then"
---
```

- Agent 写入产物后自行检查
- Review Agent 在 Gate 中按规则复核

**验收标准**：
- [ ] 产物模板中包含 validation 字段
- [ ] 必填字段缺失时 Agent 自我纠正

### 6.2 交叉引用一致性检查

**对应缺陷**：D6

**需求**：
- Compound 阶段执行交叉引用检查：

```
├─ task.yaml 中 artifacts 列出的文件都存在？
├─ milestones 的 ref 指向存在？
├─ ADR 互相引用的文件都存在？
├─ 新增文件被 task.yaml 记录？
└─ 删除文件从引用中移除？
```

- 不通过时提示修复，但不阻断

**验收标准**：
- [ ] 工作流完成时执行交叉引用检查
- [ ] 断裂引用在报告中列出

### 6.3 模板强制使用

**对应缺陷**：D7

**需求**：
- 阶段产物格式必须匹配对应模板
- Gate 中验证：模板必填字段是否填写
- 不匹配 → Gate 不通过

**验收标准**：
- [ ] Gate 验证逻辑包含模板匹配检查
- [ ] 模板缺失必填字段时 Gate 阻断

## 7. 版本对比

| 维度 | V1 | V1.5 |
|------|----|------|
| **Gate 结构** | 程序化验证 + 对抗性审查 | 方向审查 + 程序化 + 对抗性 + 行为审查 |

| **Gate 重量** | Agent 主观判断 | 三维量化（scope/risk/user-signal） |
| **Agent 约束** | 无 | Skills + 检查清单 + Review Agent 行为审查 |
| **产物验证** | 无自动验证 | 结构化验证 + 模板强制 |
| **交叉引用** | 靠人维护 | Compound 阶段自动检查 |
| **ADR 工具** | 无 | `/adr` 命令 + 自动 task.yaml 更新 |

## 10. 未解决问题

1. **Skills 自动加载机制**：Skill 放在 `.opencode/skills/` 目录是否自动生效？还是需要在 opencode.json 中注册？待实测验证。
2. **`/adr` 命令的 task 识别**：多个 in_progress task 时，取哪个？建议取最近启动的。
3. **Checkpoint 回溯的技术实现**：底层 Agent 不支持 checkpoint 机制时，降级为"手动跳转"模式。
4. **Review Agent 互审的可靠性**：Review Agent 本身也是 Agent，谁来审 Review Agent？目前没有好的解决方案，暂时接受"有胜于无"。
5. **Gate 量化 vs Human 覆盖**：Human 说"我就是要全量审查"时，量化结果是否应该被覆盖？建议允许 Human 覆盖。
