# Self-Workflow 需求规格说明

> 版本：v0.1（草案）
> 状态：草稿
> 最后更新：2026-06-06

---

## 1. 产品定位

### 1.1 核心价值主张

Self-Workflow 让软件工程从一次性开发活动演进为**可持续、自学习、可治理**的智能系统。

传统 AI coding 模式：每次任务是独立的、无状态的，项目知识随会话结束而流失。

Self-Workflow 模式：每次任务都有上下文、有记录、有沉淀，项目经验不断累积复用。

### 1.2 目标用户

| 用户角色 | 核心诉求 | 使用场景 |
|---------|---------|---------|
| 独立开发者 | 保持个人项目的一致性和质量 | 日常开发、快速原型 |
| 小团队 | 建立团队共识的开发规范和工作流 | 协作开发、代码审查 |
| 技术负责人 | 确保项目可治理、知识可传承 | 项目交接、新成员 onboarding |

### 1.3 设计原则

- **渐进式**：不要求一次性定义所有规范，随项目演进逐步补充
- **非侵入**：不改变现有开发工具链，作为辅助层运行
- **价值导向**：每项能力都必须解决真实痛点，不堆砌功能

## 2. 设计哲学

> 参考 Superpowers、Grill-Me、Trellis、Compound Engineering 四个框架的设计理念构建。

### 2.1 核心原则

| 原则 | 来源 | 含义 |
|------|------|------|
| **信任执行，验证结果** | Superpowers / CE | Agent 在执行阶段有完全的自主权——技能定义流程，Agent 决定如何执行。但执行结果必须通过 Phase Gate 验证。信任的是 Agent 的能力，验证的是产出的质量 |
| **强制约束，不协商** | Superpowers | 当某个 Skill 适用于当前任务时，Agent **必须**使用，没有选择。这不是建议，是规定 |
| **每个阶段设对抗性审查门** | Grill-Me / Trellis | 阶段之间设 **Phase Gate**，输出由独立角色或程序化工具审查通过后，才能进入下一阶段 |
| **程序化验证优先** | Trellis (Ralph Loop) | 能用 lint/typecheck/test 验证的，不依赖 AI 自我判断。程序化验证通过才算通过 |
| **证据驱动** | Superpowers / CE | "应该可以"不算完成。测试通过、lint 通过、产物存在——可验证的输出才算完成 |
| **可复利：每次工作让后续更简单** | Compound Engineering | 每次工作流的最后一步是沉淀经验，让下一次同类任务更快、更少踩坑 |

### 2.2 三层约束模型

Self-Workflow 对 Agent 的约束分三个层次，按执行时序排列：

```
Layer 1: 自主遵循 (Trust) —— 执行阶段持续生效
   └─ Agent 在执行过程中读取 Spec、Skill，自觉遵守
   └─ 适用于 SHOULD / MAY 级别规范
   └─ 不阻断，仅引导
   └─ 时序：Phase 执行中

Layer 2: 程序化门 (Programmatic Gate) —— Phase Gate 处先执行
   └─ lint / typecheck / test 等确定性工具，自动运行
   └─ 适用于 MUST 级别的可自动化规则
   └─ 不通过则阻断，Ralph Loop 重试（最多 N 次）
   └─ 时序：Phase Gate 第一步

Layer 3: 对抗性审查 (Adversarial Gate) —— 程序化门通过后执行
   └─ 由独立 Review Agent（预定义的 Agent 定义，由适配器生成）执行
   └─ Grill-Me 风格：逐个决策树分支审查设计/实现正确性
   └─ 适用于设计合理性、架构一致性等无法自动化判断的维度
   └─ 时序：Phase Gate 第二步（程序化门通过后）
```

**执行顺序**：Layer 1（持续）→ Layer 2（快，自动化）→ Layer 3（慢，深度审查）

**为什么 Layer 2 在 Layer 3 之前**：程序化验证成本低、速度快，能在几秒内拦截明显问题（lint 错误、类型不匹配），避免浪费对抗性审查的资源。只有通过程序化门的输出才值得深度审查。

### 2.3 工作流作为技能，而非引擎

Self-Workflow **不包含独立的工作流运行时引擎**。工作流通过以下机制分布式运作：

- **Skill**：定义"当前阶段要做什么"——一组结构化的操作指令
- **Command**：用户触发工作流的入口（`/feat`、`/debug` 等）
- **Hook**：自动触发阶段切换、上下文注入、经验提取
- **Review Agent**：每个 Phase Gate 处执行对抗性审查的独立子 Agent。该 Agent 由适配器在安装时预生成定义，不同于执行工作流的 Agent——它没有代码变更权限，只有读取 Artifact、分析 Spec、输出审查报告的能力
- **Programmatic Gate**：在 Phase Gate 处自动运行的验证脚本，由 Hook 触发，执行 lint/typecheck/test 等确定性命令
- **主 Agent**：执行工作流各阶段的 Agent（即用户对话的 Agent），拥有代码变更权限。执行阶段完成后，触发 Phase Gate，主 Agent 等待审查结果

> **重要**：Review Agent 与主 Agent 是**不同的角色**，职责分离：
> - 主 Agent：信任其执行能力，允许变更加密
> - Review Agent：不信任其判断被自身利益污染，无变更权限，仅审查
> 
> 两者由适配器在安装时分别生成独立的 Agent 定义。如果目标平台不支持子 Agent 机制，则降级为同一 Agent 内的"角色切换"模式。工作流定义（YAML）是静态的流程描述，执行靠主 Agent + Review Agent + Hook 共同完成。

### 2.4 Phase Gate 模型

每个工作流由若干阶段组成，阶段之间设 **Phase Gate（阶段门）**。Gate 内按顺序执行三层检查：

```
[阶段 N 执行] → [Phase Gate] → [阶段 N+1 执行]
                      │
              ┌───────┴───────┐
              │   Gate 流程    │
              │               │
              │  Step 1:      │
              │  程序化验证    │ ← lint / typecheck / test
              │  (自动执行)    │   失败 → Ralph Loop 重试
              │               │
              │  通过 ↓       │
              │               │
              │  Step 2:      │
              │  对抗性审查    │ ← Review Agent 审查
              │  (独立 Agent)  │   输出审查报告
              │               │
              │  通过 ↓       │
              │               │
              │  Step 3:      │
              │  人工确认      │ ← 可选，按工作流配置
              │  (用户审批)    │
              └───────┬───────┘
                      │
                全部通过 → 进入下一阶段
```

Phase Gate 的执行顺序（与三层约束模型一致）：
1. **程序化验证**（必须，第一步）：lint / typecheck / test 自动执行。不通过 → Ralph Loop 重试，不消耗对抗性审查资源
2. **对抗性审查**（必须，第二步）：程序化门通过后，Review Agent 审查输出质量。不通过 → 返回阶段 N 修复，重新触发 Phase Gate（从 Step 1 开始）
3. **人工确认**（可选，第三步）：按工作流配置，关键节点需要用户审批。超时自动进入下一阶段（可配置默认行为）

## 3. 核心概念

> 定义文档中使用的关键术语，确保理解一致。

| 术语 | 定义 |
|------|------|
| **工作流（Workflow）** | 针对特定开发任务类型定义的结构化执行流程，由若干阶段和 Phase Gate 组成 |
| **Artifact** | 工作流执行过程中产生的结构化记录，包括设计文档、审查报告、验证报告等 |
| **规范（Spec）** | 项目层面的约束定义，含 MUST/SHOULD/MAY 三级，通过程序化门或对抗性审查执行 |
| **经验（Experience）** | 从历史会话中提取的可复用知识，包括解决方案、模式、注意事项 |
| **Session** | 一次工作流从触发到首次通过所有 Phase Gate 的完整执行过程。一次 Session 可能包含多次迭代：Phase Gate 不通过 → 返回修复 → 重新审查 |

## 4. 整体架构

### 4.1 组件划分

> Self-Workflow 本身不是独立运行的 CLI 工具，而是安装到项目中的**工具链包**。它通过为目标 Agent（如 OpenCode）生成配置来运作——将工作流、规范、经验映射为 Agent 可理解的 agent、skill、command、hook。

```
Self-Workflow/
├── workflows/             # 工作流定义（YAML 结构化描述）
│   ├── feat.yaml          # 特性开发工作流
│   ├── debug.yaml         # 问题修复工作流
│   ├── doc.yaml           # 文档工作流
│   └── review.yaml        # 评审工作流
│
├── specs/                 # 规范模板
│   ├── coding-style.yaml
│   ├── architecture.yaml
│   └── ...
│
├── templates/             # 产物模板（ADR、总结、报告等）
│
├── adapters/              # 适配层 - 将工作流/规范/模板编译为目标工具配置
│   ├── opencode/          # OpenCode 适配器
│   │   ├── agents/        # → 生成 agent 定义
│   │   ├── skills/        # → 生成 skill 定义
│   │   ├── commands/      # → 生成 slash command
│   │   └── hooks/         # → 生成生命周期 hook
│   └── future/            # 未来适配其他工具
│
└── user-workspace/        # 用户项目目录（安装时生成）
    ├── .self-workflow/
    │   ├── artifacts/     # 产物归档（工作流执行时写入）
    │   ├── specs/         # 项目规范定义（用户可自定义）
    │   └── experiences/   # 经验沉淀（工作流完成时写入）
    └── opencode.jsonc     # 适配器生成的 OpenCode 配置
```

### 4.2 安装结构

安装后，用户项目目录下产生：

```
<project>/
├── .self-workflow/
│   ├── artifacts/
│   ├── specs/
│   │   ├── coding-style.yaml
│   │   └── architecture.yaml
│   └── experiences/
│
├── .opencode/
│   ├── agents/              # Self-Workflow agent 定义
│   ├── commands/            # Self-Workflow slash command 定义
│   ├── hooks/               # Self-Workflow lifecycle hook 定义
│   └── skills/              # Self-Workflow skill 定义
│
└── opencode.jsonc           # 主配置（引用上述 agent/command/hook/skill）
```

### 4.3 数据流

```
用户在 OpenCode 中输入自然语言
    │
    ▼
┌── OpenCode ─────────────────────────────────────────────────────┐
│                                                                  │
│  Hook（自动触发）          Command（用户显式触发）                  │
│  ├─ on:session-start      ├─ /feat                               │
│  ├─ on:workflow-start     ├─ /debug                              │
│  ├─ on:phase-complete     ├─ /doc                                │
│  └─ on:workflow-complete  └─ /review                             │
│                                                                  │
│  Agent 执行流程：                                                 │
│                                                                  │
│  ┌─ Phase N ─────────────────────────────────────────────┐       │
│  │  1. 加载 Skill → 获取本阶段工作流定义                   │       │
│  │  2. 加载相关 Spec + 经验                                │       │
│  │  3. 执行阶段任务                                        │       │
│  │  4. 写入 Artifact                                       │       │
│  └────────────────────────────────────────────────────────┘       │
│          │                                                       │
│          ▼                                                       │
│  ┌─ Phase Gate ─────────────────────────────────────────┐       │
│  │  1. 程序化验证 (lint/typecheck/test) ← Ralph Loop     │       │
│  │  2. 对抗性审查 (Review Agent / Grill 流程)            │       │
│  │  3. 人工确认 (可选, 按工作流配置)                     │       │
│  │  通过 → 进入 Phase N+1                               │       │
│  │  不通过 → 返回 Phase N 修复                          │       │
│  └────────────────────────────────────────────────────────┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
.self-workflow/     ← Agent / Review Agent 读写
 ├─ artifacts/      ← 工作流各阶段产物 + 审查报告
 ├─ specs/          ← 规范被加载 (Agent) + 被审查 (Review Agent)
 └─ experiences/    ← 经验被提取和检索 (仅已验证经验进入自动推荐)
```

## 5. 使用方式

### 5.1 安装

获取安装包后，通过一条命令安装到目标项目目录：

```bash
self-workflow init [--target <project-dir>]
```

安装过程：
1. 在项目目录下创建 `.self-workflow/` 元数据目录
2. 适配器根据项目类型和目标 Agent（当前为 OpenCode），生成配套的工具链配置：
   - agent 定义 → `.opencode/agents/`
   - skill 定义 → `.opencode/skills/`
   - command 定义 → `.opencode/commands/`
   - hook 定义 → `.opencode/hooks/`
   - 主配置引用 → `opencode.jsonc`
3. 初始化空的规范目录和经验知识库
4. 输出安装报告

### 5.2 初始化配置

安装完成后，用户可选择：
- 创建/修改项目规范（`.self-workflow/specs/`）
- 导入历史经验（如有）
- 在 `opencode.jsonc` 中启用/禁用特定工作流

### 5.3 日常使用

安装之后，用户**不再直接操作 Self-Workflow**。所有交互通过 OpenCode 进行。

#### 通过 Command 显式触发工作流

用户输入 OpenCode 的 slash command 来启动工作流：

```
/feat 实现用户登录模块
/debug 登录接口返回 500 错误，错误信息：...
/doc 补充 API 文档
/review 审查 auth 模块的变更
```

Command 被触发后，对应的 Agent 自动加载工作流定义、项目 Spec、相关经验，开始执行。

#### Phase Gate 自动执行

每个阶段完成后，Phase Gate 自动触发，无需用户介入：

| Gate 类型 | 执行者 | 行为 |
|-----------|--------|------|
| 程序化验证 | Hook 触发自动脚本 | 运行 lint / typecheck / test，失败则阻止推进（Ralph Loop 重试机制） |
| 对抗性审查 | Review Agent（独立子 Agent） | 读取本阶段输出 Artifact，按 Grill 风格逐个决策审查，输出审查报告 |
| 人工确认 | 用户 | 可选配置。关键阶段（如设计评审）需要用户显式确认 |

**对抗性审查的工作方式**（Grill-Me 风格）：

Review Agent 不负责"确认"或"批准"，而是**挑战假设**：
- 逐决策树分支提问，一次一个问题
- 每个问题附带 Review Agent 自己的推荐答案
- 能通过代码库探索回答的问题，不打扰用户
- 直到所有分支解决，或明确标记未解决的问题

#### 通过 Hook 自动触发

部分环节通过 lifecycle hook 自动运行：

| Hook | 触发时机 | 行为 |
|------|---------|------|
| `on:session-start` | 每次新会话启动 | 加载 Spec + 检索相关经验到上下文 |
| `on:workflow-start` | 工作流开始 | 创建 Artifact 目录、记录触发条件 |
| `on:phase-complete` | 阶段完成 | 触发 Phase Gate（验证 + 审查 + 日志） |
| `on:workflow-complete` | 工作流完成 | 触发 Compound 流程：提取经验 → 归档产物 → 标记完成 |
| `on:tool-call` | Agent 调用工具 | 记录关键工具调用到 Artifact（可选） |

#### 通过 Agent 自主学习

Agent 在 Skill 中加载工作流阶段定义，在执行过程中自主：
1. 读取 `.self-workflow/specs/` 获取规范约束
2. 读取 `.self-workflow/experiences/` 检索相关经验（仅已验证经验）
3. 在 `.self-workflow/artifacts/` 中写入产物

#### Compound：完成时自动沉淀

工作流完成时，Compound 流程自动执行（`on:workflow-complete`）：
1. 从当前工作流的 Artifact 中提取可复用的解决方案、模式、注意事项
2. 生成经验草稿，标记为 `draft` 等级
3. 存入 `.self-workflow/experiences/`
4. （可选）通知用户有新的经验待审查

#### 查询历史

用户直接在 OpenCode 中询问即可：

```
"上次用户登录模块的开发记录在哪里？"
"我们项目对 API 设计有什么规范？"
"之前遇到过类似的数据库问题吗？"
```

Agent 读取 `.self-workflow/` 中的内容来回答。

## 6. 核心特性详述

### 6.1 可追溯（Traceability）

#### 目标
每次开发任务的设计、决策、结果都能够被持久化，形成完整的项目决策链路。

#### 运行机制

**记录内容（按阶段）：**

| 阶段 | 记录内容 | 格式 |
|------|---------|------|
| 启动 | 任务描述、触发条件、关联 Issue | YAML front matter |
| 分析 | 需求理解、约束识别、设计选项 | Markdown + 决策矩阵 |
| 设计 | 架构决策（ADR）、接口设计、数据模型 | ADR 模板 |
| 实施 | 变更文件清单、关键实现决策 | 内联注释 + 变更日志 |
| 验证 | 测试策略、验证结果、问题记录 | 结构化报告 |
| 完成 | 总结、遗留问题、经验提取 | 总结模板 |

**存储结构：**

```
.self-workflow/artifacts/
├── feat-user-login-20260606/
│   ├── workflow.yaml         # 工作流实例元数据
│   ├── 01-analysis.md        # 分析阶段产物
│   ├── 02-design.md          # 设计阶段产物
│   ├── 03-implementation.md  # 实施阶段记录
│   ├── 04-verification.md    # 验证阶段报告
│   └── 05-summary.md         # 总结与经验提取
└── debug-auth-timeout-20260605/
    └── ...
```

**查询能力：**
- 按时间范围检索工作流执行记录
- 按工作流类型过滤（feat/debug/doc/review）
- 按关键词搜索决策内容
- 查看指定文件的变更历史及其关联的工作流

#### 验收标准

- [ ] 每个工作流执行完成后，产物目录下生成完整的记录文件
- [ ] 记录文件中包含各阶段的决策和理由
- [ ] 支持通过 OpenCode 以自然语言查询历史记录（Agent 读取 .self-workflow/artifacts/）
- [ ] 关联代码变更（commit）与对应的工作流记录

---

### 6.2 可约束（Constraint）

#### 目标
支持渐进式披露项目的 Spec 规范，引导 Agent 遵守项目约定，保证输出的一致性。

#### 运行机制

**三层约束模型：**

Self-Workflow 不依赖 Agent 自我监督。约束通过三个独立层次叠加实现：

```
Layer 1: 自主遵循（Trust）
   执行者: Agent 自身
   适用范围: SHOULD / MAY 级别规范
   机制: Agent 在执行阶段读取 Spec 并自觉遵守
   不通过后果: 无阻断，但会在审查报告中记录

Layer 2: 程序化门（Programmatic Gate）
   执行者: Hook 触发的验证脚本
   适用范围: MUST 级别中可自动化的规则（lint、类型检查、测试）
   机制: 每个 Phase Gate 处自动运行验证脚本
   不通过后果: 阻断推进，Ralph Loop 重试（最多 N 次）
   关键技术: eslint, tsc, pytest, 自定义 lint 规则

Layer 3: 对抗性审查（Adversarial Gate）
   执行者: Review Agent（独立子 Agent）
   适用范围: 设计正确性、架构一致性、Spec 合规性（无法自动化的维度）
   机制: Phase Gate 处由 Review Agent 执行 Grill 式审查
   不通过后果: 输出审查报告，标记未通过项，返回修复
```

**Spec 定义方式：**

Spec 以结构化文件定义，放置在 `.self-workflow/specs/` 目录下：

```
.self-workflow/specs/
├── coding-style.yaml      # 编码规范（命名、格式、注释）
├── architecture.yaml      # 架构规则（分层、依赖方向、模块边界）
├── tech-stack.yaml        # 技术栈约束（语言、框架、库版本）
├── workflow-protocol.yaml # 工作流协议（产物模板、质量标准）
└── workflow-specific/     # 按工作流类型细分的规范
    ├── feat.yaml
    ├── debug.yaml
    ├── doc.yaml
    └── review.yaml
```

**渐进式披露机制：**

| 原则 | 说明 |
|------|------|
| 按需加载 | 仅加载与当前工作流类型和阶段相关的规范，而非全部 |
| 分层披露 | Phase 1 披露编码风格 → Phase 2 披露架构规则 → Phase 3 披露测试规范 |
| 优先级排序 | 按规范的重要程度排序，避免信息过载 |

**约束级别：**

| 级别 | 含义 | 执行层 | 违反后果 |
|------|------|--------|---------|
| `MUST` | 强制遵守 | Layer 2 + Layer 3 | Phase Gate 阻断，审查报告记录，修复后重新审查 |
| `SHOULD` | 建议遵守 | Layer 1 | Phase Gate 产生警告，不阻断 |
| `MAY` | 可选参考 | Layer 1 | 仅作为参考信息提供，不纳入审查 |

**关于 MUST 级别的重要说明：**

MUST 级别的规范不是靠 Agent "自觉遵守"，而是通过 Phase Gate 发现并阻断：
- 可自动化的 MUST（如命名规范、禁止模式）→ 在程序化门中被 lint 规则捕获
- 不可自动化的 MUST（如"异步操作必须使用 try-catch"）→ 在对抗性审查中被 Review Agent 发现
- **Agent 不必在执行时自我检查**，但必须接受 Phase Gate 的审查结果

**Spec 中的 `enforcement` 字段：**

```yaml
rules:
  - id: naming-001
    description: 函数命名使用 camelCase
    level: MUST
    scope: typescript
    enforcement: lint       # → Layer 2: 程序化门
  - id: error-handling-001
    description: 异步操作必须使用 try-catch
    level: MUST
    scope: async-function
    enforcement: review     # → Layer 3: 对抗性审查
```

**对抗性审查的 Spec 审查流程（Grill-Me 风格）：**

Review Agent 在每个 Phase Gate 处执行：
1. 读取本阶段产出的 Artifact
2. 加载对应 Spec（仅与本阶段相关的规则）
3. 逐条检查 Spec 合规性
4. 对每个违规点：
   - 定位违规位置
   - 引用违规的 Spec 规则 ID
   - 给出修复建议
   - 标记严重级别（critical / warning / info）
5. 输出审查报告到 Artifact 目录
6. 如果有 critical 级别问题 → Phase Gate 不通过

**Spec 生命周期：**
- **创建**：初始化时生成默认模板，或用户手动创建
- **修订**：项目演进中通过 review 工作流更新规范，或从审查报告中提炼新规则
- **版本化**：Spec 随项目代码一起版本控制，随 PR 评审更新
- **废弃**：明确标记废弃的规范条目，保留历史记录

#### 验收标准

- [ ] Spec 文件采用结构化格式（YAML），支持 MUST/SHOULD/MAY 级别
- [ ] 工作流启动时仅加载相关 Spec 和当前阶段相关的部分
- [ ] MUST 级别规范通过 Phase Gate（程序化门 + 对抗性审查）执行，而非 Agent 自我监督
- [ ] 程序化门在 Phase Gate 处自动运行 lint/typecheck/test
- [ ] 对抗性审查由独立 Review Agent 执行，输出结构化审查报告
- [ ] 新项目初始化时生成默认规范模板
- [ ] Spec 随项目代码纳入版本控制

### 6.3 可复利（Compounding）

#### 目标
将会话中有价值的经验进行沉淀，为后续开发提供参考和指导，使项目知识持续累积——**每次工作流让下一次同类任务更简单**。

#### 运行机制

**Compound 流程（工作流完成时自动执行）：**

工作流完成后，`on:workflow-complete` Hook 触发 Compound 流程：

```
工作流完成
    │
    ▼
┌─ Compound ─────────────────────────────────────────┐
│  1. 从本工作流 Artifact 提取可复用的知识             │
│     - 解决方案 / 架构决策                            │
│     - 踩坑记录 / 注意事项                            │
│     - 审查报告中的重复模式                           │
│                                                     │
│  2. 生成经验草稿，标记为 draft                       │
│                                                     │
│  3. 存入 .self-workflow/experiences/                 │
│     - 自动关联工作流 ID、标签、适用工作流类型          │
│                                                     │
│  4. 可选：更新 CLAUDE.md / AGENTS.md                │
│     (高频重复经验提升为全局上下文)                    │
└─────────────────────────────────────────────────────┘
```

**经验质量分级与晋升机制：**

经验不自污染——未验证的经验**不进入自动推荐通道**：

```
                    draft
                      │  工作流完成时自动提取
                      │  Review Agent 审核
                      ▼
                  verified ⭐
                      │  Review 工作流确认
                      │  用户手动晋升
                      ▼
               promoted ⭐⭐
                      │  高频复用确认
                      │  提升为全局上下文
                      ▼
              elevated (写入 CLAUDE.md)
```

| 等级 | 含义 | 来源 | 自动检索? |
|------|------|------|-----------|
| `draft`（草稿） | 自动提取，尚未审查 | Compound 流程 | ❌ 不检索 |
| `verified` ⭐ | 经过 Review Agent 或用户审查确认 | Review 工作流 / 用户确认 | ✅ 检索 |
| `promoted` ⭐⭐ | 高频复用，价值经验证 | 多次被成功检索引用 | ✅ 优先推荐 |
| `elevated`（提升） | 提升为全局持久上下文 | 手动写入 `CLAUDE.md` | ✅ 每次会话加载 |

**经验捕获时机：**

| 捕获时机 | 来源 | 自动/手动 | 初始等级 |
|---------|------|-----------|---------|
| 工作流完成时 | 从总结中自动提取 | 自动 | draft |
| 用户主动沉淀 | 用户标记值得记录的知识 | 手动 | verified |
| Review 过程中 | 从审查报告中提炼 | 自动 | draft |
| 审查经验时 | Review Agent 审核 draft | 自动 | → verified |

**经验格式：**

```yaml
# .self-workflow/experiences/auth-pattern.yaml
title: 用户认证模块实现方案
type: solution-pattern
workflow: feat-user-login-20260606
tags: [auth, jwt, security]
applicable-to: [feat, review]
quality: verified    # draft → verified → promoted → elevated
reviewed-by: review-agent-20260607
date: 2026-06-06
content: |
  ## 问题
  需要在现有系统中实现用户认证

  ## 方案
  采用 JWT + HttpOnly Cookie 方案...

  ## 关键决策
  - 选择 refresh token 旋转策略，有效期 7 天
  - Token 存储在 HttpOnly Cookie 而非 localStorage，防止 XSS

  ## 注意事项
  - 登出时需要同时清除 access token 和 refresh token
  - Token 刷新接口需要做速率限制

  ## 关联文件
  - src/auth/jwt.ts
  - src/middleware/auth.ts
```

**检索与复用：**

| 方式 | 触发时机 | 检索范围 | 行为 |
|------|---------|----------|------|
| 自动关联 | 工作流启动时 | 仅 `verified` 及以上 | 根据任务描述和标签匹配经验 |
| 主动查询 | 用户自然语言询问 | 全部（含 `draft`） | "之前遇到过类似问题吗？" Agent 读取全部经验 |
| 上下文推荐 | 执行过程中 | 仅 `verified` 及以上 | 基于当前阶段和产物推荐相关经验 |

#### 对抗性审查在经验质量中的应用

经验晋升需要经过对抗性审查：
- `draft → verified`：由 Review Agent 按 Grill 风格审查。Review Agent 不是"确认"经验是否正确，而是挑战经验的假设、完整性和可复用性
- `verified → promoted`：取决于实际复用效果——如果某条经验被反复检索且未被后续审查纠正，可自动或手动晋升
- 用户可随时手动降级、修改或删除任何经验

#### 验收标准

- [ ] 工作流完成时自动触发 Compound 流程，生成经验草稿（draft）
- [ ] draft 等级经验不进入自动检索通道
- [ ] 支持 Review Agent 将 draft 审查晋升为 verified
- [ ] 支持用户手动标记、沉淀、修改、删除经验
- [ ] 工作流启动时自动检索 verified 及以上等级的相关经验
- [ ] 支持通过 OpenCode 以自然语言搜索经验库
- [ ] 经验遵循结构化格式，包含问题、方案、决策、注意事项

## 7. 工作流详细定义

### 7.1 通用结构

每个工作流由若干**阶段（Phase）** 和阶段之间的 **Phase Gate** 组成：

```
┌─────────────────────────────────────────────────────────┐
│  Phase N                                                 │
│  ├── 执行内容                                          │
│  ├── 输入：上一阶段产物 + 相关 Spec + 相关经验            │
│  ├── 执行：Agent 自主执行                                │
│  └── 输出：本阶段产物（写入 Artifact）                    │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│  Phase Gate                                              │
│  ├── 程序化验证：lint / typecheck / test（自动）          │
│  ├── 对抗性审查：Review Agent 按 Grill 风格审查           │
│  ├── 人工确认：（可选）                                   │
│  ├── 通过 → 进入 Phase N+1                              │
│  ├── 不通过 → 返回 Phase N 修复                          │
│  └── 产物：审查报告（写入 Artifact）                      │
└─────────────────────────────────────────────────────────┘
```

每个阶段定义：

```
[阶段名]
├── 执行内容：该阶段的核心活动
├── 输入：阶段开始前需要准备的材料
├── 产物：阶段产生的输出
└── 超时/异常处理：阶段执行失败时的降级策略
```

每个 Phase Gate 定义：

```
[Phase Gate]
├── 程序化验证：自动运行的验证命令
├── 对抗性审查：Review Agent 的审查维度
├── 人工确认：是否需要用户显式确认
└── 不通过处理：修复策略
```

### 7.2 feat：需求/特性开发

**触发条件：**
- 用户显式发起特性开发请求
- 从 Issue/需求文档中提取的特性任务

**流程概览：**

```
需求分析 → [Gate] → 方案设计 → [Gate] → 代码实现 → [Gate] → 功能验证 → [Gate] → 总结沉淀 → Compound
```

**阶段与 Phase Gate 定义：**

| 环节 | 类型 | 内容 |
|------|------|------|
| **需求分析** | Phase | 理解需求、识别约束、明确验收标准。输出：需求分析文档 |
| **[Gate] 分析审查** | Phase Gate | 程序化验证：无（设计阶段尚无代码）。对抗性审查：Review Agent 检查需求完整性、约束覆盖、与现有架构的兼容性。人工确认：✅ 需要用户确认需求理解正确。不通过 → 返回分析阶段修正 |
| **方案设计** | Phase | 架构决策、接口设计、数据模型设计。输出：设计文档 / ADR |
| **[Gate] 设计审查** | Phase Gate | 程序化验证：无（设计阶段尚无代码）。对抗性审查：Review Agent 按 Grill 风格审查——逐决策分支挑战设计选择、检查备选方案、评估 trade-off（参见下方"对抗性设计评审"）。人工确认：✅ 可选（复杂设计需要）。不通过 → 返回设计阶段修正 |
| **代码实现** | Phase | 按设计进行编码、单元测试。输出：代码变更 |
| **[Gate] 实现审查** | Phase Gate | 程序化验证：lint / typecheck / 单元测试 → Ralph Loop 重试机制。对抗性审查：Review Agent 检查 Spec 合规性、与设计的一致性。人工确认：❌。不通过 → 返回实现阶段修复 |
| **功能验证** | Phase | 运行测试、边界检查。输出：验证报告 |
| **[Gate] 验证审查** | Phase Gate | 程序化验证：完整测试套件。对抗性审查：Review Agent 检查测试覆盖、边界条件处理。人工确认：❌。不通过 → 返回实现阶段修复 |
| **总结沉淀** | Phase | 提取经验、补充文档。输出：总结报告、经验条目（draft） |
| **Compound** | 自动触发 | `on:workflow-complete` Hook 执行 Compound 流程 → 提取经验 → 归档产物 |

**对抗性设计评审（Grill 风格）：**

方案设计阶段的 Phase Gate 中，Review Agent 执行以下审查流程：

1. 读取设计文档中的每个架构决策
2. 对每个决策：
   - 分析备选方案是否被充分评估
   - 检查选择的理由是否成立
   - 挑战隐含假设：这个方案在 X 场景下是否仍然成立？
   - 确认 trade-off 被明确记录
3. 按决策树分支逐一提问，一次一个问题
4. 每个问题附带 Review Agent 自己的推荐答案
5. 产生设计审查报告

**产物清单：**
- 需求分析文档
- 设计文档 / 架构决策记录（ADR）
- 设计审查报告
- 代码变更（diff）
- 测试结果报告
- 实现审查报告
- 经验提取条目（draft）

### 7.3 debug：问题定位/修复

**触发条件：**
- 用户报告 Bug，提供复现步骤或错误信息
- 从 Issue 跟踪系统同步的问题项

**流程概览：**

```
问题复现 → [Gate] → 根因分析 → [Gate] → 方案制定 → [Gate] → 代码修复 → [Gate] → 回归验证 → [Gate] → 总结沉淀 → Compound
```

**阶段与 Phase Gate 定义：**

| 环节 | 类型 | 内容 |
|------|------|------|
| **问题复现** | Phase | 确认 Bug 可复现、明确影响范围。输出：复现步骤文档 |
| **[Gate] 复现确认** | Phase Gate | 程序化验证：无。对抗性审查：Review Agent 验证复现步骤的完整性和确定性。人工确认：✅ 需要用户确认复现成功。不通过 → 返回复现阶段 |
| **根因分析** | Phase | 日志分析、代码追溯、数据流追踪。输出：根因分析报告 |
| **[Gate] 根因审查** | Phase Gate | 程序化验证：无。对抗性审查：Review Agent 检查根因推理链路——是否排除了其他可能性？证据是否充分？是否混淆了症状和原因？人工确认：可选。不通过 → 返回分析阶段 |
| **方案制定** | Phase | 评估修复方案、考虑副作用。输出：修复设计 |
| **[Gate] 方案审查** | Phase Gate | 程序化验证：无。对抗性审查：Review Agent 检查修复范围是否适当（最小修复原则）、是否存在副作用、是否缺少边界处理。人工确认：可选。不通过 → 返回方案阶段 |
| **代码修复** | Phase | 实施修复、补充针对性的测试。输出：代码变更 |
| **[Gate] 修复审查** | Phase Gate | 程序化验证：lint / typecheck / test。对抗性审查：Review Agent 检查修复正确性、测试覆盖。人工确认：❌。不通过 → 返回修复阶段 |
| **回归验证** | Phase | 验证 Bug 已修复、无回归问题。输出：验证报告 |
| **[Gate] 回归审查** | Phase Gate | 程序化验证：完整测试套件。对抗性审查：Review Agent 检查是否引入了新的问题。人工确认：❌。不通过 → 返回修复阶段 |
| **总结沉淀** | Phase | 提炼防同类问题的经验。输出：经验条目（draft） |
| **Compound** | 自动触发 | 提取经验、归档产物 |

**关键原则：**
- **最小修复**：只修复问题本身，不在此过程中重构——Phase Gate 会审查是否引入了不必要的变更
- **测试先行**：先编写复现测试用例，再实施修复
- **回归验证**：确保修复不引入新的问题

**产物清单：**
- Bug 复现步骤
- 根因分析报告
- 修复方案说明
- 修复审查报告
- 测试用例变更
- 验证报告
- 经验提取条目（draft）

### 7.4 doc：补充/完善文档

**触发条件：**
- 用户明确要求补充文档
- 代码审查发现文档缺失
- 工作流总结标记了文档缺口

**流程概览：**

```
缺口分析 → [Gate] → 内容撰写 → [Gate] → 格式审查 → [Gate] → 内容审核 → [Gate] → 发布归档 → Compound
```

**阶段与 Phase Gate 定义：**

| 环节 | 类型 | 内容 |
|------|------|------|
| **缺口分析** | Phase | 评估现有文档覆盖范围、识别缺失。输出：文档缺口分析报告 |
| **[Gate] 缺口审查** | Phase Gate | 对抗性审查：Review Agent 检查缺口清单的完整性，是否有遗漏的重要内容。人工确认：✅ 需要用户确认范围 |
| **内容撰写** | Phase | 按文档模板编写内容。输出：文档初稿 |
| **[Gate] 内容审查** | Phase Gate | 程序化验证：格式检查、链接检查。对抗性审查：Review Agent 检查技术准确性、表达清晰度、示例可用性 |
| **格式审查** | Phase | 格式一致性、术语统一。输出：格式化文档 |
| **[Gate] 格式门** | Phase Gate | 程序化验证：链接有效性、格式规范检查。对抗性审查：无。人工确认：❌ |
| **内容审核** | Phase | 技术准确性、表达清晰度。输出：终稿 |
| **[Gate] 终审** | Phase Gate | 人工确认：✅ 需要用户审核通过 |
| **发布归档** | Phase | 纳入文档体系、更新索引。输出：文档归档记录 |
| **Compound** | 自动触发 | 归档产物 |

**产物清单：**
- 文档缺口分析报告
- 文档内容
- 审核意见记录

---

### 7.5 review：评审文档/代码/测试

> Review 工作流是所有其他工作流的 Phase Gate 中使用的对抗性审查机制的核心。它可以作为独立工作流运行，也可以嵌入到其他工作流的 Phase Gate 中。

**触发条件：**
- 其他工作流的 Phase Gate 自动触发 Review Agent
- 用户显式发起评审请求
- 针对指定范围（文件、模块、文档）发起评审

**评审维度：**

| 维度 | 检查内容 | 审查方式 | 执行时机 |
|------|---------|----------|---------|
| 规范遵循 | 是否符合项目 Spec | Review Agent 逐条核对 | 实现审查 Gate |
| 质量检查 | 代码质量、测试覆盖、边界处理 | Review Agent 分析 + 程序化 gate | 验证审查 Gate |
| 安全审查 | 常见安全漏洞 | Review Agent 安全检查清单 | 实现审查 Gate |
| 设计一致性 | 是否与现有架构一致 | Review Agent + 历史 ADR 比对 | 设计审查 Gate |
| 文档完整性 | 相关文档是否同步更新 | Review Agent 检查 + 程序化 gate | 总结阶段 |

**Grill 风格审查流程（对抗性审查的核心机制）：**

```
1. 审查准备
   ├── 读取被审查的 Artifact（设计文档 / 代码 / 文档）
   ├── 加载对应的 Spec 规则
   ├── 加载相关历史 ADR 和经验
   └── 识别关键决策点

2. 逐项审查（Grill 模式）
   ├── 每次聚焦一个决策/一个问题
   ├── 分析隐含假设
   ├── 检查备选方案是否被考虑
   ├── 验证与现有架构/规范的一致性
   └── 给出推荐意见

3. 输出审查报告
```

**审查报告格式：**

```yaml
review-report:
  scope: "src/auth/*"
  workflow: "feat-user-login-20260606"
  gate: "implementation-review"
  summary: "发现 2 个 critical 问题，需要修复"
  findings:
    - severity: critical
      location: "src/auth/jwt.ts:42"
      description: "Token 密钥硬编码"
      rule: "secrets-management-001"
      evidence: "字符串字面量 'my-secret-key' 出现在源码中"
      suggestion: "使用环境变量注入，参考 src/config/env.ts 的现有模式"
      status: blocking    # 阻断 Phase Gate
    - severity: warning
      location: "src/auth/middleware.ts:15"
      description: "缺少速率限制"
      rule: "security-rate-limit-001"
      evidence: "/login 接口没有速率限制中间件"
      suggestion: "添加 express-rate-limit 中间件"
      status: non-blocking
    - severity: info
      location: "src/auth/README.md"
      description: "缺少环境变量说明"
      rule: "doc-completeness-001"
      evidence: "README 中没有 TOKEN_SECRET 说明"
      suggestion: "补充环境变量说明"
      status: non-blocking
  pass: false   # critical 存在，不通过
  recommended-action: "修复 critical 问题后重新提交审查"
```

**审查发现的阻断规则：**

| 严重级别 | 含义 | Phase Gate 行为 |
|---------|------|----------------|
| `critical` | 阻断性问题（安全漏洞、功能错误、Spec 违规） | Phase Gate 不通过，必须修复 |
| `warning` | 非阻断性问题（质量、风格、覆盖不足） | Phase Gate 可通过，但记录到审查报告 |
| `info` | 建议项 | 仅记录，不影响 Phase Gate |

**产物清单：**
- 评审报告
- 问题清单（按严重级别分类）
- 改进建议

## 8. 非功能性需求

### 8.1 兼容性

| 维度 | 要求 |
|------|------|
| 目标工具 | OpenCode（v1.x），后续适配其他 Agent |
| 项目类型 | 不限语言/框架，通用设计 |
| 版本控制 | Git（当前），其他 VCS 后续支持 |
| 操作系统 | macOS / Linux / Windows |

### 8.2 性能

- Spec 加载和匹配时间 < 500ms
- 经验知识库检索（verified 等级）时间 < 1s
- Phase Gate 程序化验证执行时间 < 5s（lint/typecheck/test）
- 产物存储空间：单次工作流 < 1MB（纯文本）

### 8.3 安全

- 产物存储不记录敏感信息（密钥、密码等），或提供脱敏机制
- Spec 中不包含凭据类配置
- 本地存储，不强制依赖外部服务

## 10. 术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| 工作流 | Workflow | 针对特定开发任务的结构化执行流程，由若干阶段和 Phase Gate 组成 |
| 产物 | Artifact | 工作流执行过程中产生的结构化记录，包括设计文档、审查报告、验证报告等 |
| 规范 | Spec | 项目约束定义，含 MUST/SHOULD/MAY 三级，通过程序化门或对抗性审查执行 |
| 经验 | Experience | 从历史会话中提取的可复用知识，分 draft/verified/promoted/elevated 四级 |
| 会话 | Session | 一次工作流从触发到完成的完整执行过程 |
| 适配器 | Adapter | 将工作流/Spec/模板编译为目标 Agent 工具配置的组件 |
| 渐进式披露 | Progressive Disclosure | 按工作流类型和当前阶段按需加载 Spec，避免信息过载 |
| 架构决策记录 | ADR | Architecture Decision Record，记录关键架构决策及其上下文 |
| **阶段门** | **Phase Gate** | 阶段之间的质量闸门，包含程序化验证 + 对抗性审查 + 可选人工确认 |
| **对抗性审查** | **Adversarial Review** | Review Agent 以 Grill 风格逐决策挑战的输出审查机制 |
| **程序化验证** | **Programmatic Gate** | 通过 lint/typecheck/test 等确定性工具自动执行的验证 |
| **Compound** | **Compound** | 工作流完成时的经验沉淀流程，自动提取可复用知识 |
| **Ralph Loop** | **Ralph Loop** | 程序化验证失败后的自动重试循环，达到最大次数后标记失败 |
| **Skill** | **Skill** | 定义工作流阶段执行方式的结构化指令集，Agent 必须遵循 |
