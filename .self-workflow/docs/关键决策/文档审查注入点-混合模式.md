# 文档审查注入点——混合模式

## 背景

将 `docs/`、`configs/`、`specs/` 的文档变更纳入 Gate 审查范围。当前 feat-workflow 的 4 个 Gate 仅审查阶段产物（`artifacts/*.md`），文档变更绕过审查体系。需要决策：文档审查在哪个环节执行。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A（Per-Gate） | 每个 Gate 入口检测本阶段文档变更，纳入该 Gate 对抗性审查 | 及早发现问题；与现有 Gate 体系融合 | 每个 Gate 增重；Phase 5 无 Gate 需特殊处理；Gate weight=skip 时文档审查也会被跳过 |
| B（Compound 集中） | 仅在 Compound 归档阶段增加文档审查步骤 | 简单，单一审查点；不干扰现有 Gate 流程 | 发现问题晚，修复成本高；从变更到审查间隔长 |
| C（混合） | Gate 做轻量检测+标记；Compound 做完整审查 | 平衡及时性和简洁性；Gate 不增重；文档审查不依赖 Gate weight | 两处修改，但改动量都很小 |

## 选择

方案 C（混合）

## 理由

1. **Gate 保持轻量**：Gate 仅增加一条检查清单项（"本阶段是否产生文档变更？"），检测到则标记到 task.yaml，不触发额外的对抗性审查流程。
2. **Compound 做实质审查**：Compound 已有交叉引用检查步骤，文档审查是交叉引用检查的自然扩展。在 Compound 做完整审查，不受 Gate weight 影响。
3. **Phase 5 自然覆盖**：Phase 5 无 Gate，其文档产出直接在 Compound 被审查。Gate 2-4 的文档变更也会被 Compound 覆盖（Compound 扫描所有阶段的变更）。
4. **与现有错误经验一致**：Compound 的交叉引用检查已有"确认产物完整性"的职责，文档审查是同一职责的扩展。

## 影响

- **正面**：Gate 流程不受影响，文档审查独立且强制；Compound 的一次性审查覆盖所有阶段的文档变更。
- **负面**：审查时机稍晚（Compound 在最后），但文档变更通常是补充性的，晚期审查风险可控。

## 分级处理

不同类型的文档变更采用不同审查策略：

| 文档类型 | 审查策略 | 审查环节 |
|---------|---------|---------|
| **spec 类**（`.self-workflow/specs/`） | 在 Gate 2 方向审查中快速校验——检查是否与已有 ADR/spec 矛盾 | Gate 2 |
| **configs 类**（`.self-workflow/configs/`，如 feat-workflow.md） | 已纳入 Gate 2/3 的设计/实现审查 | Gate 2/3 |
| **docs 类**（`.self-workflow/docs/`） | 格式/规范类问题为主，Compound 审查 | Compound |

## 关联

- 关联需求：F1a, F1b, F1c
- 关联约束：文档审查不受 Gate weight 影响
- 关联 spec：`feat-workflow.md` Compound 审查步骤

## 适用场景

- 新增文档审查流程时选择注入点
- Gate 与 Compound 职责划分
- 文档变更分级处理（spec/configs/docs 不同处理策略）
