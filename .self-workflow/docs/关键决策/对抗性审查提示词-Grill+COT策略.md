# ADR-002：对抗性审查提示词的改写策略

## 背景

V1.10 需求：将 Gate 1/3/4 的 Review Agent 提示词从确认式改为 Grill 对抗式。当前仅 Gate 2 使用 Grill 风格提示词，Gate 1/3/4 使用确认式提示词：

- Gate 1：`"检查：需求理解是否与用户输入一致？功能清单是否覆盖全部需求？验收标准是否可测试？..."`（确认式）
- Gate 3：`"审查代码实现。对照设计文档...检查：实现是否与设计一致？代码质量？..."`（确认式）
- Gate 4：`"审查验证结果文档...检查：测试覆盖是否充分？边界条件是否处理？..."`（确认式）

需要决策：采用统一模板还是按 Gate 差异化定制。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A（统一 Grill 模板） | 所有 Gate 使用相同的 Grill 风格提示词前缀 | 一致性好，维护简单；Agent 行为可预测 | 不同 Gate 审查对象不同（分析/代码/验证），通用模板精度不够 |
| B（按 Gate 差异化） | 每个 Gate 定制 Grill 提示词，匹配审查对象特点 | 精准匹配各 Gate 的审查需求；可针对性覆盖已知问题模式 | 维护 3 份提示词变体；需要理解各 Gate 审查重点 |
| C（最小改动） | 仅在现有提示词中增加 Grill 指令前缀，不改动原有检查项 | 改动最小，向后兼容 | Grill 效果可能不充分——原有检查项仍是确认式措辞 |
| D（方案 B + COT 推理链） | 在方案 B 基础上，要求 Review Agent 输出逐项推理步骤 | Grill 有效性有保障——COT 强制 Agent 不能跳跃到结论 | 提示词长度增加，Review Agent 输出可能变长 |

## 选择

方案 D（方案 B 按 Gate 差异化 + COT 推理链）

## 理由

1. **审查对象差异大**：Gate 1 审查需求分析（文字理解），Gate 3 审查代码实现（对照设计），Gate 4 审查验证结果（对照验收标准）。方案 B 的差异化提示词精准匹配这些差异。
2. **COT 保障审查有效性**：Review Agent 是内置 Agent，对 Grill 提示词的敏感度不可控。COT 要求"逐项输出推理链"强制 Agent 不能跳跃到结论，即便其系统提示词偏确认式，推理链也会暴露逻辑漏洞。
3. **已知问题模式可针对性解决**：
   - Gate 1 已知问题：验收标准不够可测试 → 提示词增加"每条验收标准是否可客观判断通过/失败？尝试构造一个让它失败的场景"
   - Gate 3 已知问题：实现与设计可能细节不一致 → 提示词增加"逐项对照设计文档的接口定义，找出任何偏差"
   - Gate 4 已知问题：形式化验证（静态文件检查而非运行时）→ 提示词增加"验证报告中的'通过'是否有实质证据？是否仅做了文件存在性检查？"

## 影响

- **正面**：各 Gate 审查精度提升；已知问题模式被显式覆盖。
- **负面**：需要维护 3 份不同的 Grill 提示词前缀。如果未来 Gate 数量增加，维护成本线性增长。

## 反对意见

- 方案 A（统一模板）的简洁性值得考虑，但 Grill 风格的核心是"针对特定审查对象找问题"，统一模板无法提供足够的上下文——如"尝试证伪"对需求分析和代码实现意味着完全不同的操作。

## 关联

- 关联需求：F2a, F2b, F2c
- 关联约束：Review Agent 是内置 Agent，仅能通过 prompt 影响其行为
