# 委托提示词歧义：提取 ≠ 保留副本

## 错误场景

V1.9 实施中，将 task.yaml 模板从 `feat.md` 内联提取为独立文件 `feat-task.yaml`。delegation prompt 中写了：

> 保留 YAML 块作为 fallback 注释

意图是：replaced with a reference, with a brief note about fallback.

deep agent 的理解是：keep the full 68-line YAML block as an inline fallback.

**结果**：模板从 1 份变成了 2 份（外部文件 + 内联 dup），完全没有实现"提取"的意图。

## 根因

| 层面 | 问题 |
|------|------|
| **语义歧义** | "保留 YAML 块" 被理解为 "keep the YAML block"，而非 "keep a reference note about the YAML file" |
| **操作描述不清** | "extract and replace" 是两个操作，但 prompt 只明确了"extract"（创建文件），"replace"（删除 inline）未明确 |
| **验证缺失** | 只检查了"目标文件是否存在"，未检查"源文件中的内联是否已移除" |

## 教训

### 提取/迁移类任务的 prompt 写法

❌ 歧义写法：
```
保留 YAML 块作为 fallback 注释
```

✅ 精确写法：
```
将内联 YAML 块（第 X-Y 行）完整删除。
在删除位置写入从外部文件加载的引用指令。
内联 YAML 块内容不得保留在任何形式中。
```

### 操作类 prompt 的结构

对于"extract + replace"类操作，prompt 必须用 **delete-then-create** 的否定式描述：

```
MUST DO:
- DELETE 行 X 到行 Y 的全部内容（包括 ```yaml 和 ``` 标记）
- 在删除位置写入"从 <path> 加载模板"

MUST NOT DO:
- 不得以"fallback"、"备份"、"注释内保留"等任何形式保留被删除的 YAML 内容
```

### 验证清单

提取类变更的验证应包含：
- [ ] 新文件存在且内容正确
- [ ] 旧位置**不存在**被提取的内容（搜索关键字段如 `name: <slug>` 确认已删除）
- [ ] 引用路径与 MANIFEST 部署路径一致
