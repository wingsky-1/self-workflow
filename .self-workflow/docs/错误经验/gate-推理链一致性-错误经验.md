# Gate 推理链一致性 —— 踩坑记录

## 摘要

V1.5.2 的 Phase 1 需求分析将"合并 task.yaml 和 workflow.yaml"明确列为不纳入范围，并给出结论："两个文件生命周期不同，分开优于合并"。Phase 2 方案设计选择了合并，但 ADR-001 的初稿完全未提及此反转，也未解释为何推翻分析结论。Gate 2 对抗性审查将此标记为 critical（推理链断裂），要求补上反转说明后才放行。

## 根因

Phase 1→2 之间缺乏"强制回应前置约束"的机制。Phase 2 的设计者（Agent）可以自由推翻 Phase 1 的结论而不需提供理由——因为 Gate 2 审查时通常只看 Phase 2 的产物，不交叉对照 Phase 1 的输出。

这不是 Agent "故意"忽略——而是在大量的上下文切换中，Phase 1 分析文档中的一行"不纳入范围"容易被遗忘。

## 正确做法

当后续阶段的决策推翻前阶段的分析结论时：

1. **必须在决策文档中显式说明反转原因**（是证据变化？范围调整？用户方向确认？还是前阶段判断有误？）
2. **Gate 审查必须交叉对照前置阶段的约束章节**——不只审查当前阶段产物

V1.5.2 补上的反转说明结构（可作为模板）：

```
## 与阶段 1 分析结论的反转说明

阶段 1 分析得出 [前阶段结论]。当前决策反转了此结论，理由：
1. 新证据：[发现的新问题]
2. 判断调整：[为什么前阶段的推理不再成立]
3. 方向确认：[Human 的明确选择]
4. 原则取舍：[精简优于冗余 / 解决实际问题优先于防范理论风险]
```

## 建议改进

在 `feat-workflow.md` 的 Gate 2（设计审查）审查要点中增加：

> **推理链一致性**：交叉对照 Phase 1 `01-analysis.md` 中的"不纳入范围"和"约束条件"章节。如设计方案推翻了 Phase 1 的任何结论，必须确认决策文档中有显式的反转说明。

## 适用范围

任何多阶段工作流中，后续阶段推翻前阶段分析结论的场景都需要此规则。不限 feat-workflow。
