# V1 实施经验

> 从 V1 实现任务中提取的可复用经验。适用于后续所有工作流和任务。

---

## 流程设计

### 1. 线性阶段模型只适用于简单任务

**问题**：V1 的`阶段1→2→3→4→5`线性模型假设开发是顺序的，实际 V1 实施中经历了多次回溯和重构。

**建议**：复杂任务应支持 Checkpoint 回溯——从阶段 N 回阶段 M 时跳过中间已通过的 Gate。详见 `V1.5-质量保障需求设计.md`。

### 2. 方向判断比实现审查更重要

**问题**：Gate 审查发现代码 bug 没有问题，但发现"方向错了"时重构成本已经很高。V1 中目录结构重构了三次。

**建议**：方案设计 Gate 应增加方向审查节点——Review Agent 先评估"这个方向对不对"，再进入程序化验证。

### 3. 决策记录应嵌入执行流程

**问题**：ADR 被当作"总结阶段的额外工作"，导致早期决策未被及时记录。

**建议**：每个阶段末尾嵌入决策捕捉步骤——Agent 自问"这个阶段有没有值得记录的决策"。详见 `tasks/20260606-V1实现/adrs/ADR-003-关键决策即时归档.md`。

---

## Agent 行为

### 4. 先质疑，再执行

**问题**：V1 初期 Agent 对 Human 的方向判断缺少质疑，导致方向性错误（目录结构混装、安装器内联内容）。

**建议**：Agent 在执行方向前应先思考"这个方向合理吗？有没有更好的方案？"——与 Human 对齐后再执行。详见 `tasks/20260606-V1实现/adrs/ADR-007-委托优先原则.md`。

### 5. 交互规范：用工具，别用文字

**问题**：Agent 多次用纯文本列举选项，Human 需要手动输入选择。

**建议**：所有选项选择必须使用 `question` 工具。详见 `tasks/20260606-V1实现/adrs/ADR-001-交互式问答优先.md`。

---

## 项目结构

### 6. 开发目录与运行时目录必须分离

**问题**：V1 初期 `.self-workflow/` 同时放了开发文档（ADRs、审计）和运行时配置（指引、模板），导致职责混乱。

**建议**：
- 开发文档 → 项目根 `docs/`
- 运行时数据 → `.self-workflow/`
- 平台配置 → `.opencode/`（仅安装器管理）
- 任务记录 → `.self-workflow/tasks/YYYYMMDD-任务名/`

详见 `tasks/20260606-V1实现/adrs/ADR-004-目录职责划分与开发使用分离.md`。

### 7. 安装器不应承担业务内容

**问题**：初始实现将工作流指引内联在安装器代码中，导致修改指引需要修改安装器。

**建议**：安装器只负责复制文件，内容在 `templates/` 目录中独立维护。详见 `tasks/20260606-V1实现/adrs/ADR-002-安装器职责边界.md`。

### 8. 模板提取应一步到位

**教训**：feat-workflow.md 经历了 730→355→314 行的三次提取。每次只做一部分，造成重复劳动。

**建议**：在设计阶段就识别所有输出格式，一次性提取到独立模板文件。

## 协作模式

### 9. Human + Agent 的分工边界

V1 验证的有效分工：

| 角色 | 职责 | 典型行为 |
|------|------|---------|
| **Human** | 架构判断、方向把控、原则坚持 | 纠正目录结构、定义安装器边界、推动 ADR |
| **Agent** | 落地执行、发现问题、提出建议 | 按计划实施、暴露设计缺陷、记录决策 |

### 10. 文档归档原则（修正版）

| 文档类型 | 归档位置 | 本质定位 | 约束力 |
|---------|---------|---------|--------|
| 项目文档（需求、路线图、设计） | `docs/` | 描述项目本身 | ❌ 参考 |
| 实施经验（教训、最佳实践） | `.self-workflow/docs/` | Agent 沉淀的经验 | ❌ 建议 |
| 项目规范（MUST/SHOULD/MAY） | `.self-workflow/specs/` | 约束后续 Agent 行为 | ✅ Gate 强制 |
| 通用规范模板 | `installer/specs/` | 预置给所有用户的规范 | ✅ Gate 强制 |
| 任务记录（ADR、日志） | `.self-workflow/tasks/<id>/` | 历史存档 | — |

### 11. 规范应自举沉淀，非预设

**问题**：V1 预先设计了 3 个 YAML 格式的通用 spec（coding-style、architecture、workflow-protocol），格式复杂且未经验证。

**经验**：规范应该从自举开发中自然产生，而不是猜测。正确顺序：

```
自举开发 → 发现问题 → 沉淀为 spec → 
  ├─ 能融入框架？→ 改 workflow/模板/Gate 逻辑（根治）
  └─ 不能融入？→ 保留为 spec，考虑通用性
```

**建议**：
- spec 用 Markdown 格式，用户在 MUST/SHOULD/MAY 章节加列表项即可
- `@lint` 标记表示有自动工具可捕获，否则由 Review Agent 审查
- 能通过改框架解决的问题，不留到 spec 层面
