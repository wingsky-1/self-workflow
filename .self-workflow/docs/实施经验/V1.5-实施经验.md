# V1.5 实施经验

> 从 V1.5 质量加固任务中提取的可复用经验。适用于后续所有工作流和任务。

---

## 流程设计

### 1. 提前质疑设计方案，减少 Gate 返工

**问题**：V1.5 方案设计阶段加入了"强制质疑节点"——进入设计前先自问方向合理性。在自身实施中，这个节点帮助发现了 2 个设计缺陷（ADR 模板引用位置错误、Checkpoint 格式不一致），如果在 Gate 阶段才发现会多一轮返工。

**建议**：方案设计入口的质疑节点不是形式，确实能节省时间。质疑内容聚焦三个问题：
- 方案方向合理吗？（如果合理说理由，不合理提替代）
- 约束有遗漏吗？
- 风险需要关注吗？

**验证**：V1.5 自身实施验证了该机制的有效性——质疑阶段没有发现问题（方案方向正确），但后续的 Grill 审查发现了细节问题。

### 2. 设计阶段多问"有没有更简单的方案"

**问题**：V1.5 中 Checkpoint 回溯的原设计是文件级 YAML 快照（`checkpoints/` 目录存储各阶段产物状态），方案已经完整了、备选方案也分析了。用户提出"基于 Git 不是更简单吗？"——确实如此，Git tag + worktree 不仅省掉了自造数据结构的维护，还额外解决了多会话开发问题。

**建议**：设计架构方案时，主动问"现有工具中有没有能替代自定义方案的？"。Git 从来不只是版本控制——tag 天然是 checkpoint，worktree 天然是多会话工作区。标准工具往往比自造方案更可靠。

**教训**：原设计 ADR-012 的三个备选方案忽略了 Git 选项。不是因为不知道 Git，而是因为思维局限在"文件系统方案"内。

### 3. ADR 分类比一刀切更实用

**问题**：V1 的 ADR 只有一种模板结构（背景+决策+理由+影响+关联），所有决策无论大小都用同一格式。V1.5 中设计了三种模板（simple/complex/review），实际使用体验更好：
- **simple**：简单决策 5 个字段，30 秒完成
- **complex**：复杂架构决策含备选方案表+ trade-off 分析
- **review**：审查结果升级为 ADR，保留审查上下文

**建议**：模板分类优于通用模板。后续新增模板类型时也遵循"三档分类"思路：简单/标准/完整。

### 4. 安装器从项目根目录运行

**问题**：V1.5 实现阶段从 `packages/installer/` 目录运行安装器，导致文件被写入 `packages/installer/.opencode/` 和 `packages/installer/.self-workflow/`，而不是项目根目录。

**根因**：安装器使用 `process.cwd()` 作为默认目标路径，`cd packages/installer` 后 cwd 变了。

**建议**：安装器始终从项目根目录执行：`node packages/installer/index.js init`。也可考虑让安装器自动向上寻找项目根目录（通过检测 `package.json` 或 `.self-workflow/`）。

## Gate 审查

### 5. 设计审查不放过格式一致性

**问题**：V1.5 的 Grill 审查发现 Checkpoint 数据格式使用 JSON（新设计）但项目其他部分都使用 YAML（task.yaml、workflow.yaml）。这个不一致在设计时完全没注意到。

**建议**：审查设计时加一条"与现有系统的一致性"检查——尤其是数据格式、命名规范、目录结构这类基础设施层面。同一个项目中不要混用 JSON/YAML。

### 6. Gate weight 量化可用

**问题**：V1 中 Gate 降级靠 Agent 主观判断（"≤ 3 个文件"），行为不可预期。V1.5 替换为三维量化（scope/risk/user-signal），在实际 Gate 审查中计算过程透明、结果可预期。

**建议**：Gate 量化公式可推广——三个维度各取 -1/0/+1，总分映射到 skip/light/full。这个粒度对大部分场景够用。Human 可覆盖结果（要求全量审查）。

## 项目结构

### 7. 经验沉淀应在工作流中完成，而非结束后单独做

**问题**：V1 的经验沉淀是在任务结束后单独提取的，提取时部分上下文已经淡化。V1.5 在工作流中内置了经验提炼步骤（阶段 5 的"经验草稿"），提取时上下文清晰。

**建议**：经验文档的 quality 字段用 draft 标记，后续可晋升。重要的是时机——趁热打铁，不要等忘了再写。

### 8. 产物目录应包含 task.yaml

**问题**：V1 后期才引入 task.yaml，导致任务元数据分散在多个文件中。V1.5 在启动时就创建了 task.yaml，`workflow.yaml` 负责阶段状态，`task.yaml` 负责任务级元数据，职责清晰。

**建议**：新 task 目录的三件套：`task.yaml`（任务元数据）+ `workflow.yaml`（阶段状态）+ 产物目录。启动时创建，不再补充。

## 协作模式

### 9. 用户纠正设计 vs Agent 自行修正

**V1.5 验证的模式**：

| 场景 | 谁发现 | 处理方式 |
|------|--------|---------|
| ADR 模板引用错误 | Review Agent（Grill 审查） | Agent 自主修复 |
| Checkpoint 方案不合理 | Human（设计评审） | Agent 接受修改方案 |
| Git 替代文件级 Checkpoint | Human | Agent 评估后确认更优 |

**关键区分**：Agent 能修复的是实现层面的错误（模板引用路径、格式一致性）。方向层面的优化（"有没有更简单的方案"）需要 Human 判断——因为 Human 知道 Git worktree 等工具能力，Agent 可能不知道。

**建议**：Agent 在设计阶段主动说"我考虑了 X 方案，但不知道 Y 工具是否适用"——承认知识边界，减少方向性返工。
