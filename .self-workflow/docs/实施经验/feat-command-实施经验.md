# /feat 命令实施经验

> 从 `/feat` 命令自举实施中提取的可复用经验。适用于后续 Command 开发和自举验证场景。

---

## 流程设计

### 1. 需求文档与实施产物之间存在"翻译损耗"

**问题**：`docs/feat-command-需求设计.md`（v0.1）中的 task.yaml 模板与 `feat-workflow.md` 中定义的 `workflow-metadata-template.yaml` 不完全一致。需求文档写了 `created: <YYYY-MM-DD>`，但模板要求 `created: <YYYY-MM-DDTHH:mm:ss±HH:MM>`。这种细节差异在设计审查阶段才被发现——需求阶段以为"照着模板写就行"，但需求文档中并未引用模板。

**建议**：需求分析阶段产物中，对引用外部模板的字段，应直接引用模板路径而非自行描述格式。例：
```markdown
| `created` | `<YYYY-MM-DDTHH:mm:ss±HH:MM>` | 格式见 `workflow-metadata-template.yaml` |
```
而非：
```markdown
| `created` | `<当前时间>` |
```

**验证**：本次实施中，时段格式不一致在 Phase 2 设计审查时被纠正（需求文档 v0.2 补充了精确的 ISO 8601 含时区格式），避免了实现阶段的返工。

### 2. 自举验证是需求文档的"压力测试"

**问题**：需求文档写 `/feat` 命令"自动生成 workflow-id"，但没有定义 slug 为空时的回退策略。实际执行中`实现feat命令`这个描述恰好不含空格和特殊符号，但如果是 `/feat --quick `（空格后无描述）或 `/feat @#$%`（全特殊符号），slug 会为空，需要回退逻辑。

**建议**：自举验证时故意用"边界描述"测试——全特殊符号、空描述、超长描述。不要因为当前任务恰好不触发边界就忽略。

**教训**：本次 ADR-002 追加了"若结果为空 → 使用 'task'"的回退规则，就是在设计阶段主动思考边界时发现的。

## Gate 审查

### 3. Gate 重量量化公式首次实战验证

**问题**：V1.5 引入了三维量化（scope/risk/user-signal → skip/light/full），但此前未经实际 Gate 验证。

**本次实战结果**：

| Gate | scope | risk | user-signal | 总分 | 实际 weight | 行为 |
|------|-------|------|-------------|------|------------|------|
| 分析审查 | 0 | 0 | 0 | 0 | light | 程序化验证跳过（无代码），对抗性自审+用户确认 |
| 设计审查 | 0 | 0 | 0 | 0 | light→full | 虽然量化=light，但设计审查 Gate 本身声明 weight=full，保留方向审查+Grill |
| 实现审查 | 0 | 0 | 0 | 0 | light | 仅程序化验证，对抗性审查正确跳过 |
| 验证审查 | 0 | 0 | 0 | 0 | light | 程序化验证+自审 |

**发现**：设计审查 Gate 在 `feat-workflow.md` 的附录表中声明为 `full`，但量化公式计算结果为 `light`（0 分）。这里存在"固定权重"与"动态量化"的优先级冲突。当前行为是"保留固定权重的方向审查步骤 0，但对抗性审查按量化降级"。需要明确规范：量化结果与 Gate 声明权重冲突时，以哪个为准？

**建议**：量化结果应覆盖声明权重。声明权重只是"典型场景的默认值"，量化结果才是运行时实际值。需要在 `feat-workflow.md` 中显式说明此优先级。

### 4. 用户即时反馈比 Gate 更早发现问题

**问题**：设计审查 Gate 通过后，用户问"feat.md *2 位置是什么意思"，引发了对 ADR-001（文件分发策略）的重新审视。用户随后明确指示".opencode目录的文件不要处理，通过安装器安装"——这个约束在设计文档中是隐含的（ADR-001 方案 C 提到安装器写入），但没有在实现步骤中显式标注。

**建议**：设计审查 Gate 的人工确认步骤不应是"可选的"——对于涉及文件分发的决策，必须让用户确认"哪些文件手动创建、哪些通过工具生成"。

**教训**：`feat-workflow.md` 的设计审查 Gate 中"人工确认：可选"应细化为"涉及 .opencode/ 文件写入、安装器行为变更时，人工确认改为必需"。

## 项目结构

### 5. 经验应两级沉淀：task 级（上下文）+ doc 级（提炼）

**问题**：V1.5 的经验沉淀仅在 task 的 `artifacts/05-summary.md` 中记录。本次补充了 `.self-workflow/docs/` 一级索引——task 级保留原始上下文（谁、什么时候、做了什么）、doc 级提炼可复用的通用经验。

**分级规则**（本次确立）：

| 级别 | 位置 | 内容 | 读者 |
|------|------|------|------|
| task 级 | `tasks/<id>/artifacts/05-summary.md` | 本次任务的"得与失"、决策记录、经验草稿 | 后续 Agent 查看该任务史 |
| doc 级 | `.self-workflow/docs/<name>.md` | 跨任务可复用的教训、最佳实践、模式 | 所有 Agent、所有任务 |

**建议**：task 级经验写"这次遇到了什么"；doc 级经验写"下次遇到同类问题应该怎么做"。前者不删，后者不冗余。

## 协作模式

### 6. Command 开发中 Human 的关键介入点

本次实施中 Human 两次关键介入：

| 时机 | Human 输入 | 影响 |
|------|-----------|------|
| 设计审查后 | "feat.md *2 位置是什么意思" | 暴露 ADR-001 的沟通不充分——Human 不理解双文件同步的需求 |
| 确认后 | ".opencode目录的文件不要处理，通过安装器按照" | 实现策略调整——`.opencode/` 不手动创建，安装器是唯一写入入口 |

**规律**：Human 对"文件归谁管"（`.opencode/` 由安装器管、`docs/` 由人管、`.self-workflow/` 共同管）的边界比 Agent 更敏感。Agent 倾向于"需要什么就创建什么"，Human 倾向于"通过既定机制创建"。这是需要 Agent 学习的项目特定约束。

**建议**：涉及 `.opencode/` 文件创建时，Agent 应先确认"通过安装器写入还是直接创建？"。此约束应写入项目 spec。
