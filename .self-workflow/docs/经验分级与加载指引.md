> 版本：v0.1
> 日期：2026-06-06
> 来源：`/feat` 命令自举实施评审 + `.self-workflow/todo.md` #9

## 一、分类定义

`.self-workflow/docs/` 中的经验文档按三种分类组织，每类有不同的命名约定和 Agent 加载时机。

| 分类 | 文件名模式 | 内容 | 示例 |
|------|-----------|------|------|
| **实施经验** | `*-实施经验.md` | 从具体任务中提取的教训、验证的模式、踩过的坑 | `feat-command-实施经验.md` — "安装器从项目根目录运行" |
| **参考模式** | `*-参考模式.md` | 可复用的解决方案模板（设计模式、架构模式、流程模式） | `双文件分发模式：安装器模板 + 运行时文件` |
| **错误经验** | `*-错误经验.md` | 遇到过的陷阱、错误模式、失败原因 | `从 packages/installer/ 运行安装器导致路径错误` |

### 分类判断标准

Agent 在 Phase 5 写经验时，按以下标准选择分类：

| 你遇到了... | 写入... | 原因 |
|-----------|---------|------|
| 框架缺陷/设计漏洞 | `*-实施经验.md` | 这类经验帮助后续任务避免重复踩坑 |
| 可推广的解决方案 | `*-参考模式.md` | 设计阶段可以检索并复用 |
| 操作错误/环境陷阱 | `*-错误经验.md` | 任务启动前预防性加载 |

### 一个经验跨分类时的处理

如果一个经验同时属于多个分类（如"安装器路径问题导致框架缺陷"），创建多个文件或使用复合内容：
- 优先拆分为多个文件（如 `installer-错误经验.md` + `installer-实施经验.md`）
- 如果主题高度相关不便拆分，可在文件名中体现（如 `installer-实施经验+错误经验.md`）

## 二、渐进式披露机制

Agent 在不同阶段按需加载经验文档，避免全文加载浪费上下文。

### 加载时机

| 阶段 | 触发条件 | 检索范围 | 操作 |
|------|---------|---------|------|
| **任务启动** | 新任务开始 | 扫描 `.self-workflow/docs/` 文件名 | 匹配任务描述关键词 → 加载匹配文件的**摘要**（第一段 + 关键条目），不加载全文 |
| **设计阶段** | 进入方案设计 | `*-参考模式.md` | Agent 自问："之前解决过类似架构问题吗？" → 检索匹配的参考模式 |
| **遇到错误** | 错误发生时 | `*-错误经验.md` | Agent 自问："这个错误模式之前遇到过吗？" → 检索匹配的错误经验 |
| **Phase 5** | 任务完成 | 所有分类 | 判断本次是否有可沉淀经验 → 写入对应分类文件 |

### 检索方式

Agent 通过文件名关键词匹配来判断相关性：
- 任务描述中的关键词 → 匹配文件名中的领域（如 `feat-command` 匹配任务 `feat-*`）
- 错误信息中的关键词 → 匹配错误经验文件名
- 设计问题中的关键词 → 匹配参考模式文件名

> 当前为 V1.5 手动检索。V2 将考虑系统自动注入（Hook 机制）。

## 三、文件格式约定

### 文件名

```
<领域>-<分类>.md
```

示例：
- `feat-command-实施经验.md`
- `installer-参考模式.md`
- `路径处理-错误经验.md`

### 内容结构建议

```markdown
# <标题>

> 分类：<实施经验 | 参考模式 | 错误经验>
> 领域：<涉及的模块/命令>
> 日期：<创建日期>
> 关联任务：<workflow-id>

## 摘要

<一段话概括核心经验>

## 详细内容

<具体经验描述>

## 适用场景

<什么情况下 Agent 应加载此经验>
```

## 四、与任务级经验的区分

| 维度 | task 级经验 | doc 级经验 |
|------|-----------|-----------|
| 位置 | `tasks/<workflow-id>/artifacts/05-summary.md` | `.self-workflow/docs/` |
| 作用域 | 本任务 | 跨任务 |
| 内容 | 本次工作流得与失 | 可复用的模式/教训 |
| 创建时机 | Phase 5 自动 | Phase 5 判断后手动 |
| 读取者 | 回顾本任务时 | 启动新任务时 |

**判断何时从 task 级提升到 doc 级**：
1. 本任务是**首次发现**某个模式或缺陷（不是已知问题的重复）
2. 经验**不依赖本任务的特殊上下文**，可在其他任务中独立使用
3. 经验描述的是**框架/流程/Agent行为**层面的问题，而非单次实现决策
