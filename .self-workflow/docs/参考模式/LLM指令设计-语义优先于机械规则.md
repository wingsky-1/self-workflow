# LLM 指令设计原则：语义理解优先于机械规则

## 问题

当 LLM Agent 需要处理"语义相关"的输出（如生成有辨识度的名称、撰写有意义的摘要）时，使用机械规则（字符串替换、正则匹配）往往产生低质量结果。

例如：Slug 生成规则"大写转小写 + 标点替换 `-`"将 `V1.9版本` 变为 `v1-9版本`——丢失了语义信息。问题是**输入本身不包含足够信息**（`V1.9版本` 是版本标记，而非工作内容描述）。

## 方案

将指令分为两层：

### 第一层：语义理解（主力）
利用 LLM 的理解和生成能力处理语义问题：
- 读取上下文（如 todo.md 关联项）理解任务的**实际工作内容**
- 提炼核心关键词生成有辨识度的输出
- 语言策略根据项目特点选择（中文项目用中文，英文项目用英文）

### 第二层：安全网（兜底）
保留最小必要的机械规则，处理**非语义的、确定性的约束**：
- 路径非法字符过滤
- 长度截断
- 连续分隔符压缩

## 适用场景判断

| 任务类型 | 适用方式 | 示例 |
|----------|---------|------|
| 生成名称/标题/描述 | **语义优先** | slug、commit message、task title |
| 路径安全检查 | 机械规则 | 禁止 `/`、截断到文件系统限制 |
| 格式转换 | 机械规则 | JSON↔YAML、编码转换 |
| 内容提炼 | **语义优先** | 摘要生成、关键词提取 |

## 反模式

❌ 用机械规则处理语义问题：
```markdown
将输入中的所有标点替换为 `-`，大写转小写，截断至 40 字符
```

✅ 语义理解 + 机械安全网：
```markdown
理解任务的工作内容，提炼 2-4 个核心关键词生成 slug。
安全网：空格/标点→`-`，压缩连续 `-`，截断至 40 字符。
```

## 关联经验

- **`Spec 文档写法——规范而非步骤`**（`参考模式/`）—— 与本文的差异：本文讲语义层面的指令设计（LLM 理解优先于机械替换），该文档讲格式层面的规范写法（MUST/SHOULD/MAY、规则而非步骤脚本）。该文档是本文"语义优先"原则在 spec 领域的具体落地。
