# 提取/删除类变更——防执行偏差

## 案例

V1.9 将 task.yaml 模板从 `feat.md` 内联提取为独立文件 `feat-task.yaml`。

**预期结果**：1 份外部模板 + 1 行引用指令

**实际结果**：1 份外部模板 + 68 行内联副本——完全没有实现"提取"意图。

## 根因：委托提示词歧义

Prompt 中写了"保留 YAML 块作为 fallback 注释"——意图是"保留引用注释"，deep agent 理解为"保留完整 68 行 YAML 作为内联 fallback"。

| 层面 | 问题 |
|------|------|
| 语义歧义 | "保留 YAML 块"被理解为保留内容而非保留引用 |
| 操作描述不清 | prompt 只明确了 extract（创建文件），未明确 replace（删除内联） |
| 验证缺失 | 只检查"目标文件是否存在"，未检查"源文件内联是否已移除" |

## 方案一：防歧义的 Prompt 写法

对于 extract + replace 操作，用 **delete-then-create** 的否定式描述：

```
MUST DO:
- DELETE 行 X 到行 Y 的全部内容（包括 ```yaml 和 ``` 标记）
- 在删除位置写入"从 <path> 加载模板"

MUST NOT DO:
- 不得以"fallback""备份""注释内保留"等任何形式保留被删除内容
```

❌ 歧义写法：`保留 YAML 块作为 fallback 注释`

✅ 精确写法：`将内联 YAML 块完整删除。在删除位置写入引用指令。不得保留在任何形式中。`

## 方案二：双重验证

对替换/删除类变更，每条验收标准写为**一对检查**：

```markdown
- [ ] A2a：正向——新文件存在且内容正确
- [ ] A2b：反向——旧位置不存在被提取的内容
```

**通用自检**（标记验收标准为 PASS 前）：

> 这条变更是否涉及"删除"或"替换"旧内容？
> 如果是 → 我是否同时检查了"新内容存在"**和**"旧内容不存在"？

## 验证清单

- [ ] 新文件存在且内容正确（正向）
- [ ] 旧位置不存在被提取的内容——搜索关键字段确认已删除（反向）
- [ ] 引用路径与 MANIFEST 部署路径一致

## 关联经验

- **`Phase Gate 验证不能形式化`**（`错误经验/`）—— 与本文的差异：本文讲"正确的验证怎么做"（正向+反向双重检查），该文档讲"什么是不对的验证"（只做文件存在性检查的形式化验证）。两篇合在一起形成完整的验证方法论——先知道什么不对，再知道什么对。
