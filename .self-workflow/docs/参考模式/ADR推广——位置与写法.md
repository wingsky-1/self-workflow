# ADR 推广——位置与写法

Phase 5 晋升 ADR 从 `tasks/<id>/adrs/` 到 `docs/关键决策/` 时，需同时关注两个维度：**放哪里**和**怎么写**。

## 位置：权威来源唯一

完整内容只在 `docs/关键决策/` 中。产物文档（如 `02-design.md`）仅保留摘要表 + 链接。

**反模式**（应避免）：

```
02-design.md                  adrs/ADR-001.md
┌─────────────────┐          ┌─────────────────┐
│ ## ADR-001       │          │ # ADR-001        │
│ 背景：...        │  重复！   │ 背景：...        │
│ 备选方案：...    │ ←──────→ │ 备选方案：...    │
│ 选择：...        │          │ 选择：...        │
└─────────────────┘          └─────────────────┘
```

**正确模式**：

```
02-design.md                  adrs/ADR-001.md
┌──────────────────────┐     ┌──────────────────────┐
│ ## 架构决策记录       │     │ # ADR-001 (完整内容)  │
│ | ADR | 决策 | 选择|  │     │ 背景、备选方案、      │
│ | 001 | xxx  | x   |──┼────→│ 选择、理由、后果      │
│ | 002 | yyy  | y   |  │     └──────────────────────┘
│ > 完整内容见 adrs/    │
└──────────────────────┘
```

**判断标准**：
- ✅ 完整内容出现在 `docs/关键决策/` → 正确（权威来源）
- ✅ 以摘要表 + 链接形式出现在设计文档 → 正确（引用）
- ❌ 完整内容同时出现在两个文件 → 重复，需精简

## 写法：面向未来，不提历史

ADR 从 task 晋升到 `docs/` 时容易直接复制原文——保留"V1.8 需要…""已废弃…""超驰 ADR-008…"等历史语境。但 `docs/` 是持续服务项目的运行时文档。

**三项规则**：

- **面向未来**：描述当前规则是什么，不描述规则是怎么来的
- **精炼**：任务 ADR 的完整决策过程留在 task 目录；推广版只保留决策结论 + 适用规则
- **不提历史**：删除"已废弃""超驰 ADR-xxx""V1.x 需求""从 Skill 迁移"等——这些留在任务 ADR 中

**命名**：按内容命名（如 `spec文件格式标准.md`），去 `ADR-00X-` 前缀。

**前言**：遵循 `docs/README.md` 的 frontmatter 格式（title/category/tags/date/source/quality）。

## 推广检查清单

- [ ] 完整内容只在 `docs/关键决策/` 中（不在产物中内联）
- [ ] 标题不含 `ADR-` 前缀
- [ ] 无"V1.x 需求""已废弃""超驰 ADR-xxx"等历史语境
- [ ] 内容精炼为当前规则（而非决策过程的叙事）
- [ ] frontmatter 完整（title/category/tags/date/source/quality）

## 关联经验

- **`跨任务决策沉淀与 ADR 治理`**（`关键决策/`）—— 与本文的差异：本文讲推广的"怎么做"（位置+写法），该文档讲推广的"什么时候做"（晋升标准：跨任务性/框架级/持久性/可引用性）。决定哪些 ADR 值得推广时需读该文档。

- **`ADR 与设计文档交叉一致性审查`**（`参考模式/`）—— 与本文的差异：本文定义"ADR 不应内联到产物"的规则，该文档定义如何审查这个规则是否被遵守（三层交叉验证——声明层/设计层/实施层）。规则 + 执法需同读。
