# /todo-organize 命令实现方案

> 最后更新：2026-06-07

## 模块定位

/todo-organize 是一条**面向 todo 生命周期治理**的 Agent 命令，负责对 .self-workflow/todo.md 进行结构性评审与排期调整。

**一句话**：5 阶段管道——读取现状 → 评审新增项 → 整理活跃版本 → 生成排期建议 → 确认写入，最终输出一份经 Human 确认的、结构清晰的路线图。

**在上层系统中的位置**：

- **父级系统**：.self-workflow/todo.md 是项目任务总览文件，由 /feat 工作流在 Phase 5（总结）结束时通过 Compound 机制附加待办项，同时也可由 Human 直接在文件中编写。
- **协作关系**：/todo-organize 不参与 /feat 的单次开发循环，而是**周期性维护命令**——在若干 /feat 周期之后，当"新增"章节堆积、优先级过期或版本结构混乱时，由 Human 主动调用整理。
- **与 /feat 的区别**：/feat 驱动单次开发（分析→设计→实现→验证→总结），而 /todo-organize 负责对多个待办项做**批量评审和排期**，不直接涉及代码开发。

## 架构概览

### 5 阶段管道

`
/todo-organize [--dry-run]
    │
    ▼
阶段 1: 读取现状
    ├── 读取 todo.md 全文
    ├── 快速感知项目上下文（README / specs / commands / docs / errors）
    │   （条件性扫描——不存在则跳过，不全面审计）
    ▼
阶段 2: 评审新增项
    ├── 路径定位——对含路径描述的项先 glob/read 确认目标存在性
    ├── 优先级判断（P0/P1/P2 框架）
    ├── 归属判断——优先并入已有活跃版本，避免默认新建
    ▼
阶段 3: 整理活跃版本
    ├── 归档已完成版本（所有项 [done] → 移入"已关闭"折叠）
    ├── 调整优先级标记
    ├── 重排序版本段（P0 在前，同优先级按版本号）
    └── 合并重复项
    ▼
阶段 4: 生成排期建议
    ├── 产出一张排期建议表（版本/优先级/内容/来源）
    ├── 逐版本校验项数（3+ 拆分预警，1 项建议合并）
    ▼
阶段 5: 确认写入
    ├── 用 question 工具展示排期建议表 + 结构性变更摘要
    ├── 待 Human 确认后写入 todo.md
    └── --dry-run 则仅预览，不写入
`

### 与 .self-workflow/todo.md 文件结构的关系

`
.self-workflow/todo.md（假设结构）
├── ## 新增（待评审排期）  ← 阶段 2 的核心输入
├── ## V1.x                  ← 阶段 3 的活跃版本
├── ## V1.y                  ← 阶段 3 的活跃版本
├── ## 延后                   ← 阶段 3 处理
└── ## 已关闭                 ← 阶段 3 的归档目标
`

### 文件分布

| 文件 | 角色 | 管理方式 |
|------|------|---------|
| .opencode/commands/todo-organize.md | Agent 命令定义（执行指令 + 阶段流程 + 原则） | 由安装器管理，模板在 packages/installer/templates/commands/todo-organize.md |

## 关键数据流

### 阶段 2 优先级判断框架

评审新增项时，按此条件矩阵决定 P0/P1/P2：

| 条件 | 优先级 |
|------|--------|
| 框架核心机制不工作/被绕过 | P0 |
| 直接影响当前每轮对话质量（如 Gate 跳过、实现无确认） | P1 |
| 效率/体验改善，非阻塞 | P2 |
| 依赖其他未完成能力 | P2 或延期 |

设计依据：优先级锚定在**框架健康度**上，而非简单的业务急迫性。保证 P0 项直接修复框架级缺陷，P1 项保障每轮对话核心体验不被绕过。

### 版本归属算法

`
输入：新增项 descriptors[]
输出：项 → 目标版本映射

步骤：
1.  列出所有活跃版本（非"已关闭"、非"延后"、非"新增"）
2.  for each 新增项:
      // 优先检查可并入的已有版本
      // 条件：主题匹配 + 版本项数 ≤ 3
3.      找到匹配版本 → 归入该版本
4.      未找到匹配 → 创建新版本
5.  // 避免默认新建版本的原则在代码层面体现为先遍历后新建
`

- **粒度约束**：每版本 2~3 项，单次 /feat 可消化
- **超限处理**：超过 3 项触发拆分建议，仅 1 项（非延后/远期）触发合并建议

### Diff-before-write 确认模式

阶段 5 的写入确认流程是区别于常规直接写入的关键设计：

1. 生成**排期建议表**（Markdown 表格形式）
2. 生成**结构性变更摘要**（如：新增 #3 移入 V1.8、V1.5 归档、V1.6 降级为延后等）
3. 使用 question 工具呈现给 Human，等待确认
4. 确认后写入 todo.md

`
phase 4 output → question(展示排期建议表 + 变更摘要)
    ├── Human 确认 → write(todo.md)
    └── Human 拒绝 → 不写入，结束
`

--dry-run 模式在阶段 4 输出后直接结束，不发起 question。

### 条件性上下文感知（阶段 1）

扫描策略的核心考量——**只感知，不审计**：

- 目标不是读懂全部项目细节，而是为阶段 2 的优先级判断提供侧写
- 每项场景限制 1~2 次查询，不存在即跳过
- 至少了解主题分类和关键概念，**仅统计文件数量不提供有效信息**

### 错误处理

| 场景 | 行为 |
|------|------|
| todo.md 不存在 | 提示先运行 self-workflow init |
| "新增（待评审排期）" 章节缺失 | 警告后继续，跳过阶段 2 |

## 与相关模块的关系

### 与 /feat 工作流

/todo-organize 和 /feat 共享同一个 todo.md，但职责正交：

| 维度 | /feat | /todo-organize |
|------|-------|---------------|
| 调用时机 | 每次开发 | 周期性维护 |
| 操作范围 | 附加 1 项到"新增" | 批量评审 + 排期全量 |
| 交互模式 | 5 阶段 + 4 Gate | 5 阶段（无 Gate） |
| 写入策略 | Compound 自动归档 | Diff-before-write + Human 确认 |

### 与 Gate 审查机制

当前 /todo-organize **不引入 Gate**。原因：

- /todo-organize 的操作对象是 todo.md（规划文件），而非代码/架构
- 输出是一份排期建议表，Human 在阶段 5 通过 question 确认——这本身构成了审查屏障
- 未来若 /todo-organize 获得对 todo.md 的自动写权限（去除 Human 确认步骤），则需引入 Gate 作为替代质量关口。见 docs/实现方案/gate-审查机制实现方案.md。

### 与 Plugin 注入机制

命令定义文件 todo-organize.md 存在于 .opencode/commands/ 目录下，由 Plugin 自动发现并注入到 Agent 的 system prompt 中。遵循 docs/实现方案/plugin-注入机制实现方案.md 描述的自动发现机制。
