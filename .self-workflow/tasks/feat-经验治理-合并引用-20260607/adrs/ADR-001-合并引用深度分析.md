# ADR-001：文档合并与交叉引用深度分析

> 原始分析文档归档。本文档为 exp-governance 审查第二阶段——24 份文档的逐对去重与引用分析。

## 决策背景

一期治理（feat-经验治理-优化执行-20260607）完成了 frontmatter/tag/source/quality 的批量修复。在此基础上，需要对文档间的内容重叠和阅读路径进行深度分析。

## 分析结论

### 应合并（3 组）

| 合并对 | 关系类型 | 理由 |
|--------|---------|------|
| installer-错误经验 + design-可定制性声明验证 | 同根异面 | 同一根因（三层架构认知缺失）的执行面+设计面 |
| 产物权威来源唯一 + 推广ADR-运行时文档 | 互补 | WHERE+HOW——Agent 晋升 ADR 时需同时回答两个问题 |
| 验收验证双重检查 + 委托提示词歧义 | 因果链 | 同一 V1.9 模板提取事件——根因+方案，分离造成诊断链路断裂 |

### 应引用（10 组）

| 方向 | 级别 | 理由 |
|------|------|------|
| 跨任务决策沉淀 → ADR推广 | MUST | WHEN+HOW 互补——晋升时需两篇同读 |
| ADR推广 → 跨任务决策沉淀 | MUST | 同上（反向） |
| ADR推广 → ADR交叉一致性审查 | MUST | 规则+执法——审查前必须先读规则 |
| ADR交叉一致性审查 → ADR推广 | MUST | 同上（反向） |
| 提取删除防偏差 → phase-gate | MUST | 正面方案+反面警告——完整验证方法论 |
| phase-gate → 提取删除防偏差 | MUST | 同上（反向） |
| gate强制步骤 ⇄ gate推理链 | SHOULD | 同领域互补——排查时两篇都读 |
| readme权威源 → spec注入架构 | SHOULD | 内容模式+运行时机制 |
| plugin-session → spec注入架构 | SHOULD | 参考实现+设计决策 |
| spec文档写法 ⇄ LLM指令设计 | SHOULD | 格式层+语义层 |

### 不合并（其余）

17 份文档独立完整——V1/V1.5 版本历史、spec 三项独立决策、plugin 两项独立记录等。

## 执行

详见 Phase 2 方案设计文档，按 M1→M2→M3 顺序合并，再添加引用。
