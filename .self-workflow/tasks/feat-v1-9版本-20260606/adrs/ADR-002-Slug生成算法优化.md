# ADR-002：Slug 语义生成

## 背景

当前 slug 生成规则（定义于 `feat.md` 步骤 1）是对用户输入的**机械变换**：CJK 保留 + ASCII 小写 + 标点替换 `-` → 压缩 → 截断。这导致：

- `V1.9` → `v1-9`（丢失语义，看不出是版本号）
- `V2.1.3` → `v2-1-3`（完全不可读）
- `开始V1.7` → `开始v1-7`（`开始` 无信息量——每个任务都是"开始"）
- `先做v1-5-2的需求` → slug 是临时笔记语言的直译，不是正式名称

根本问题是：**用户输入本质上是"任务标识"而非"任务描述"**。`V1.9版本` 是版本标记，不是对工作内容的描述。用机械规则处理这类输入，产出自然无辨识度。

## 决策

**从机械规则改为语义理解**。Agent 在生成 slug 时：

1. **理解任务**：阅读用户描述 + todo.md 中关联的 todo 项，理解任务的实际工作内容
2. **提炼关键词**：从工作内容中提取 2-4 个核心关键词/短语，用 `-` 连接
3. **优先语义而非字面**：将"V1.9版本"理解为"安装器模板重构与清理"而非"v1.9"
4. **语言策略**：中文项目使用中文关键词（如 `安装器重构-模板提取`），英文项目使用英文；嵌入的版本号/技术术语按原样保留
5. **辨识度要求**：生成的 slug 应在所有 task 中具有唯一辨识度——人类通过 slug 即可辨别该任务做了什么

**保留的机械规则**（作为安全网，对关键词短语做最终清理）：
- 空格/标点 → `-`
- 连续 `-` 压缩
- 首尾 `-` 去除
- 截断至 40 字符

**不再执行的规则**：
- ~~大写转小写~~（中文 slug 无此需求）
- ~~所有标点无条件替换为 `-`~~（语义生成已避免不必要标点）

**示例对比**：

| 用户输入 | 当前 slug | 新 slug | 理由 |
|----------|----------|---------|------|
| `/feat V1.9版本` | `v1-9版本` | `安装器重构-模板清理` 或 `installer-refactor` | 提取 todo 中 4 项实际工作内容的关键词 |
| `/feat 实现用户登录模块（含OAuth2.0）` | `实现用户登录模块-含oauth2-0` | `用户登录-OAuth2` | 提炼核心功能关键词 |
| `/feat 修复 README 中的拼写错误` | `修复-readme-中的拼写错误` | `readme-typo-fix` | 英文作为 slug 风格可接受，简洁有辨识度 |

## 理由

**方案 A（采纳）**：Agent 语义理解生成 slug
- 根本解决了"用户输入≠工作内容"导致的辨识度问题
- 利用 LLM 的语义理解能力，而非用机械规则对抗语义缺失
- 生成结果与任务实质内容吻合，人类可读性显著提升
- 保留机械规则作为安全网，不产生路径非法字符

**方案 B（放弃）**：优化机械规则（保留数字间点号 + 大小写）
- 治标不治本——`V1.9版本` 仍只是一个版本号，不反映 4 项重构的工作内容
- `开始v1-7` 中的 `开始` 仍然是噪音
- 机械规则无法解决"用户输入为版本标记/临时笔记"这类根本性的语义缺失问题

**方案 C（放弃）**：要求用户输入更好的描述
- 增加用户负担（`/feat` 的便利性是其核心价值）
- 用户输入版本号是最自然的用法，不应强迫用户改变习惯

## 影响

- **title 字段**：由 Agent 根据任务内容生成精炼标题（如 `V1.9 安装器重构与模板清理`），而非直接使用用户输入的 `V1.9版本`
- **description 字段**：Agent 生成，包含 todo 项摘要和任务目标，而非原始输入的简单复制
- **commit message**：与 slug 命名优化配合，形成一致的语义表达
