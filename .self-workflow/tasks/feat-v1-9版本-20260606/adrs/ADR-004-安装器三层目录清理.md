# ADR-004：安装器三层目录清理

## 背景

`packages/installer/` 目录下存在三个包含 `.opencode/` 和 `.self-workflow/` 副本的位置：

1. `packages/installer/templates/` —— 权威模板源（MANIFEST 引用）
2. `packages/installer/.opencode/` + `packages/installer/.self-workflow/` —— **废弃死代码**
3. `/` 根目录下的 `.opencode/` + `.self-workflow/` —— 运行时部署副本（由安装器生成）

探索发现：层 2（`installer/.opencode/` 和 `installer/.self-workflow/`）未被任何代码（index.js、grep 扫描）引用，内容不完整（`.opencode/commands/` 缺 `feat.md`）、已漂移（`review-agent.md` 3554 vs 3779 字节）、docs/ 和 tasks/ 为空。运行 `init --force` 时完全不涉及它们。

## 决策

1. **删除** `packages/installer/.opencode/` 和 `packages/installer/.self-workflow/`（废弃死代码）
2. **保留** `packages/installer/templates/` 作为唯一模板源（已是唯一源，无需变更）
3. **新增** `packages/installer/README.md` 说明三层职责边界：
   ```
   templates/       # 模板源（权威源，修改后 init --force 同步）
   init --force     # 安装器分发（从 templates/ 复制到运行目标）
   .opencode/       # 运行时 OpenCode 文件（由安装器管理）
   .self-workflow/  # 运行时工作流文件
     ├── configs/   # 由安装器管理（需通过模板源修改）
     ├── specs/     # 由安装器管理（需通过模板源修改）
     ├── docs/      # 可由工作流直接写入（无需通过模板源）
     └── tasks/     # 可由工作流直接写入（无需通过模板源）
   ```
4. MANIFEST 自动发现不纳入本版本（风险/收益不匹配）

## 理由

**方案 A（采纳）**：删除废弃副本 + templates/ 唯一源
- 消除维护负担（3 份副本降到 1 份源 + 1 份运行时部署）
- 消除混淆风险（"到底应该改哪个？"）
- README 文档化职责边界，新贡献者可快速理解架构

**方案 B（放弃）**：直接在 `.opencode/` + `.self-workflow/` 维护（无模板）
- 丧失"模板"语义——无法 `init` 新项目、无法 `--force` 重置默认值
- 违反"安装器管理"的既有设计（ADR-002：安装器职责边界）

**方案 C（放弃）**：保留 installer 内部副本并修复漂移
- 引入第 3 份需维护的副本，职责与 templates/ 重叠
- 违反"产物权威来源唯一"原则（参考模式文档）
