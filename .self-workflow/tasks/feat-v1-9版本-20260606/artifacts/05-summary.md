## 得与失

### 做得好的

1. **ADR 驱动设计审查**：方向审查（Gate 2）发现 ADR-001 部署路径与反对理由矛盾——`configs/templates/` 混放违背了 ADR 自身的"职责分离"原则。在实施前修正为 `configs/tasks/` 避免了技术债务。

2. **语义 slug 替代机械规则**：认识到"用户输入≠工作内容"（如 `V1.9版本` 只是版本标记），用 LLM 语义理解替代机械变换是根本性的正确方案。这呼应了项目的核心理念：利用 AI 而非对抗 AI。

3. **deep agent 单次委托**：4 项变更涉及共享文件（feat.md、feat-workflow.md），拆分为并行子 Agent 会导致编辑冲突。单次 `deep` 代理串行执行全部变更，3.5 分钟完成——效率与正确性兼得。

### 可以改进的

1. **阶段 1 决策推导初版缺失**：初版 01-analysis.md 直接从 todo→决策，被指出是黑盒。补充"决策推导"章节后（todo→发现→方案→理由），可追溯性显著提升。**后续每个 /feat 都应默认包含此章节**。

2. **`init --force` 覆盖风险**：`--force` 会覆盖用户定制的模板文件。虽然 `init` 无 `--force` 时跳过已有文件，但用户可能在"想更新其他配置"时误用 `--force` 导致定制丢失。长期应考虑分离"更新"和"覆盖"操作。

## 经验草稿

### 实施经验

#### ADR 内部一致性审查经验

**背景**：设计审查前，ADR-001 声明"拒绝与阶段产物模板混放"（方案 C），但 MANIFEST 部署路径仍是 `configs/templates/`。方向审查发现了这一矛盾。

**教训**：ADR 的决策正文与实施细节（MANIFEST 条目、部署路径）必须保持一致。审查时需同时阅读 ADR（声明层）和设计文档（实施层），交叉验证一致性。

**适用场景**：任何涉及 ADR 的设计审查——审查者应明确检查"ADR 说了什么→设计文档写什么→实施代码做什么"三层一致性。

#### 死代码清理经验

**背景**：`packages/installer/.opencode/` 和 `.self-workflow/` 是历史遗留，17 个文件未被任何代码引用、内容已漂移、目录为空。其存在仅增加认知负担——"应该修改 templates/ 还是 installer/.opencode/?"

**教训**：
- 目录职责从"双层"（模板源+内部副本）变为"单层"后，改什么改哪里一目了然
- 清理前先 grep 确认无外部引用，避免误删
- 用 README 文档化当前架构，防止未来再次产生混淆

#### LLM 指令设计经验：语义 > 规则

**背景**：Slug 生成最初是机械规则（数字间点号保留+大小写保留），后改为语义理解（Agent 读 todo 提炼关键词）。前者是"用规则对抗语义缺失"，后者是"发挥 LLM 的长处"。

**教训**：面向 LLM 的指令设计中，当规则处理的是"语义问题"（如"生成有辨识度的名称"）时，机械规则永远是次优解。应直接利用 LLM 的理解和生成能力，机械规则仅作为安全网（路径合法字符检查）。

**适用场景**：所有 Agent 行为指令的设计——判断"这个任务是需要精确规则，还是需要语义判断"，选择对应的指令风格。
