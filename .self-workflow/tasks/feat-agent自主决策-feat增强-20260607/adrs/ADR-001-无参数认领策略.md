# ADR-001：/feat 无参数认领策略

## 背景

当前 `/feat`（无参数）仅展示任务仪表盘。V1.11 要求 Agent 自动分析 todo.md、识别未认领版本任务、启动工作流。需决策认领粒度和判断规则。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A：版本段认领 | 认领整个版本段（如 V1.12 的 3 项一起），启动一个 feat 工作流 | 与现有 `/feat V1.11` 模式一致；一个版本 = 一个 task 目录；实现简单 | 版本粒度过粗时可能任务过多 |
| B：单 item 认领 | 认领单个 todo 项，每项一个 feat 工作流 | 更灵活、可并行 | 与现有粒度不匹配；需要额外拆分逻辑 |

## 选择

**方案 A**（版本段认领）

## 理由

1. 与现有 `/feat` 用法一致——用户输入 `/feat V1.11` 即认领整个版本段的所有项
2. 粒度匹配当前 task 目录设计：1 个版本 = 1 个 task = 1 个 workflow-id
3. 实现最简单：判断依据仅需检查版本段是否有对应的 in_progress task

## 认领判断规则

- 解析 todo.md 的 `## Vx.y.z` 版本段
- 跳过 `## 已关闭` 和 `## 新增（待评审排期）`
- 从版本段标题用正则 `/V\d+\.\d+(?:\.\d+)?/` 提取版本号（如 "V1.11"）
- 对 `.self-workflow/tasks/*/task.yaml` 的 description 首行用相同正则提取版本号
- **精确字符串比对**（非子串匹配），大小写敏感
- 第一个无 in_progress 任务的版本段 → 自动认领

## 后果

- `/feat`（无参数）行为从"查看仪表盘"变为"自动认领"（breaking change）
- 仪表盘仅作为无未认领版本时的兜底
- task.yaml description 首行格式需约定为 `Vx.y.z ...`（见数据模型补充约定）
