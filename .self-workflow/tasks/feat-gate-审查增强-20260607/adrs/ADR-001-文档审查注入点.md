# ADR-001：文档变更审查的注入点选择

## 背景

V1.10 需求：将 `docs/`、`configs/`、`specs/` 的文档变更纳入 Gate 审查范围。当前 feat-workflow 的 4 个 Gate 仅审查阶段产物（`artifacts/*.md`），文档变更绕过审查体系。

需要决策：文档审查在哪个环节执行。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A（Per-Gate） | 每个 Gate 入口检测本阶段文档变更，纳入该 Gate 对抗性审查 | 及早发现问题；与现有 Gate 体系融合 | 每个 Gate 增重；Phase 5 无 Gate 需特殊处理；Gate weight=skip 时文档审查也会被跳过 |
| B（Compound 集中） | 仅在 Compound 归档阶段增加文档审查步骤 | 简单，单一审查点；不干扰现有 Gate 流程 | 发现问题晚，修复成本高；从变更到审查间隔长 |
| C（混合） | Gate 做轻量检测+标记；Compound 做完整审查 | 平衡及时性和简洁性；Gate 不增重；文档审查不依赖 Gate weight | 两处修改，但改动量都很小 |

## 选择

方案 C（混合）

## 理由

1. **Gate 保持轻量**：Gate 仅增加一条检查清单项（"本阶段是否产生文档变更？"），检测到则标记到 task.yaml，不触发额外的对抗性审查流程。
2. **Compound 做实质审查**：Compound 已有交叉引用检查步骤，文档审查是交叉引用检查的自然扩展。在 Compound 做完整审查，不受 Gate weight 影响。
3. **Phase 5 自然覆盖**：Phase 5 无 Gate，其文档产出直接在 Compound 被审查。Gate 2-4 的文档变更也会被 Compound 覆盖（Compound 扫描所有阶段的变更）。
4. **与现有错误经验一致**：Compound 的交叉引用检查已有"确认产物完整性"的职责，文档审查是同一职责的扩展。

## 影响

- **正面**：Gate 流程不受影响，文档审查独立且强制；Compound 的一次性审查覆盖所有阶段的文档变更。
- **负面**：审查时机稍晚（Compound 在最后），但文档变更通常是补充性的，晚期审查风险可控。

## 反对意见

- 方案 A 的支持理由"及早发现问题"有一定道理，但：
  - Gate 已足够繁忙，不应因文档问题阻断阶段推进
  - 文档问题通常是格式/规范类而非功能性，晚期修复成本低
  - 如果文档变更是"设计文档的一部分"（如在 Phase 2 创建 spec），该 spec 本身会成为 Phase 2 的产物（纳入 Gate 2 审查），不适用此场景

- **风险分级补充**（审查反馈后增加）：上述"文档问题通常是格式类"假设存在例外——如果 Phase 2 创建的 spec 包含错误技术指引（如与现有 ADR 矛盾的接口定义），这是**内容级别**错误而非格式错误。对此类风险的分级处理：
  - **spec 类文档变更**（.self-workflow/specs/）：在 Gate 2 方向审查中增加快速校验——检查是否与已有 ADR/spec 矛盾
  - **docs 类文档变更**（.self-workflow/docs/）：格式/规范类问题为主，Compound 审查足够
  - **configs 类文档变更**（.self-workflow/configs/）：如果修改的是 feat-workflow.md 等核心流程文件，已纳入 Gate 2/3 的设计/实现审查

- 方案 A vs C 的重新评估：需求分析约束已明确"文档审查不受 Gate weight skip/light 影响"，因此方案 A 的"weight=skip 时文档审查被跳过"缺点可通过同一条约束规避——方案 A 也可以设计为 Gate skip 时不跳过文档检测。但方案 C 仍更优：它保持 Gate 流程完全不受文档审查干扰，单一职责更清晰。

## 关联

- 关联需求：F1a, F1b, F1c
- 关联约束：文档审查不受 Gate weight 影响
