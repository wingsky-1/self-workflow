## 得与失

### 做得好的

1. **小切口、精准改动**：P2 版本的约束（"改小而精准"）被严格遵守。所有改动集中在 `feat-workflow.md`，不触及命令流程、不创建新 Agent、不修改内置 Review Agent 配置。
2. **对抗性审查"吃自己的药"**：Gate 4 的对抗性审查发现了 2 个 critical 问题（task.yaml 结构损坏、F3b 未实现）——恰好证明对抗性审查的价值。新实现的 Grill 提示词在这个工作流中尚未生效（它是修改部署副本之后才存在的），但 Review Agent 自身的审查质量已经足够发现这些问题。
3. **设计文档质量**：经过 Gate 2 两轮对抗性审查，设计质量显著提升——标记机制从"冗余 task.yaml 字段"简化为"确认 commit"，ADR-002 从方案 B 升级为方案 D（+COT），ADR-001 增加风险分级。
4. **经验驱动**：4 个功能点（F3/F4）直接来源于已有的错误经验和参考模式文档，证明了经验库的价值。

### 可改进的

1. **F3b 遗漏**：反转说明检查在初始设计中未被覆盖，Gate 4 审查时才被发现。根本原因：Phase 2 设计时只对照了"交叉一致性"（F3a），漏掉了 F3b 的"反转说明"子需求。改进建议：设计阶段应在"与 Phase 1 验收标准对照"的检查清单中逐条对照，而非仅对照标题。
2. **task.yaml 编辑失误**：Phase 2 checkpoint 编辑时覆盖了 `- id: 3`。根本原因：edit 工具的 oldString 匹配范围过大（跨越了 `- id: 2` 和 `- id: 3` 的边界）。改进建议：task.yaml 编辑时 oldString 应严格限定在当前 phase 范围内，不跨 phase 边界。
3. **Phase 4 验证仍偏形式化**：验证文档的"反向检查"声称旧内容已移除，但未提供 git diff 证据。这恰好是 V1.10 自己在解决的问题——但"吃了自己的药"还不够彻底。改进建议：验证报告中的每条断言应附带可复现的证据（grep 命令 + 行号）。
4. **运行时等价验证仅停留于提醒**：Phase 4 清单中的"运行时等价验证"是一个检查项——它提醒 Agent 要做运行时验证，但没有提供如何做的机制。这不是实现缺陷而是设计限制（V1.10 是流程改进，不涉及新工具开发）。

## 经验草稿

### 候选 doc 级经验（待 Human 确认是否沉淀）

1. **参考模式：Gate 审查提示词的 Grill + COT 设计模式**
   - 问题：确认式审查提示词产生假阴性（pass 但遗漏问题）
   - 方案：三件套——"以质疑的立场"前缀 + "攻击以下弱点"结构化清单 + "逐项输出推理链"COT 要求
   - 适用场景：任何使用 Review Agent 的 Gate 审查
   - 建议沉淀到 `.self-workflow/docs/参考模式/`

2. **错误经验：task.yaml 编辑时的跨 phase 边界问题**
   - 问题：edit 工具的 oldString 跨越 `- id: N` 边界时可能丢失列表标记
   - 根因：oldString 从 phase N 的 checkpoint 跨越到 phase N+1 的 name
   - 修复：oldString 严格限定在单个 phase 内
   - 建议沉淀到 `.self-workflow/docs/错误经验/`
