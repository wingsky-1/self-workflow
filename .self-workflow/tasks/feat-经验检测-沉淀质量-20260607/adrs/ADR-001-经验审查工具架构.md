# ADR-001：经验审查以 Skill 为入口，不创建独立 Command

## 背景

V1.16+V1.17 需要实现经验检测和沉淀质量审查。用户的设计方向 D1 是："一致性审查、去重和刷新要做成skill或者command的形式，不仅是Agent在归档时要考虑，用户也可以手动触发"。

初始方案考虑过 Command + Skill 双入口设计（原 ADR-001），但在 Gate 1 审查后被推翻——skill 本身就可以被用户调用，双入口是过度设计。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 仅 Command | `/exp-review` 命令 | 用户直接输入 `/exp-review`，符合已有 command 模式 | Agent 需解析 command 输出文本，脆弱；Agent Phase 5 集成不自然 |
| B: 仅 Skill | `exp-governance` skill | Agent 在 Phase 5 加载执行；用户可通过"审查经验"等自然语言触发 skill；只有一个入口 | skill 不为用户提供显式的 `/命令` 提示，但用户指令自然会触发 |
| C: Command + Skill | 双入口（原方案） | 覆盖两种场景 | 过度设计：skill 已覆盖两个场景；两个入口独立维护；注册机制不同 |

## 选择

方案 B：仅 Skill `exp-governance`

## 理由

1. **Skill 同样可被用户调用**：用户说"审查一下经验文档"或"用 exp-governance skill 检查"即可触发，与 `/exp-review` 等效但更自然。
2. **减少维护成本**：单一入口，单一注册路径（skill 目录），不与 command 重复。
3. **为 V2.0 铺路**：skill 可直接演进为阶段子 Agent 的加载项，command 则不能。
4. **项目尚无 skill template**：创建首个 skill 模板本身就是本次成果的一部分。

## 影响

- 仅需创建 `packages/installer/templates/skills/exp-governance/` 目录结构
- MANIFEST 注册 skill 模板
- 不创建 command 文件，不注册 command
- `feat-workflow.md` Phase 5 引用 `exp-governance` skill
- 用户触发方式：自然语言指令 → Agent 识别意图后 `skill(name="exp-governance")`

## 反对意见

- 方案 A：用户输入 `/exp-review` 有明确的命令提示，skill 无此提示。→ 缓解：在 skill 的 description 中注明触发关键词，Agent 会识别
- 方案 C：两个入口分离关注点，用户侧和 Agent 侧的审查标准可能不同。→ 反驳：当前需求无此区分，规格统一即可

## 关联

- 关联需求：V1.16 F1（去重检测）、F2（一致性审查）、V1.17 F3（质量评估）、F4（复利机制）
- 关联 spec：`specs/default/exp-governance.md`（审查规则定义）
