# ADR-002：经验审查合并为单一 Skill，审查维度统一输出

## 背景

V1.16+V1.17 涉及多个审查维度：去重检测、一致性审查、质量评估、复利机制。初始方案考虑过是否拆分为多个独立的审查单元（多 command/skill），被 Gate 1 审查否决。

ADR-001 已确定以单一 skill 为入口，本 ADR 确认 skill 内部的审查维度组织和输出结构。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 多 skill | `exp-dedup`、`exp-check`、`exp-quality` 三个独立 skill | 职责单一，每个可独立演进 | 用户需记住多个入口；维度间有数据关联（去重影响质量），需重复扫描 docs/ |
| B: 单一 skill + 分节输出 | `exp-governance` skill 包含全部审查维度，按分节输出（采用） | 一次扫描覆盖所有维度；skill description 注明支持参数细分 | 单一 skill 指令较长 |
| C: 单一 skill + 子 task | `exp-governance` 内部 explode 为多个子 task 并行执行 | 并行效率高 | V1 过度设计；skill 触发 task 嵌套增加复杂度；为 V2.0 留余地但 V1 不需要 |

## 选择

方案 B：单一 skill `exp-governance`，审查维度分节输出，支持用户通过自然语言指定审查范围（如"只检查去重"）

## 理由

1. **用户无需记忆多个入口**：用"审查经验"一种表达即可覆盖所有维度
2. **一次扫描共享上下文**：docs/ 目录扫描只需执行一次，各维度共享结果
3. **未来可拆**：若 skill 逻辑持续增长，可分化为多个 skill 共享一个扫描基类，但与 ADR-001 保持一致，V1 不拆
4. **与现有 /todo-organize 模式一致**：单一入口做多维度操作是已验证的模式

## 影响

- Skill `exp-governance` 的指令结构按审查维度分节：frontmatter 完整性 → tag 规范性 → 去重检测 → 质量评估 → 生命周期建议
- Skill 的 description 注明自然语言触发词和可选的审查范围限定

## 反对意见

- 方案 A：职责单一 → 反驳：当前审查维度紧密耦合（去重结果影响质量评分），强行拆分需额外协调
- 方案 C：并行效率 → 反驳：V1 的 docs/ 规模（~27 份文档），单次扫描足以，并行化收益微乎其微

## 关联

- 关联 ADR：ADR-001（确定 skill 为唯一入口）
- 关联需求：V1.16 F2（一致性审查）、V1.17 F3（质量评估）
