# ADR-003：去重策略——Agent 判断，不硬编码算法

## 背景

去重是经验复利的基础——相同经验反复沉淀造成负债。常见方案是预定义算法（编辑距离阈值、SimHash/MinHash），但用户明确指示："去重和合并是一个复杂的事情，先有Agent来判断，不定义算法"。

此决策影响 `exp-curation` skill 的指令设计和 spec 中的去重规则定义。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 算法驱动 | 预定义编辑距离(≤3)、SimHash(≥0.7)等阈值，程序化输出结果 | 确定性，可复现；结果不依赖 Agent 能力 | 阈值难以跨场景适用；标题不同但内容相似的文档漏检；Agent 判断被架空；参数调优成本高 |
| B: Agent 判断 | Skill 提供审查指南（判断维度+输出格式），Agent 读取文档后自行判断相似性，输出理由 | 灵活适应各种去重场景；Agent 可理解语义相似性；符合用户指示；无需维护算法参数 | 不同 Agent 可能产出不同结果；依赖 Agent 能力 |
| C: 混合 | 元数据初筛（标题/tag/category 近似）+ Agent 对候选做深度判断 | 兼顾效率和质量；减少 Agent 全量阅读开销 | 初筛仍依赖阈值定义；增加实现复杂度；当前 docs/ 规模（~27份）不需要 |

## 选择

方案 B：Agent 判断

## 理由

1. **用户明确要求**："先有Agent来判断，不定义算法"——方案 A 直接违反此要求
2. **当前规模适用**：docs/ 约 27 份文档，Agent 可全量阅读并在上下文中比较
3. **去重本质是语义问题**：两份文档可能标题完全不同但讲同一件事（如中文版和英文版），算法无法捕获语义等价；Agent 可以
4. **Skill 指令可迭代**：审查质量通过优化 SKILL.md 指令提升，无需改算法/调参数——符合"经验复利"理念
5. **方案 C 是 premature optimization**：混合方案在文档规模 100+ 时有价值，V1 不需要

## 影响

- `exp-curation` skill 指令中的去重部分提供判断维度（标题相似性、主题重叠、内容重复程度），要求 Agent 输出"疑似重复"判定 + 具体理由
- spec `exp-curation.md` 中去重规则使用 MUST 定义判断流程，SHOULD 定义判断维度，不定义算法阈值
- 如果 docs/ 规模增长到 100+，可重新评估引入方案 C 的初筛层

## 反对意见

- 方案 A：确定性 → 反驳：去重本质是语义问题，确定性阈值不如 Agent 理解准确；且用户明确不要算法
- 方案 C：效率 → 反驳：27 份文档 Agent 全量阅读在单次对话中完全可行，premature optimization
