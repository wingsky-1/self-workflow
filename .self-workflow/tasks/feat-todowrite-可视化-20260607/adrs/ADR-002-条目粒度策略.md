# ADR-002：Todowrite 条目粒度策略

## 背景

todowrite 条目的信息密度是核心设计问题。过粗（一条"做 Phase 1"）无意义，过细（每条工具调用）浪费 token。需要统一的粒度规则。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A | **阶段级**：每 Phase 一条 | 最少条目数 | 信息量极低，无帮助 |
| B | **操作级**：每次工具调用一条 | 最透明 | Token 浪费，噪音多 |
| C | **混合粒度**：Phase 入口 + Gate + 委托(>30s) | 平衡信息量与维护成本 | 有一定主观性 |
| D | **三级嵌套** | 结构化 | todowrite 不支持嵌套 |

## 决策

选择 **方案 C** — 混合粒度

## 理由

- 工具约束：todowrite 不支持嵌套（方案 D 不可行）
- 信息价值原则：条目标记"流程转折点"——阶段、Gate、委托——Human 最关心的里程碑
- 实践经验：V1.11 评审中"每步标记"被表扬，"批量标记"被批评

## 后果

- 正面：典型 /feat 工作流 ~15-25 条条目，密度适中
- 负面：边界判断有主观性，需 spec MUST/SHOULD 明确
