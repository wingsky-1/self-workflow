## 二、架构决策

### AD-1：F2 Review Agent 调用在指引中的声明方式

**背景**：当前 4 个 Gate 的对抗性审查步骤用自然语言描述"调用 Review Agent"，措辞为描述性而非指令性，Agent 容易跳过。需要在指引中以更强的约束方式表达。

**备选方案**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A：Inline task() 代码块 | 在 Markdown 中直接写入 `task(subagent_type="review-agent", prompt="...")` 代码块，Agent 看到后直接复制执行 | 确定性最高；Agent 无需理解语义即可执行；prompt 模板可复用 | 代码块过长会打断文档可读性；prompt 内容与上下文耦合 |
| B：强制指令语句 | 使用"必须执行"、"不可跳过"等措辞强化指令性，但保留自然语言描述 | 可读性好；Agent 需要理解语义后自行构造 prompt | 不够确定性——Agent 仍可能跳过或构造不完善的 prompt |
| C：混合方案 | 每个 Gate 的"对抗性审查"子步骤中，用自然语言描述审查要求，后附一个 `task()` 代码块示例 | 兼顾可读性和确定性 | 文档长度增加 |

**选择**：方案 C（混合方案）

**理由**：
1. 自然语言描述保留审查要点的上下文，方便 Agent 理解审查意图
2. `task()` 代码块提供"抄作业"级的具体指令，Agent 无法以"不知道怎么写 prompt"为借口跳过
3. prompt 中的占位符（如 `<workflow-id>`, `<artifact-path>`）可被 Agent 替换为实际值，模板本身不锁定上下文
4. 相比纯方案 A（只给代码块），保留自然语言描述让降级路径（Review Agent 不可用时主 Agent 自审）仍然有据可依

### AD-2：F4 /adr 命令的去类型参数策略

**背景**：`/adr` 当前要求用户指定 `simple|complex|review` 类型参数，与 V1.5 验收标准 v1.2 重新定位的"Agent 自主归档为主，/adr 为兜底"不一致。Agent 已经在设计文档中写好了 ADR 内容，不需要交互式选择模板。

**备选方案**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A：command 参数 `--auto` | `/adr --auto <标题>` — Agent 将完整 ADR 内容作为上下文传递给命令，命令跳过模板选择和交互式收集，直接写入 | 语义清晰；Agent 显式声明"我已准备好内容" | 依赖 OpenCode Command 是否支持自定义 flag 参数；Agent 需理解 --auto 的触发条件 |
| B：指引约定 | 不在命令层面做改变，而是在 feat-workflow.md 和 agent-reasoning Skill 中约定"当 Agent 已有完整 ADR 内容时，直接用 write 工具写入文件" | 不改变命令定义；无参数解析依赖 | Agent 可能仍然调用 /adr 走交互式流程；不够统一 |
| C：命令内部智能判断 | `/adr` 检测当前上下文是否有已生成的 ADR 内容，有则跳过交互，无则走交互式收集 | 用户/Agent 无需关心模式选择 | 检测逻辑不可靠——"上下文中有 ADR 内容"难以自动判断 |

**选择**：默认 auto + 降级交互

> ⚠️ **设计变更（经 Human 审查后修正）**：原设计为 `--auto` 显式 flag，改为默认 auto 模式。
> **理由**：V1.5 验收标准 v1.2 定位"Agent 自主归档为主路径"，`/adr` 为兜底。若 auto 是关注路径而非例外路径，应该默认 auto，失败时降级交互——而非要求 Agent 记住加 `--auto`。
> 命令签名简化为 `/adr <标题>`，无 mode flag。Agent 在上下文中准备内容，内容不足时自动降级为 question 交互。

### AD-3：T3 经验分类的文件组织方式

**背景**：`.self-workflow/docs/` 当前仅有 `*-实施经验.md` 一种模式。需建立三种分类（实施经验、参考模式、错误经验）的组织方式，并定义 Agent 何时加载。

**备选方案**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A：文件名约定 | 文件命名：`<领域>-实施经验.md`、`<领域>-参考模式.md`、`<领域>-错误经验.md` | 简单直观；Agent 扫描文件名即可判断分类；无需解析文件内容 | 文件名可能变长；一个文件内混入多种经验时分类不准 |
| B：frontmatter 字段 | 文件内 YAML frontmatter 定义 `category: 实施经验` | 一个文件可包含多个分类（分节）；元数据与内容在同一文件 | 需解析文件内容才能获取分类；Agent 扫描效率低 |
| C：目录分类 | `.self-workflow/docs/实施经验/`、`.self-workflow/docs/参考模式/`、`.self-workflow/docs/错误经验/` 三个子目录 | 文件系统级分类；Agent 扫描目录即可 | 目录层级增加；一个经验可能属于多个分类时难以放置 |

**选择**：方案 A（文件名约定）

**理由**：
1. Agent 扫描文件名是成本最低的操作——无需读取文件内容
2. 渐进式披露的核心是"在正确时机加载正确的文档"，文件名约定使匹配关键词变得简单
3. 如果未来需要一个文件包含多种分类，可以创建复合文件名（如 `<领域>-实施经验+参考模式.md`）或拆分为多文件
4. 方案 C 的目录分类在交叉分类场景下反而增加复杂度

**渐进式披露的 Agent 加载时机**（设计为非强制性指引——Agent 根据场景自行判断）：

| 分类 | 文件名模式 | Agent 加载时机 | 触发条件示例 |
|------|-----------|--------------|------------|
| 实施经验 | `*-实施经验.md` | 启动同类任务时 | 任务描述关键词匹配文件名中的领域 |
| 参考模式 | `*-参考模式.md` | 设计阶段入口 | Agent 自问："之前解决过类似架构问题吗？" |
| 错误经验 | `*-错误经验.md` | 任务启动前（预防） + 遇到错误时（诊断） | Agent 遇到错误后自问："这个模式之前遇到过吗？" |

### 3.2 /adr 命令重构（M1）

**修改文件**：
- `.opencode/commands/adr.md`
- `packages/installer/templates/commands/adr.md`（安装器同步）

**变更概要**：

| 变更项 | 当前 | 修改后 |
|--------|------|--------|
| 用法 | `/adr <simple\|complex\|review> <标题>` | `/adr [--auto] <标题>` |
| 参数：模板类型 | 必选，3 选 1 | 移除，由 Agent 自行判断 |
| 新增参数 | — | `--auto`：Agent 已准备好 ADR 内容时跳过交互 |
| 步骤 3：选择模板 | 根据用户指定的类型读取对应模板 | 改为：Agent 根据决策复杂度自行选择模板（simple/complex/review），模板文件不变 |
| 步骤 4：填写模板 | 通过 question 工具向用户收集 | 普通模式：question 收集；--auto 模式：跳过，直接使用 Agent 提供的完整内容写入 |
| ADR 编号 | per-task（不变） | per-task（不变，但补充说明：编号仅在当前 task 的 adrs/ 内递增） |

**--auto 模式的流程差异**：

```
普通模式（人类使用）：
  /adr "某个决策标题"
  → 识别 task → 确定编号 → Agent判断模板 → question收集内容 → 写入 → 更新task.yaml

--auto 模式（Agent 使用）：
  /adr --auto "ADR-003-某个架构决策"
  → 识别 task → 确定编号 → 跳过交互 → Agent提供完整内容 → 写入 → 更新task.yaml
```
