# ADR-003：元数据模板填充策略

| 字段 | 值 |
|------|-----|
| ID | ADR-003 |
| 状态 | 已选择 |
| 决策者 | Agent (Sisyphus) |
| 日期 | 2026-06-06 |

## 背景

`/feat` 步骤 4 需要从 `workflow-metadata-template.yaml` 读取模板，填充字段后写入 `workflow.yaml`。

## 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A — 模板驱动填充 | 读模板 → 改字段 → 写新文件 | 模板变更自动跟随 | 需要 YAML 解析能力 |
| B — 硬编码完整内容 | 命令中直接写入完整内容 | 简单、快速 | 模板变更时命令不同步 |

## 选择

方案 A

## 理由

1. `workflow.yaml` 的 phases 结构可能在未来版本变更（如新增 phase 6）
2. YAML 解析在 OpenCode 环境中可用
3. 需填充的字段有限且明确：`workflow-id`, `type`, `status`, `created`, `updated`, `description`

## 填充规则

| 模板字段 | 操作 | 值来源 |
|---------|------|--------|
| `workflow-id` | 替换 | 步骤 1 生成 |
| `type` | 设置 | `feat`（固定） |
| `status` | 设置 | `in_progress` |
| `created` | 替换 | 当前时间 ISO 8601 |
| `updated` | 替换 | 同 created |
| `description` | 替换 | 用户原始描述 |
| `phases[*]` | 保留原值 | 模板定义的标准结构 |

## 后果

- 正面：模板是单一真相源
- 负面：依赖 YAML 解析。缓解：已验证可用

## 关联

- 关联需求：F3（元数据创建）
- 关联模板：`.self-workflow/configs/templates/workflow-metadata-template.yaml`
