# ADR-003：关键决策即时归档机制

> ⚠️ **勘误**：本文中提到的 ADR 存储路径 `.self-workflow/decisions/` 已被 ADR-004 和 ADR-006 更新。实际存放位置为 `.self-workflow/tasks/<task-name>/adrs/`。以下 3 处标注已对应修正。

## 背景

在开发过程中，用户会做出各种关键决策（如技术选型、架构调整、原则确立）。如果没有及时记录，这些决策会被后续对话冲淡，导致：

1. 同一问题被反复讨论
2. 决策上下文丢失，后续难以理解"为什么这么设计"
3. 新加入的参与者（或新会话）不知道之前的决策

之前已经有了 ADR-001（交互式问答优先）和 ADR-002（安装器职责边界），但"需要记录决策"这件事本身还没有被明确为流程。

## 决策

**当用户做出关键决策时，必须及时整理归档为 ADR，并参考已有的经验和文档。**

### 具体规则

1. **识别触发**：当用户明确说"记录这个决策"、"记住这个原则"、"这是一个关键决定"等话术时，立即触发归档流程
2. **主动识别**：当用户做出以下类型的决策时，Agent 应主动提议归档：
   - 架构性决策（组件边界、职责划分、技术选型）
   - 原则性决策（设计理念、约束条件、质量标准）
   - 流程性决策（开发方式、协作模式、交互方式）
   - 反模式纠正（指出 Agent 做错了什么，要求改变）
3. **归档格式**：使用 `.self-workflow/tasks/<task-name>/adrs/<id>-<标题>.md` 的 ADR 格式（初稿为 `decisions/`，ADR-004/006 更新为此路径）
4. **交叉引用**：每条 ADR 必须检查并引用：
   - 已有的 ADR 决策（当前 task 的 `adrs/` 目录）
   - 需求草案中的设计原则（§1.3 和 §2.1）
   - 相关的 ROADMAP 里程碑或执行计划
5. **用户确认**：归档后提交给用户确认（使用 question 工具）

### ADR 模板

每次归档遵循以下结构：

```yaml
---
id: <顺序号>
title: <决策标题>
date: <YYYY-MM-DD>
status: 已采纳 | 提议中 | 已废弃
---

# ADR-<id>：<标题>

## 背景

<为什么需要这个决策>

## 决策

<具体的决策内容>

## 理由

<为什么选择这个方案>

## 影响

<这个决策带来的后果>

## 关联

- 关联 ADR：<链接>
- 关联需求草案：<章节>
- 关联 ROADMAP：<版本/里程碑>
```

### 与现有 ADR 的交叉引用检查清单

归档新 ADR 时，逐项检查：

- [ ] 有没有已有的 ADR 覆盖了类似主题？
- [ ] 这个决策与已有 ADR 是否一致？如果有冲突，需要注明原因
- [ ] 需求草案的设计原则是否涉及此决策？
- [ ] ROADMAP 中的版本规划是否受此决策影响？

## 理由

- **知识持久化**：决策不依赖 Agent 的对话记忆，而是写入文件系统
- **可追溯**：后续可以通过 ADR 理解设计演进的脉络
- **自举验证**：如果 Self-Workflow 的工作流能产出 ADR，说明工作流本身是可用的
- **降低重复讨论**：决策写入文件后，不需要每次重新确认

## 影响

- `.self-workflow/tasks/<task-name>/adrs/` 目录会持续积累（初稿为 `decisions/`，已更正），需要定期回顾和清理废弃决策
- Agent 在做出建议前应先查阅已有 ADR（"参考已有经验"）
- ADR 的编号顺序反映决策的时间线
