# ADR-001：交互式问答优先

## 背景

在开发 Self-Workflow 的过程中，Agent 在向用户提出选项时使用纯文本方式（在对话中列出选项），而不是使用交互式问答工具（`question` API）。

这导致了两个问题：
1. 用户需要手动输入选项文字，增加了交互成本
2. 用户体验不一致——本该一键选择的地方变成了打字

## 决策

**无论当前会话中还是正在开发的 Self-Workflow 项目，所有带选项的提问优先使用交互式问答工具。**

### 具体规则

1. **选项选择**：涉及 2 个以上选项供用户选择时，使用 `question` 工具，而非纯文本列举
2. **确认/继续**："是否继续？"类提问使用 `question` 工具
3. **偏好选择**：让用户在多个方案中做选择时使用 `question` 工具
4. **排除项**：开放式提问（"还有什么？"）、需要自由输入的场景不使用 `question`
5. **总结先行**：询问用户是否继续下一步前，先总结已完成的内容，再说明下一步要做什么。格式：`已完成：X、Y、Z → 下一步：做 A（涉及 B、C、D）`
6. **选项描述具体**：`question` 的 `description` 字段说明选择带来的后果，而非笼统描述

### 工具选择

```markdown
✅ 正确做法：
<question questions=[{options: [{label: "方案 A", description: "..."}, ...]}]>

❌ 错误做法：
"您想选择方案 A 还是方案 B？"
```

## 理由

- **降低交互成本**：用户只需点击而非打字
- **减少歧义**：选项固定，无需自然语言解析
- **提升体验**：更接近 GUI 应用的交互流畅度
- **适用自举**：Self-Workflow 项目自身的交互也遵循此原则

## 影响

- Agent 交互方式需调整：有选项必用 `question`
- 工作量：无额外开发成本，仅改变交互习惯
