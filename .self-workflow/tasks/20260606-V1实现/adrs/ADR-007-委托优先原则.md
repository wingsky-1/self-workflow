# ADR-007：委托优先原则

## 背景

AI Agent 的上下文窗口是有限的（通常 32K~200K token）。当主 Agent 同时承担"决策者"和"执行者"两个角色时，实施细节（代码 diff、grep 输出、文件内容）会迅速填满上下文，挤占用于全局判断和流程把控的空间。

M0 审计已验证 OpenCode 支持通过 `task()` 创建子 Agent。但实践中发现：
- 简单操作（读已知文件、修 typo）直接做更快，不需要委托
- 复杂任务（跨模块分析、多文件实现）全部自己扛，上下文爆炸
- 缺少明确的委托粒度判断标准

## 决策

**主 Agent 的上下文只保留决策链路，不保留执行细节。**

但委托不等于盲从。在执行任何方向之前，主 Agent 必须先质疑方向的合理性。

### 总则：先质疑，再委托，后验证

```
Human 提出方向
    ↓
质疑阶段：Agent 独立思考
├─ 这个方向合理吗？（有没有明显的问题？）
├─ 有更好的方案吗？（备选方案评估）
├─ Human 考虑过 X 吗？（补全遗漏的约束）
    ↓
对齐阶段：与 Human 讨论
├─ 质疑成立 → Human 调整方向
├─ 质疑不成立 → 确认方向，继续执行
    ↓
执行阶段：按委托规则实施
    ↓
验证阶段：结果复核
```

**质疑不是对抗**。Agent 的角色是补全 Human 可能遗漏的视角——Human 有经验判断，Agent 有计算广度。两者互补才能做出更好的决策。

### 具体规则

#### 什么必须自己做（不需要委托）

```
- 读已知路径的文件（read）
- 简单 grep 搜索（grep/glob）
- 单文件 typo 修复
- 查看 git 状态
- 对其他 Agent 产出的直接验证
```

这些操作开销小、不需要额外上下文、委托反而增加通信成本。

#### 什么应该委托

```
✅ 并行独立任务 → background agents
   - 例：同时搜索 3 个模块的实现模式
   - 例：同时调研 2 个不同库的 API

✅ 专业领域任务 → specialist agents
   - 代码库深入探索 → explore
   - 外部文档调研 → librarian
   - 复杂架构分析 → oracle

✅ 上下文隔离任务 → fresh sub-agent
   - 需要在独立上下文中进行的分析
   - 出错不应污染主上下文的操作
```

#### 委托的粒度判断

以"通信成本 < 执行收益"为准：

```
适合委托 ↔ 不适合委托
    │
    ├─ 执行时间 > 30 秒 → ✅ 委托（并行收益大）
    ├─ 执行时间 5-30 秒 → ⚠️ 视复杂度
    ├─ 执行时间 < 5 秒 → ❌ 自己做
    │
    ├─ 需要 3+ 步才能完成 → ✅ 委托
    ├─ 1 步就能完成 → ❌ 自己做
    │
    ├─ 输出 > 50 行会冲击上下文 → ✅ 委托
    └─ 输出 < 10 行 → ❌ 自己做
```

#### 委托后的验证

委托不是"丢出去不管"。主 Agent 必须验证结果：

```
委托 → 子 Agent 执行 → 子 Agent 返回结果
    ↓
主 Agent 验证：
├─ 结果符合预期？→ 继续
├─ 结果有问题？→ 重新委托或自行修复
└─ 结果需要整合/转化？→ 主 Agent 做 final touch
```

### 与现有机制的关系

当前 Sisyphus 系统的 `task()` API 已支持：
- `run_in_background=true` → 并行委托
- `subagent_type` → specialist 选择（explore/librarian/oracle 等）
- `category` → 领域优化（quick/deep/ultrabrain 等）

本 ADR 不是引入新机制，而是**明确委托的判断标准**。

## 理由

- **上下文预算**：主 Agent 的上下文是最稀缺资源，应该留给决策而非实施
- **并行收益**：独立任务并行执行，总吞吐量倍增
- **错误隔离**：子 Agent 的失败不影响主流程上下文
- **互补决策**：Human 有经验判断力，Agent 有计算广度。Agent 不质疑 = Human 在单方面决策，两者的优势没有互补
- **自洽**：Self-Workflow 本身的开发也应遵循此原则——本 ADR 就是一次"主 Agent 记录决策，而非自己分析所有方案"

## 影响

- 需要为主 Agent 建立"委托习惯"：遇到复杂任务先想"能不能拆成并行子任务"
- 委托不是免费的——每次委托都有通信开销，粒度太细反而亏
- 委托结果需要验证——信任但核验
- 后续可提炼到需求草案 §2 设计哲学
