# ADR-002：安装器职责边界

## 背景

在实现 M1.2 安装器时，初始版本将工作流指引、Review Agent 定义等业务内容以 JavaScript 模板字符串（template literal）的形式内嵌在 `packages/installer/index.js` 中。导致：

1. `index.js` 膨胀到 596 行，其中大部分是业务内容而非安装逻辑
2. 修改工作流指引需要修改安装器代码
3. 职责混合——安装器同时承担了"内容维护"和"安装执行"两个职责

## 决策

**安装器只负责安装，不内联任何业务内容。**

### 具体规则

1. **安装器的职责**：创建目录结构、从模板目录复制文件、输出安装报告
2. **模板的职责**：`templates/` 目录独立维护工作流指引、Agent 定义等内容
3. **维护方式**：修改工作流内容只需编辑 `templates/` 下的文件，不碰安装器代码
4. **清单驱动**：安装器通过 manifest 声明哪些文件需要复制，而不是硬编码内容

### 实施方式

```
packages/installer/
├── index.js               ← 安装器逻辑（纯编排，当前 154 行）
└── templates/             ← 模板内容（独立维护）
    ├── guides/            ← 工作流指引
    └── agents/            ← Agent 定义
```

安装器通过 `MANIFEST` 数组声明文件映射关系：

```javascript
const MANIFEST = [
  [".self-workflow/guides/feat-workflow.md",  "guides/feat-workflow.md"],
  [".opencode/agents/review-agent.md",        "agents/review-agent.md"],
];
```

## 理由

- **关注点分离**：安装逻辑和业务内容不应耦合
- **可维护性**：修改工作流指引不需要理解或修改 JS 代码
- **可扩展性**：新增模板只需在 manifest 添加一行，不增加安装器复杂度
- **适用自举**：工作流内容本身也应遵循工作流来修改——这与将内容嵌入 JS 的做法矛盾

## 影响

- 安装器代码量从 596 行降至 154 行
- `templates/` 目录作为内容源，与 `.self-workflow/` 中已安装的内容对应
- 未来 V2 的 Adapter（YAML → Markdown）产出直接写入 `templates/` 目录
- **补充约束**：`.opencode/` 和 `.self-workflow/` 下的文件应通过安装器导入，不手动修改。安装器是这些目录的唯一入口
