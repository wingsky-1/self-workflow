# ADR-004：目录职责划分——开发与使用分离

## 背景

在 V1 实现过程中，三个核心目录的职责逐渐模糊：

| 目录 | 实际混入的内容 | 问题 |
|------|--------------|------|
| `.self-workflow/` | 开发文档（ADRs、审计报告）+ 运行时（指引、模板、产物） | 同一个目录同时承载开发和使用两种职责 |
| `.opencode/` | 运行时 Agent 定义 + 开发依赖（node_modules） | 平台集成层混入开发依赖 |
| `packages/installer/templates/` | 部分运行时模板，但不完整 | 产物模板缺失，安装器不是完整的来源 |

开发活动（写代码、做决策、审计能力）和使用活动（执行工作流、记录产物、沉淀经验）在文件系统中没有物理隔离，导致：
1. 修改时不确定应该改哪个目录
2. 安装器生成的"编译产物"和"源文件"无法区分
3. 自举开发场景下（用 Self-Workflow 开发 Self-Workflow），分不清哪些是由安装器管理的、哪些是可以自由编辑的

## 决策

**将整个项目结构按三层划分，严格分离开发与使用。**

### 三层架构

```
self-workflow/
│
├── [第一层] 开发层（Domain A：开发 Self-Workflow 自身）
│   ├── docs/           ← 参考性文档（ADR、审计、需求、路线图）
│   ├── tasks/          ← 行动性文档（执行计划）
│   └── packages/       ← 源码（安装器、Adapter 等）
│
├── [第二层] 平台集成层（Domain B：与 OpenCode 的接口）
│   └── .opencode/      ← agent / skill / command / hook / plugin
│                         （严格由安装器管理，不直接编辑）
│
└── [第三层] 工作流运行时层（Domain B：工作流执行的数据）
    └── .self-workflow/
        ├── configs/    ← 工作流配置（指引、模板）— 安装器管理
        ├── tasks/      ← 运行时任务记录（产物、错误）— V1可编辑，V2工作流管理
        ├── docs/       ← 经验沉淀（分级分类）— V1可编辑，V2工作流管理
        └── specs/      ← 规范约束（分级分类）— V1可编辑，V2工作流管理
```

### 各目录详细职责

#### 第一层：开发层（项目根目录）

| 目录 | 内容 | 编辑策略 |
|------|------|---------|
| `docs/` | 需求草案、路线图、ADR 决策记录、审计报告 | 直接编辑 |
| `tasks/` | 执行计划、Sprint backlog | 直接编辑 |
| `packages/` | 安装器源码、Adapter 源码 | 直接编辑 |

**`.gitignore` 策略**：全部纳入版本管理。

#### 第二层：平台集成层（`.opencode/`）

| 内容 | 来源 | 编辑策略 |
|------|------|---------|
| `agents/review-agent.md` | `packages/installer/templates/agents/` | ❌ 仅安装器 |
| `command/feat.md`（未来） | `packages/installer/templates/commands/` | ❌ 仅安装器 |
| `skills/`（未来） | `packages/installer/templates/skills/` | ❌ 仅安装器 |
| `node_modules/` | npm install | ❌ 仅包管理 |

**核心原则**：`.opencode/` 下的任何文件都不应直接创建或修改，必须通过安装器或包管理工具。

#### 第三层：工作流运行时层（`.self-workflow/`）

| 子目录 | 内容 | V1 策略 | V2 策略 |
|--------|------|---------|---------|
| `configs/` | 工作流指引(guides/)、产物模板(templates/) | 仅安装器 | 仅安装器 |
| `tasks/` | 工作流产物(artifacts/)、错误日志(errors/) | 自主编辑 | 工作流管理 |
| `docs/` | 经验沉淀（分级分类） | 自主编辑 | 工作流管理 |
| `specs/` | 规范约束（MUST/SHOULD/MAY） | 自主编辑 | 工作流管理 |

**V1→V2 的演进**：
- V1 阶段：工作流尚未完全自举，`tasks/` `docs/` `specs/` 允许自主编辑
- V2 阶段：自举完成后，所有 `.self-workflow/` 的操作都通过工作流进行

#### 安装器的角色

`packages/installer/templates/` 是第一层到第二、第三层的桥梁：

```
packages/installer/templates/
├── configs/guides/feat-workflow.md    → 安装到 .self-workflow/configs/guides/
├── configs/templates/*.md             → 安装到 .self-workflow/configs/templates/
├── agents/review-agent.md             → 安装到 .opencode/agents/
└── commands/（未来）                   → 安装到 .opencode/commands/
```

**修改流程**：
1. 编辑 `packages/installer/templates/` 中的源文件
2. 运行 `node packages/installer/index.js init`
3. 安装器将更新复制到 `.self-workflow/configs/` 和 `.opencode/`
4. 开发文档（ADRs、审计）编辑 `docs/` 下的文件

## 理由

- **开发 vs 使用物理隔离**：不会混淆"开发 Self-Workflow"和"使用 Self-Workflow"
- **Source of Truth 明确**：`packages/installer/templates/` 是运行时配置的唯一来源
- **自举路径清晰**：V1 允许编辑运行时内容 → V2 全部由工作流管理，过渡路径明确
- **符合渐进式原则**：与需求草案 §1.3 "渐进式采用"保持一致

## 影响

- `docs/adr/` 取代 `.self-workflow/decisions/`（ADR 是开发文档，不是运行时数据）
- `docs/` 和 `tasks/` 从 `.self-workflow/` 中移出到根目录
- V1-执行计划.md 从根目录移到 `tasks/`
- 安装器 MANIFEST 需要更新以匹配新路径
- feat-workflow.md 中的内部引用需要更新（`templates/` → `configs/templates/`）
