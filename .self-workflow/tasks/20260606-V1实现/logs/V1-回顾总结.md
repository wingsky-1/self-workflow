# V1 合作开发回顾总结

> 周期：2026-06-06（单日完成 V0→V1.5）
> 合作方：用户（决策者/架构师） + Sisyphus（执行者/建议者）

---

## 一、开发历程回顾

### 整体节奏

```
需求草案（已有） → ROADMAP 评审 → V1 执行计划
  → M0 审计 → M1.1 指引 → M1.2 安装器 → M1.3 Review Agent
  → M1.4 归档 → 架构重构（三次迭代） → M1.5 Catchup
```

实际执行顺序与 plan.md 基本一致，但中间穿插了**三次目录结构重构**：

| 轮次 | 结构 | 触发 |
|------|------|------|
| 初版 | `.self-workflow/` 混装开发+运行时 | — |
| ADR-004 | 三层分离（docs/tasks/packages + .opencode + .self-workflow） | 用户纠正 |
| 修正 | `tasks/` 从根目录移到 `.self-workflow/` 下 | 用户纠正 |
| 定型 | 入口文件 + adrs/ + logs/ 分层 | 用户提出 |

### 关键转折点

1. **安装器内联内容 → 模板分离**：用户指出安装器不应承担业务内容职责
2. **目录结构三次迭代**：从混装到分层，每次都是用户推动
3. **ADR 归档位置迁移**：从 `.self-workflow/decisions/` → `docs/adr/` → task 目录
4. **交互方式纠正**：用户要求用 `question` 工具而非纯文本

## 二、角色评价

> 以下评价基于角色分工：**Human（人类决策者）**——把控方向、做架构判断；**Agent（AI 执行者）**——落地实施、发现问题。

### Human（人类决策者）表现

| 维度 | 评价 | 实例 |
|------|------|------|
| **架构判断** | 准确 | 指出安装器职责边界、三层分离、task 在 `.self-workflow/` 下 |
| **原则坚持** | 一致 | "开发和使用分离"、"Agent 优先"、"安装器唯一入口" |
| **反馈及时** | 关键 | 每次 Agent 走偏立即纠正，避免累积技术债务 |
| **决策记录意识** | 强 | 推动 ADR-003（关键决策归档）、提醒记录未记录的决策 |
| **交互体验要求** | 高 | 推动 question 工具使用、总结先行的交互规则 |

### Agent（AI 执行者）表现

| 维度 | 评价 | 实例 |
|------|------|------|
| **执行力** | 强 | 按计划逐项完成 M0→M1.5，产出 7 个 ADR、完整目录结构 |
| **文档能力** | 好 | 工作流指引 730→314 行精简、ADR 结构清晰 |
| **纠错响应** | 及时 | 用户指出的问题立即修正，不辩驳 |
| **主动判断** | 弱 ❌ | 目录结构混装、安装器内联内容——都是用户发现的问题 |
| **交互规范** | 需改进 ❌ | 多次用纯文本列举选项，被用户纠正后才用 question 工具 |
| **决策敏感度** | 初期弱 ❌ | 多次被提醒"记录这个决策"后才建立习惯 |

**核心缺陷**：Agent 容易在"看起来合理但架构上错误"的方向上走远，需要 Human 拉回来。这已经通过 ADR-007 补充"先质疑再委托"原则来缓解——Agent 应在执行前先质疑方向合理性，不盲从。

## 三、工作流优化点（推广使用视角）

> 以下分析从**新用户首次使用 Self-Workflow** 的角度出发：一个新团队拿到这套工作流，会遇到什么卡点？哪些地方会让他们放弃？

### 🔴 新人卡点（不改会劝退）

#### 1. 线性阶段模型在复杂任务中会"卡死"

**场景**：新人启动一个 `/feat`，做到阶段 3（代码实现）发现阶段 1 的需求理解有误。按照当前流程，他必须"Gate 不通过 → 返回阶段 1 修正 → 重新经过 Gate 2 → 回到阶段 3"——路径极长，挫败感强。

**本质**：`阶段 1 → 2 → 3 → 4 → 5` 是理想路径，但真实开发是**发散的、回溯的**。工作流没有为"走回头路"提供低成本路径。

**推广影响**：新人遇到第一次回溯就会觉得"这工具不适合我的工作方式"，然后弃用。

**建议**：
- 引入**Checkpoint 机制**：允许标记某个阶段为"待 revisite"，继续前进，后续再回来修正
- 或引入**轻量回溯路径**：从阶段 N 返回阶段 M（M < N）时，跳过中间已通过的 Gate，只验证受影响的 Gate

#### 2. 缺少"这个方向对不对"的判断环节

**场景**：新人做了一个架构决策（比如"把安装器模板内联到 JS 中"），Review Agent 审查了代码质量、测试覆盖——都通过了。**但没人质疑"这个方向本身对不对"**。直到 Human 介入才发现方向错了，重构成本很高。

**本质**：当前 Gate 审查看似严格，但只在**同一个方向内**检查质量，不检查**方向本身是否正确**。

**推广影响**：新人会反复做"方向对了但实现有 bug"的事，这在传统开发中很正常（有 bug 就修）。但如果做的是"方向错了但实现完美"的事，重构成本会让新人觉得"这工作流反而拖慢了我"。

**建议**：
- 在方案设计 Gate 中增加**方向判断节点**：Review Agent 必须回答"这个方案与项目现有架构一致吗？有没有更简单的替代方案？"
- 引入**架构 ADR 前置审查**：重大决策必须先写 ADR，Review Agent 审查通过后才能进入实现

#### 3. 决策记录是"额外工作"，不是"自然产出"

**场景**：新人完成一个 `/feat` 后，被告知"请记录 ADR"。潜台词是："你刚才的工作还不够，还要额外写文档。"——这在任何工具中都是最招人反感的设计。

**本质**：ADR 应该是实施过程中**自然产生的副产品**，而不是完成后的额外步骤。当前流程把 ADR 放在"总结沉淀"阶段，但大部分值得记录的决策是在实现过程中做出的。

**推广影响**：新人会觉得"这工具文档负担太重"——这是大多数工程工具推广失败的首要原因。

**建议**：
- 将决策捕捉**嵌入**阶段流程：每个阶段末尾自动询问"这个阶段有没有值得记录的决策？"
- 引入**轻量决策记录**：一句话 + 关联文件即可，不强制 ADR 格式。后续可以升级为完整 ADR

### 🟡 易用性优化（做了体验更好）

#### 4. 新人不知道当前进度在哪

**场景**：新人执行到一半关闭 session，第二天回来输入 `/catchup`，得到的回复是"进行中的任务：XXX"。但对于一个刚接触工作流的新人，他需要知道的不只是"在做什么"，还有"做到哪一步了、接下来要做什么"。

**本质**：`/catchup` 当前只扫描 task.yaml 的结构化状态，不包含"下一步行动建议"。

**建议**：catchup 输出增加**下一步建议**——基于当前阶段和已完成内容，给出具体的"接下来你应该做 X"。

#### 5. 安装后缺少"快速上手"引导

**场景**：新人运行 `self-workflow init` 后，看到了安装报告，但不知道下一步具体怎么做。虽然报告提示了"输入 `/feat` 启动工作流"，但 `/feat` 什么？怎么描述任务才符合工作流的要求？

**本质**：安装器只负责"装"，不负责"教"。缺少一个快速上手的 walkthrough。

**建议**：安装器增加交互模式：`self-workflow init --demo`，引导用户完成一个 demo 任务。

#### 6. Gate 降级标准对新人过于模糊

**场景**：新人看到一个"实现审查 Gate (weight: full)"，被告知"如果变更很小可以自动降级"。但什么算"很小"？新人没有经验判断。

**本质**：当前 Gate 降级标准（≤3 文件、不涉及架构决策）过于依赖 Agent 的主观判断。新人无法预期 Gate 的行为。

**建议**：在 Gate 描述中直接给出降级判断的量化标准，让新人在进入阶段前就能预期 Gate 的严格程度。

### 🟢 内部效率优化（对推广影响小）

#### 7. 模板提取应一步到位

**教训**：feat-workflow.md 经历了 730→355→314 行的多次提取。这个迭代过程对新用户不可见，但影响了开发效率。

**建议**：在模板设计时一次性识别所有输出格式，统一提取。

#### 8. task.yaml 应在 task 创建时自动生成

**教训**：task.yaml 在 session 后期才引入，导致早期决策记录需要迁移。

**建议**：task 创建流程中强制包含 task.yaml 生成，由安装器或 init 命令提供模板。

## 四、自举验证的启示

这次开发本质上是**半自举的**——我们使用了 Agent 驱动的开发方式（Sisyphus 执行、用户决策），但尚未使用 Self-Workflow 定义的工作流。

有几个信号表明工作流设计方向是对的：

1. **ADR-003（关键决策归档）被真正使用了**——7 个 ADR 记录了完整的决策链路
2. **Review Agent 审查发现了真实问题**——审查 ADR-001 找到了适用范围矛盾
3. **产物归档结构被实际使用**——`tasks/`、`adrs/`、`logs/` 目录承载了真实内容

但也有信号表明工作流需要调整：

1. **线性阶段模型不符合实际**——实际开发是迭代的、回溯的
2. **Agent 需要更多的架构判断Gate**——不能等用户来纠正方向
3. **决策记录应更自然地嵌入流程**——而不是作为额外步骤

---

## 五、总结

> **V1 验证了工作流的核心概念是可行的，但阶段模型需要从"线性流水线"调整为"迭代螺旋"。**
>
> 用户角色：架构师——把控方向、纠正偏差、记录决策
> Agent 角色：执行者——落地细节、发现问题、提出建议
>
> 这个分工本身就是 ADR-007（委托优先）的最好验证。
