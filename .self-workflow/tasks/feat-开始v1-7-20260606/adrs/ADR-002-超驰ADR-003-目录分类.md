# ADR-002：超驰 ADR-003 — 分类手段从文件名约定转向目录结构

> 日期：2026-06-06
> 状态：已采纳
> 超驰：`feat-v1.5剩余问题修复-20260606/adrs/ADR-003-经验文件名约定分类.md`

## 背景

ADR-003 选择了「方案 A：文件名约定」作为经验文档的分类手段，理由是 Agent 扫描文件名成本最低。该决策在 V1.5 时做出，当时的渐进式披露依赖 Agent 自律扫描——Agent 通过遍历文件名列表来判断文档分类。

V1.7 引入 OpenCode Plugin 机制后，分类判断不再依赖 Agent 手动扫描。Plugin 在 `chat.system.transform` 时直接读取文件系统的目录树和 YAML frontmatter，目录分类的开销被 Plugin 承担，不影响 Agent 的上下文窗口。

## 决策

**超驰 ADR-003，转向「方案 C：目录结构分类」**。

分类由文件所在的子目录决定（`实施经验/`、`参考模式/`、`错误经验/`）。文件名中的分类后缀降级为人类可读的冗余标记，不再是插件或 Agent 判断分类的依据。

Plugin 通过以下流程获取分类信息：
1. 解析 `docs/README.md` 中 `### 分类名/` 标题获取合法分类列表
2. 遍历各分类目录下的 `.md` 文件
3. 读取每个文件的 YAML frontmatter 获取 `title` 和 `tags`

## 超驰理由

1. **Plugin 改变了约束条件**：ADR-003 的核心前提——"Agent 扫描成本最低"——在 Plugin 架构下不成立。Plugin 在服务端执行文件 I/O，不影响 Agent 的 token 预算
2. **目录结构对人类更直观**：用户浏览 `docs/` 时能立即从目录名理解分类，不需记忆命名约定
3. **安装器管理更方便**：分类目录 + `.gitkeep` 可通过安装器模板同步，而文件名约定是隐式规则，无法通过工具校验
4. **未来扩展性**：新增分类只需在 `docs/README.md` 加一行 + 创建目录，Plugin 自动识别，无需改代码

## 影响

- `.self-workflow/docs/` 从扁平结构变为有子目录的结构
- Plugin 的 `scanDocs()` 函数按目录遍历而非按文件名匹配
- 现有文档迁移：`git mv` 到对应分类目录，修正文件名中的分类后缀
- `docs/README.md` 成为分类定义的权威源
