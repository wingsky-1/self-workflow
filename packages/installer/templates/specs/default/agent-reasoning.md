# Agent 推理与决策规范

## 核心原则

Agent 不是被动执行者。收到任何指令后，先**判断合理性**，必要时提出质疑，再执行。主 Agent 的上下文只保留**决策链路**，不保留执行细节。

## MUST（必须遵守）

### 独立思考

收到 Human 的方向后：
1. 判断方向是否合理
2. 识别隐含假设和潜在矛盾
3. 必要时提出质疑——沉默执行错误方向比质疑更糟

### 上下文管理

主 Agent 上下文只保留：
- 关键选择 + 理由
- 约束条件
- 下一步决策点

不保留：具体实现步骤、中间结果、子 Agent 的执行细节。

### 委托决策

收到任务后先拆分为独立子任务，逐项判断：

| 预估耗时 | 决策 |
|---------|------|
| **> 30 秒** | 委托给子 Agent（`task()`） |
| **5-30 秒** | 出错不影响主流程 → 自己做；否则委托 |
| **< 5 秒** | 自己做 |

### 工具选择优先

在执行任何文件操作（创建目录、写入文件、修改文件）前，Agent **MUST** 先检查项目是否提供匹配的内置工具：

1. **优先检查 `sw_task_*` 系列**：所有以 `sw_task_` 为前缀的工具（如 `sw_task_create`、`sw_task_list`、`sw_task_read`、`sw_task_phase_update` 等），按工具名匹配使用场景
2. **检查 `task()` 委托工具**：`task()` 是委托机制本身，在无专用内置工具时应作为降级选择
3. **存在匹配工具 → MUST 使用内置工具**，不得手工重复实现（如手动 `mkdir` + `write` 替代 `sw_task_create`）
4. **不存在匹配工具 → 按委托决策表判断是自己做还是委托子 Agent**

此规则是"委托优先"原则在工具选择层面的延伸：内置工具 = 最可靠的"委托目标"。

### 委托后验证

子 Agent 返回后，必须验证：
- [ ] 输出是否符合预期？
- [ ] 是否符合已有的 ADR 和 spec 约束？
- [ ] 是否引入了新问题？

不盲目信任子 Agent 的输出。

## SHOULD（建议遵守）

- 简单的顺序操作自己做更高效，不需要为了委托而委托
- 委托后等结果即可，不需要中间状态更新
- 并行化优先：独立的子任务同时启动，而非串行等待

## 委托决策流程图

```
收到任务
  │
  ├─ 拆分为独立子任务
  │
  ├─ 逐项判断耗时
  │   ├─ < 5s  → 自己做
  │   ├─ 5-30s → 隔离执行？是→自己做 / 否→委托
  │   └─ > 30s → 委托给子 Agent
  │
  └─ 并行启动所有独立委托
       │
       └─ 全部完成后 → 逐项验证 → 继续
```
