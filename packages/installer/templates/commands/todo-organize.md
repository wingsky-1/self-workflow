# /todo-organize 命令

## 用法

```
/todo-organize              # 交互模式：评审排期 + 整理，展示 diff 后确认写入
/todo-organize --dry-run    # 仅预览，不写入
```

## 执行流程

### 阶段 1：读取现状

1. 读取 `.self-workflow/todo.md` 全文
2. 扫描项目现状（只做必要的背景调查，不要扫 task.yaml）：
   - `.self-workflow/specs/` — 有哪些规范
   - `.opencode/commands/` — 有哪些命令
   - `.opencode/agents/` — 有哪些 agent
   - 是否有未解决错误（快速搜索 errors.yaml 中非空文件）
   - `.self-workflow/docs/` 规模感知（文件数量即可）

### 阶段 2：评审新增项

读取"## 新增（待评审排期）"章节的每一项，逐一评估：

**优先级判断框架（P0/P1/P2）：**
| 条件 | 优先级 |
|------|--------|
| 框架核心机制不工作/被绕过 | P0 |
| 直接影响当前每轮对话质量（如 Gate 跳过、实现无确认） | P1 |
| 效率/体验改善，非阻塞 | P2 |
| 依赖其他未完成能力 | P2 或延期 |

**归属判断：**
- 确定性能直接做的 → 排入具体版本
- 方向对但设计不清晰 → 标记延后，说明原因和需要什么前置条件
- 可与其他项合并 → 合并到同版本，保持每版本 2~3 项的粒度

### 阶段 3：整理活跃版本

对"## 新增"以外的活跃版本段执行：

1. **归档已完成版本**：所有项 `[done]` 的版本段移入"## 已关闭"，用 `<details>` 折叠
2. **调整优先级**：用阶段 2 的判断框架重新审视每个活跃项的 P0/P1/P2 标记
3. **重排序版本段**：P0 在前，同优先级按版本号
4. **合并重复项**：相似描述合并为一个，标注 `(合并自 #N, #M)`

### 阶段 4：生成排期建议

产出一张**排期建议表**（即 question 确认的核心内容）：

| 版本 | 优先级 | 内容 | 来源 |
|------|--------|------|------|
| V1.x | P0/P1/P2 | 项 1 + 项 2 | 新增 #N / 原 V1.y |
| … | … | … | … |
| V1.y 延后 | — | 项 3 | 原因：xxx |
| 远期 | — | 项 4 | 原因：依赖 xxx |

### 阶段 5：确认写入

1. 输出上述排期建议表 + 结构性变更摘要
2. 使用 question 工具展示，待 Human 确认
3. 确认后写入 `.self-workflow/todo.md`
4. `--dry-run` 则到此结束，不写入

## 原则

- **新增项评审是核心价值**，不应跳过或"保留不变"
- **每版本 2~3 项**，内容紧密相关，单次 `/feat` 可消化
- **不确定的放延后**，标注原因和前置条件，不强行排期
- **已关闭区域**按版本号从新到旧排列

## 错误处理

| 场景 | 行为 |
|------|------|
| todo.md 不存在 | 提示先运行 `self-workflow init` |
| "新增"章节缺失 | 警告后继续，跳过阶段 2 |
